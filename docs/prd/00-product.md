## §1. Vision

**Orbis — персональная AI-операционная система для жизни.** Не очередной трекер задач, расходов или привычек, а одно место, где живёт всё, что человек хочет помнить, планировать и считать, — и AI, который умеет с этим работать.

Главный тезис:

> **Orbis продаёт не трекинг, а связность — ответы на вопросы, требующие данных из нескольких доменов сразу.**

«Могу ли я позволить себе кроссовки?» — простой вопрос, на который не отвечает ни один специализированный инструмент. Ответ требует одновременно: остатка в конверте бюджета, предстоящих recurring-платежей (аренда спишется через три дня) и уже запланированных покупок, которые тоже претендуют на этот остаток. Трекер расходов знает первое, календарь — второе, список желаний — третье. Orbis знает всё сразу, потому что всё это — один граф.

Удобный способ думать об Orbis — аналогия с операционной системой:

| Orbis | Аналог в ОС |
|---|---|
| Граф сущностей | Файловая система |
| Чат (AI + fast-path) | Терминал |
| Entity Browser | Файловый менеджер |
| Views (Budget, Agenda, будущие) | Приложения |

Приложения приходят и уходят — файлы остаются. Точно так же views в Orbis — сменные линзы поверх графа, а данные принадлежат пользователю и переживают любой view.

У Orbis два якорных сценария использования, и оба — ядро продукта, не Future:

**Личная ОС** — то, что описано выше: задачи, финансы, заметки, планирование дня и недели с ИИ, у которого большой контекст жизни владельца (граф + память + треды).

**Рабочее пространство человек + агенты** — внешние агенты (Claude Code через MCP) работают внутри Orbis над задачами проектов: тот же граф и тот же реестр тулов, что у владельца. Их действия видимы, атрибутированы, обратимы и управляемы с телефона — Orbis не прячет работу агента в чужом логе, а показывает её как работу равноправного участника графа. Первый проект в системе — разработка самого Orbis: агент ведёт задачи и пишет заметки, владелец наблюдает и отвечает из чата (см. §7).

## §2. Боли и ответы

Шесть болей, из которых вырос продукт, — и ответ Orbis на каждую.

**1. Жизнь размазана по 5–10 приложениям.** Задачи в одном, деньги во втором, календарь в третьем, заметки в четвёртом — и одна реальная вещь дублируется в трёх местах, нигде не сходясь.
→ **Один граф.** «Купить кроссовки 8000₽ к субботе» — это *одна* сущность с аспектами `task` + `financial` + `schedule`. Она видна в списке задач, в бюджете и в агенде на субботу — без дублирования и рассинхрона. При завершении такой задачи-покупки Orbis предложит записать расход как факт — одно касание, не автоматика.

**2. Friction ввода убивает трекинг.** Любой трекер умирает одинаково: на третьей неделе надоедает заполнять формы.
→ **Ввод без формы.** «обед 340» в чат — fast-path распознаёт паттерн мгновенно, офлайн и без LLM: карточка с категорией появляется быстрее, чем открылась бы форма. А раз в неделю — CSV-выписка из банка, и пропущенное руками всё равно попадает в историю.

**3. Данные мёртвые.** Накопил год транзакций и тренировок — а спросить их ни о чём нельзя: только листать графики, которые придумал разработчик.
→ **AI-запросы поверх графа.** «Могу ли я позволить X?» — и ответ учитывает конверты, recurring-платежи и planned-расходы. Данные перестают быть архивом и становятся собеседником.

**4. Инструменты навязывают модель.** Хочешь трекать сон, книги или прогресс в испанском — ищи отдельное приложение и принимай чужую структуру.
→ **Aspect registry.** Новый домен — это запись в реестре аспектов: без изменений ядра и без миграций, и AI сразу умеет с ним работать (см. 01-architecture §3). Создание собственных доменов самим пользователем — через чат или настройки — отнесено в Future Directions (§10): механизм готов, пользовательский флоу — потом.

**5. Смена инструмента = потеря истории.** Переехал из одного трекера в другой — годы данных остались в чужом экспорте.
→ **Данные существуют до views.** Views — линзы, данные — в графе. Установил Budget — он показал всю историю расходов, накопленную до его появления. Удалил view — не потерял ни записи.

**6. Работа агентов непрозрачна и живёт вне моих инструментов.** Поручил задачу внешнему агенту — статус, ход мысли и результат остаются в чужом трекере или чужом чате, вне графа жизни владельца.
→ **Агенты работают в том же графе, что и владелец.** Каждое действие агента — запись в журнале, атрибутированная (`actor_user_id`), обратимая (Undo) и подчинённая политике подтверждений (01-architecture §7.10). Агент — не система рядом с Orbis, а равноправный участник графа.

## §3. Пользователь и цель итерации

Цель этой итерации — не «личный MVP когда-нибудь», а **максимально быстрый продакшен**, в котором ежедневно работают двое: владелец продукта (с телефона и с ноута) и внешние AI-агенты (Claude Code через MCP). Критерий готовности — не демо, а факт: **работа над Orbis ведётся в Orbis**. Это честный фильтр приоритетов: строится то, что нужно для реальной жизни и реальной разработки, а не для скриншотов лендинга.

При этом задел на публичность закладывается с первого дня — там, где доделывать постфактум дорого:

- **Multi-user-ready схема.** Каждая запись принадлежит пользователю с первого дня; появление второго пользователя — вопрос регистрации, а не миграции данных.
- **Freemium-шов.** Все лимиты и проверки доступности фич идут через единый entitlements-механизм (см. 01-architecture §8); сегодня он отвечает «всё можно», завтра — по тарифу, и ни одна проверка в коде не меняется.

Биллинг, тарифная сетка и paywall — сознательно вне скоупа: продавать нечего, пока продукт не доказал себя на одном требовательном пользователе.

## §5. Принципы ввода

Ввод — главное поле боя любого трекера, поэтому у Orbis не один способ ввода, а четыре пути — от самого частого к самому редкому:

1. **Fast-path** (слайс 1) — частотные паттерны («обед 340», «+150000 зарплата»): мгновенно, офлайн, без LLM. Подавляющее большинство ежедневных записей.
2. **LLM-чат** (слайс 1) — всё сложное и cross-aspect: «купить кроссовки 8000 к субботе», «перенеси встречу и подвинь дедлайн».
3. **Quick-add в views** (слайс 2) — контекстный ввод там, где пользователь уже находится: бар добавления расхода с последними категориями прямо в Budget.
4. **CSV-импорт** (слайс 2) — массовая загрузка: банковская выписка раз в неделю закрывает всё, что не попало в первые три пути.

Ключевой принцип: **пользователь не выбирает режим**. Он просто пишет — система сама решает, каким путём обработать ввод: уверенный fast-path срабатывает мгновенно, всё остальное молча уходит в LLM (механика — см. 01-architecture §7.5). Никаких переключателей «умный режим / быстрый режим».

## §6. UX-принципы

### §6.1 Навигация

- Нижняя панель из четырёх вкладок: **Chat / Browser / Agenda / Budget**.
- Views — **обычные экраны, не модалки**: Budget — такой же полноправный экран, как Chat.
- Детальный экран сущности — push поверх текущего экрана; back-стек предсказуем: «назад» всегда возвращает туда, откуда пришёл.
- **Deep links** на сущности и треды: любую вещь в Orbis можно открыть по ссылке.
- Эволюция заложена: когда установленных views станет 3+, вкладка Budget превратится в «Views» со списком — путь к Hub Launcher не потребует перестройки навигации.

### §6.2 Corrections & Undo

Продуктовое обещание: **каждое действие AI обратимо и правится в одно касание; система учится на исправлениях.**

AI ошибся с категорией — тап по карточке, исправил поле на месте, готово. Исправил одно и то же дважды — система предложит запомнить правило как memory-факт, и оно начнёт работать везде, включая мгновенный fast-path. «Отмени последнее» работает из чата — в том числе для действий внешних агентов: эти правила распространяются на агентов целиком, а какой уровень подтверждения применяется к действию до его исполнения, решает политика Orbis, а не модель (01-architecture §7.10). Механика журнала действий и обратных операций — см. 01-architecture §7.8 (журнал действий).

### §6.3 Notification policy

В MVP **нет push-уведомлений и нет проактивных сообщений от AI**. Orbis не пишет первым.

Вся сигнализация — пассивная и на виду: бейджи на вкладках нижней панели и секции внутри экранов — просроченное поднимается наверх в Agenda, перерасход конверта виден в Budget. Пользователь приходит сам и сразу видит, что требует внимания.

Proactive AI — это Future, и появится он только вместе с явной политикой: дайджест вместо потока пингов, quiet hours, настройка каналов. Уведомления без политики — спам, который убивает доверие быстрее, чем любой баг.

## §7. День пользователя

**Утро.** Открыл PWA, в сайдбаре Browser — pinned-сущность «Daily Planning»: задачи из Inbox, список «Сегодня», агенда дня. Три минуты — день спланирован.

**День.** В обед написал в чат «обед 340» — карточка появилась мгновенно: категория «Еда», сумма 340₽. Остатка конверта карточка не показывает — конверты и Budget Overview приходят со слайса 2, а факт расхода фиксируется уже сейчас и станет историей, которую Budget подхватит, когда появится (§9). Тут же в чате — «купить кроссовки 8000₽ к субботе»: cross-aspect намерение, LLM-путь создаёт одну сущность сразу с `task`, `financial` и `schedule` — она видна и в списке задач, и в расписании на субботу, без дублирования.

**Вечер.** Такси: AI положил его в «Транспорт», а это были «Развлечения» — тап по карточке, поменял категорию. На втором таком исправлении система предложила правило — теперь запоминает сама. «Что по бюджету?» — в слайсе 1 на этот вопрос отвечает чат: агрегирует «Еду» и «Транспорт» этого месяца прямо по сущностям-транзакциям и коротко резюмирует; экран Budget Overview с остатками по конвертам и темпом появится со слайса 2. Пересел за ноут дописать код — тот же граф, тот же тред, никакой разницы с телефоном. Ещё утром, с телефона, закинул агенту (Claude Code через MCP) задачу в тред проекта «Orbis»: «разберись, почему query на списке транзакций тормозит». Сейчас видно: статус задачи — «done», в треде — заметка агента, что и почему поменял. Прочитал, задал уточнение в том же треде, поставил следующую задачу — работа над Orbis ведётся в Orbis.

**Раз в неделю.** Скачал CSV из банка, кинул в Orbis. AI распарсил выписку, разметил вероятные дубли ручных записей («обед 340» уже введён руками — строка выписки помечена как дубль), разложил по категориям и подсветил аномалии — осталось подтвердить пачкой. Механизм дедупликации — в 03-budget.md.

**Конец месяца.** Rollover: AI предложил конверты нового месяца с переносом остатков — где-то плюс, в «Еде» небольшой перерасход. Просмотрел, подтвердил одним экраном.

**Через полгода.** Решил всерьёз заняться спортом и включил fitness-аспект. Вся история «был в зале», которую полгода записывал на бегу, — теги и заметки — ретроактивно мигрирует в типизированные тренировки: график за шесть месяцев в первый же день. Это wow-момент, ради которого слой тегов и meta пишется всегда.

## §8. Метрики успеха

Шесть метрик, по которым MVP считается состоявшимся:

| Метрика | Цель |
|---|---|
| Median time-to-log расхода | < 5 сек (fast-path < 2 сек) |
| Доля AI-записей, потребовавших исправления | < 10% |
| Ежедневное использование | 30+ дней подряд |
| Cross-aspect запросы | ≥ 3 в неделю с полезным результатом (по самооценке владельца) |
| Покрытие транзакций | ручной ввод + импорт покрывают ≥ 95% транзакций банка — измеряется со слайса 2 (требует CSV-импорта) |
| Приёмка слайса 1 | Работа над Orbis ведётся в Orbis: агент создал проект «Orbis», перенёс документацию, и разработка слайса 2 идёт через задачи в самом Orbis |

Намеренно здесь нет метрики «% ввода через чат»: канал ввода — не самоцель. Важно, чтобы записывать было быстро (time-to-log), точно (correction rate) и полно (покрытие транзакций), а связность окупалась (cross-aspect запросы) — тогда привычка (30+ дней) складывается сама.

## §9. Роадмап

Фаза 0 плюс три слайса, без календарных оценок — порядок важнее дат. Это канонический состав: пометки «слайс N» в 02-core-os.md и 03-budget.md ссылаются на разбивку ниже, не дублируют её.

**Фаза 0 — спайки (блокирующие, до начала слайса 1).** **SPIKE-01** проверяет RLS через Drizzle/Bun: pooled-переиспользование соединений не путает identity пользователей, service-role — не fallback. **SPIKE-05** проверяет деплой-связность: хостинг (регион, session-pool, персистентный контейнер, секреты, бэкапы, стоимость). Продакшен нужен рано — агентная петля живёт только на задеплоенном приложении.

**Слайс 1 — «Агентная петля + ввод» (продакшен).** Сущности, проекты, задачи и заметки — CRUD через executor со статусами и связями `parent`; Browser-lite (список, сайдбар pinned, три сидированных smart lists, минимальный detail-экран: title, теги, body, аспекты, подзадачи); Chat — глобальный тред и треды сущностей, fast-path (расходы копятся с первого дня без Budget-view — данные существуют до views), LLM-путь с тулами, карточки, журнал и Undo, политика подтверждений; MCP-сервер — PAT, тот же реестр тулов, паттерн «что нового»; retry-буфер, онбординг-сидирование (категории, smart lists, настройки, глобальный тред), экспорт данных, деплой в продакшен.

**Приёмка слайса 1:** агент создал в Orbis проект «Orbis», перенёс документацию (спеки и планы — note-сущности), и разработка слайса 2 дальше ведётся через задачи в самом Orbis — агент двигает статусы и пишет заметки, владелец наблюдает и отвечает в тредах с телефона.

**Слайс 2 — «Финансовый контур + время».** Budget Overview и конверты (сразу показывают историю, накопленную со слайса 1), quick-add бар, recurring-платежи и события с Coming up и переходом planned→fact, Agenda-lite, rollover на границе месяца, CSV-импорт банковских выписок (дедуп на `entity_origins`), экран памяти AI с эскалацией повторных исправлений в правила, полный detail-экран сущности (блокировки, backlinks). На выходе: полный финансовый контур и покрытие транзакций, измеримое метрикой (§8).

**Слайс 3 — «Полировка и глубина».** Suggestion chips, бейджи вкладок, deep links, визуальная форма query-builder поверх грамматики, перф-бюджеты как гейты релиза, аспект `orbis/goal` и планирование горизонтов (день/неделя/месяц/год/жизнь) как AI-сценарии поверх уже готового механизма `progress_source`. На выходе: продукт, которым не страшно пользоваться как единственным хранилищем жизни.

## §11. Риски и митигации

| Риск | Митигация |
|---|---|
| AI неточно категоризирует — доверие к записям падает | Fast-path с резолвом по синонимам категорий; правила из повторных исправлений; контроль метрикой correction rate < 10% |
| Ручной ввод умирает через месяц — история дырявая | CSV-импорт выписок в MVP, а не «потом»: даже при пропусках руками история остаётся полной (метрика покрытия ≥ 95%) |
| Без сети приложение почти не работает — цена онлайн-первого разреза | Offline отложен осознанно, не как нерешённая проблема (Future, §10); ввод не теряется — retry-буфер fast-path-create (01-architecture §5.3), чтение — из server-state-кэша (01-architecture §5.1) |
| Чат перегружается — всё в одной бесконечной ленте | Треды сущностей: обсуждение живёт рядом с вещью, глобальный чат — только для нового |
| Scope creep платформенности — ядро расползается | Жёсткое правило: новый домен = записи в реестре аспектов + view, без изменений ядра (валидировано на трёх доменах — см. 01-architecture §11) |
| AI недоступен или провайдер сменился | LLM-agnostic слой; при сбое провайдера или исчерпанном лимите всё, кроме LLM-пути, работает как обычно — явная ошибка с «повторить», не тихая деградация (см. 01-architecture §7.9) |
| Write-агент действует без ведома владельца или портит данные — подрывает доверие к графу | Политика подтверждений по уровню риска действия (01-architecture §7.10); каждое действие агента — в журнале с атрибуцией и обратимо через Undo, как и у внутреннего AI (01-architecture §7.8) |

## Приложение: Глоссарий

| Термин | Значение |
|---|---|
| **Сущность** (entity) | Универсальная единица графа — всё, что пользователь создаёт и видит: задача, расход, заметка, категория, конверт. Имеет имя, текст, теги и аспекты |
| **Аспект** (aspect) | Типизированная грань сущности (`task`, `financial`, `schedule`…). Одна сущность может иметь несколько аспектов сразу; схемы аспектов живут в реестре (см. 01-architecture §3) |
| **View** | Линза поверх графа (Budget, Agenda, будущие Calendar и Fitness). Хранит только настройки отображения; данные живут в графе |
| **Конверт** | Лимит трат по категории на период; «потрачено» всегда вычисляется из реальных транзакций, а не хранится отдельно |
| **Категория** | Сущность (не строка), группирующая финансовые операции: имеет иерархию, синонимы для распознавания ввода и собственные правила |
| **Query-движок** | Единая грамматика живых запросов к графу с одним бэкендом (Postgres) — «одна грамматика, шесть потребителей»; питает списки, фильтры views и инструменты AI (см. 01-architecture §6) |
| **Fast-path** | Детерминированный парсер частотного ввода («обед 340»): мгновенно, офлайн, без LLM (см. 01-architecture §7.5) |
| **Memory-факт** | Знание или правило AI («бар → Развлечения»), хранимое обычной сущностью: пользователь видит, правит и курирует память напрямую |
| **Тред** | Персистентный диалог с AI: один глобальный плюс по одному у каждой сущности — обсуждение живёт рядом с вещью |
| **Pinned-сущность** | Сущность, закреплённая в сайдбаре Browser; преднастроенные Daily Planning / Upcoming / All Tasks — обычные сущности с живыми запросами в теле |
| **Retry-буфер** | Локальная очередь ещё не подтверждённых сервером create-операций fast-path-ввода; переживает разрыв сети, идемпотентна по client-UUID, не подменяет собой офлайн-режим (см. 01-architecture §5.3) |
| **Optimistic-check** | Проверка версии `updated_at` при правке `body`: правка с устаревшей версией отклоняется структурированной ошибкой вместо тихой перезаписи или merge (см. 01-architecture §5.2) |
| **PAT** (Personal Access Token) | Токен доступа для внешних агентов (например, Claude Code через MCP); в MVP один токен полного доступа на владельца, хранится хешем, показывается один раз (см. 01-architecture §9.3) |
| **entity_origins** | Таблица provenance импортированных сущностей: связывает сущность с источником и внешним ID, обеспечивает дедуп повторного импорта (см. 01-architecture §4.8, 03-budget §3.4.1) |
| **Паттерн «что нового»** | Способ агента узнавать об изменениях без push: запрос сущностей по `updated_at` новее курсора прошлого опроса плюс чтение их тредов; курсор агент хранит у себя (см. 01-architecture §9.3) |
| **Ретроактивная миграция** | Подключение нового аспекта к уже накопленной истории: записанное тегами и meta становится типизированными данными задним числом (см. 01-architecture §3.10) |
