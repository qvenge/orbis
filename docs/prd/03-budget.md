## §1. Обзор и философия

Budget — **envelope-бюджетирование с AI-ассистом**. Пользователь распределяет ожидаемый доход по конвертам (envelopes) на период; каждый расход «вычитается» из своего конверта; остаток показывает, сколько ещё можно потратить. AI не управляет деньгами автоматически — он **предлагает** (аллокации по истории трат, категории при импорте, перенос остатка), пользователь **подтверждает**. Принцип ввода тот же, что в ядре (00-product §5): низкий friction (quick-add, fast-path, CSV-импорт), но решение всегда за человеком.

### §1.1 Budget — линза, не хранилище

Следствие принципа «данные существуют до views» (01-architecture §1.3, принцип 1): **Budget не владеет ни одной записью**. Все его данные — обычные сущности графа:

- **транзакции** — сущности с `orbis/financial`;
- **конверты** — сущности с `orbis/budget`;
- **категории** — сущности с `orbis/category` (общие с ядром, см. 02-core-os §7.1).

Удаление view не теряет ничего: расходы остаются финансовыми сущностями и видны в Browser и Agenda. Budget хранит только настройки отображения в `viewPreferences["orbis-budget"]` (01-architecture §4.4).

### §1.2 Без Budget расходы всё равно трекаются

Budget — **визуализация**, а не предусловие учёта. Явная рамка слайсов (00-product §9): fast-path-ввод расходов работает с первого дня продукта — **слайс 1** — «данные существуют до views» (01-architecture §1.3, принцип 1): история копится с первого дня без единого экрана Budget. Сам экран Budget (Overview и остальные экраны §3) — **слайс 2**; при первом открытии он не стартует с пустого места, а сразу показывает историю, накопленную за слайс 1.

До установки view расход «обед 340» создаётся fast-path'ом (01-architecture §7.5), несёт `orbis/financial` с `category_ref`, виден в Browser и правится в чате. Установка Budget добавляет агрегацию и прогресс — но не меняет модель данных. Это отличает Orbis от специализированных бюджет-приложений: учёт не привязан к view.

### §1.3 Что Budget даёт сверх Entity Browser

Entity Browser (02-core-os §3) показывает финансовые сущности **сырым списком** строк (native-рендеринг `orbis/financial` — 02-core-os §3.6: сумма со знаком + бейдж категории) с фильтрами и поиском. Агрегаций, прогресса и прогноза там нет — это работа Budget.

| Возможность | Entity Browser | Budget |
|---|---|---|
| Сырой список транзакций | ✓ (native-рендеринг §3.6) | ✓ (экран «Транзакции», §3.3) |
| Фильтр/поиск по транзакциям | ✓ (фильтры §3.8) | ✓ (расширенные финансовые фильтры, §3.3) |
| **Прогресс конвертов** (spent / limit, пороги) | — | ✓ (Overview, §3.1) |
| **Графики и тренды** (мини-тренд категории) | — | ✓ (экран категории, §3.2) |
| **Carryover** (перенос остатка между периодами) | — | ✓ (Rollover, §3.5) |
| **Прогноз** (recurring + planned до конца периода) | — | ✓ (Coming up + Planned, §3.1) |
| **Темп трат** (daily_pace и прогноз исчерпания внутри периода) | — | ✓ (карточка конверта на Overview §3.1; формулы §2.4) |
| **Баланс периода** (income − expense) | — | ✓ (Overview, §3.1) |

Все агрегации Budget — **вычисления на лету** поверх графа единым query-движком (01-architecture §6), не материализованные таблицы. Тяжёлая аналитика поверх JSONB — известное ограничение (01-architecture §12, клапан — декларация `aggregations`).

## §2. Модель бюджета

Схемы полей — 01-architecture §3 (`orbis/category` §3.6, `orbis/budget` §3.5, `orbis/financial` §3.3); здесь не дублируются. Этот раздел задаёт **связи между сущностями** и **формулы**, которыми Budget читает граф.

### §2.1 Дерево: категория → конверты → транзакции

```
Категория «Еда»                              (orbis/category — вне времени)
 │  aliases, icon, color, body-правила
 │
 ├─ parent ──→ Конверт «Еда — май 2026»       (orbis/budget: period 05-01..05-31,
 │              │                               limit, carryover)
 │              ├─ parent ──→ Транзакция «Пятёрочка» (orbis/financial, planned=false)
 │              └─ parent ──→ Транзакция «Обед»      (orbis/financial, planned=false)
 │
 └─ parent ──→ Конверт «Еда — июнь 2026»       (orbis/budget: period 06-01..06-30)
                ├─ parent ──→ Транзакция «Перекрёсток»
                └─ parent ──→ Транзакция «Кофе»
```

Три уровня, все связаны relation `parent` (01-architecture §4.2: source — родитель target):

1. **Категория** (`orbis/category`) — вне времени, одна на всё. Несёт `aliases` (резолв ввода), `icon`/`color` (отображение), body-правила, иерархию и собственный тред (01-architecture §3.6).
2. **Конверт** (`orbis/budget`) — не более одного на точную комбинацию `(категория, currency, period_start, period_end)`; повтор отклоняется валидацией executor'а. Источник `parent` — категория; цель — конверт. Конверт несёт `limit`, `currency`, `period_start`/`period_end`, `carryover`.
3. **Транзакция** (`orbis/financial`) — несёт `category_ref` (на категорию-сущность) **и** входящую relation `parent` от конверта периода.

Двойная привязка транзакции (`category_ref` + relation `parent`) намеренна: `category_ref` — стабильная семантика «к какой категории относится» (переживает отсутствие конверта); relation `parent` к конверту — то, по чему конверт считает `spent`.

### §2.2 Правило `spent`: единственный источник истины

`spent` конверта **не хранится** (01-architecture §3.5, правило 3 §10) — вычисляется на лету:

```
spent(конверт) = SUM(financial.amount транзакций-детей конверта,
                     где direction = expense И planned = false
                     И occurred_on <= сегодня
                     И currency = currency конверта)
```

«Дети конверта» — сущности, у которых конверт является родителем по relation `parent` (запрос `children_of=<конверт>`, 01-architecture §6.1). Это единственный источник истины: правка суммы, рекатегоризация (перенос relation `parent`) или архивация транзакции немедленно и без рассинхрона меняет состояние конверта. Любое отображаемое `spent` — вычисление или кэш по правилу 3 (пересчёт при каждой записи, затрагивающей источник, 01-architecture §10).

Доходы (`direction=income`) в `spent` конверта не входят — они учитываются в балансе периода (§2.5) и как возврат (§5). Чужая валюта в `spent` не входит (§5).

### §2.3 Авто-привязка транзакции к конверту

Когда создаётся транзакция (через Chat, fast-path, quick-add, CSV-импорт или внешнего агента), Budget связывает её с конвертом автоматически:

1. Из транзакции берётся `category_ref` и дата `orbis/financial.occurred_on` (01-architecture §3.3). Fast-path/quick-add ставят локальную сегодняшнюю дату, CSV сохраняет дату операции из выписки, planned и recurring-инстансы — ожидаемую дату. `created_at` в финансовых периодах не используется.
2. Ищется конверт этой категории, чей период `[period_start; period_end]` **включает** дату транзакции (предикат: `period_start ≤ дата ≤ period_end`). Кандидаты конвертов категории берутся запросом `aspect=orbis/budget, category_ref=<кат>` (01-architecture §6.1), попадание даты в период проверяется при отборе.
3. Если кандидатов несколько из-за пересечения произвольного и месячного конвертов, выбирается **самый узкий период** (минимум календарных дней), затем более поздний `period_start`, затем меньший UUID — это byte-точный tie-break. UI до создания пересекающегося конверта показывает это правило и предлагает отдельную подкатегорию, если пользователь не хочет перехватывать транзакции.
4. Выбран → создаётся relation `parent` (конверт → транзакция). У транзакции может быть максимум один budget-parent (инвариант 01-architecture §4.2), поэтому двойной учёт невозможен. Фактическая транзакция (`planned=false`, `occurred_on ≤ сегодня`) входит в `spent`; будущая остаётся прогнозом.
5. Не найден (категория без конверта на этот период) → транзакция остаётся **без** `parent`-конверта и попадает в секцию **Unbudgeted** (§3.1) с предложением создать конверт (§5).

Авто-привязка — поведение view, а не инвариант ядра: транзакция валидна и без конверта. Когда создание/правка транзакции требует создать, удалить или перенести budget-parent, все связанные мутации выполняются одним `batch_execute` (01-architecture §9.2), поэтому пользователь никогда не наблюдает промежуточную двойную/устаревшую привязку, а Undo откатывает изменение целиком.

Создание, изменение периода/категории или архивация конверта повторно запускает тот же selector для всех затронутых транзакций: новый более узкий конверт может атомарно перехватить операции у месячного, удаление возвращает их следующему кандидату или в Unbudgeted. Результат зависит только от текущего набора конвертов, а не от порядка их создания.

### §2.4 Формулы конверта

```
effective_limit = limit + carryover
spent           = SUM(amount транзакций конверта, planned=false)   (§2.2)
remaining       = effective_limit − spent
daily_pace      = remaining / max(1, дней_до_конца_периода включительно)
```

- `effective_limit` — реальный потолок периода с учётом перенесённого остатка. `carryover` может быть отрицательным (перерасход прошлого периода урезает текущий лимит — 01-architecture §3.5).
- `daily_pace` — сколько можно тратить в день, чтобы уложиться в остаток. `дней_до_конца_периода` отсчитывается от сегодня до `period_end` включительно; `max(1, …)` страхует деление в последний день. Если `remaining < 0`, `daily_pace` отрицателен — индикатор перерасхода, в UI показывается как «—/день» (em-dash; см. §3.1).
- **Темп (прогноз исчерпания):** `прогноз_расхода_за_период = spent / прошедших_дней_периода × всего_дней_периода`. Сравнение с `effective_limit` даёт ответ «уложусь / перерасход» и подсвечивает конверт (§3.1). Темп — производное отображение, не хранится.

### §2.5 Баланс периода

```
income(период)  = SUM(financial.amount, direction=income, planned=false,
                       occurred_on в периоде И <= сегодня, валюта = валюта периода)
expense(период) = SUM(financial.amount, direction=expense, planned=false,
                       occurred_on в периоде И <= сегодня, валюта = валюта периода)
balance(период) = income − expense
```

Баланс считается по всем транзакциям периода независимо от наличия конверта (включая Unbudgeted), но **в рамках валюты периода** — чужая валюта в баланс не входит (§5).

### §2.6 Carryover-логика

При закрытии периода остаток каждого конверта переносится в `carryover` конверта следующего периода той же категории:

```
carryover(следующий период) = remaining(прошлый период)
                            = effective_limit(прошлый) − spent(прошлый)
```

- **Профицит** (`remaining > 0`): неистраченное добавляется к лимиту следующего периода (накопление в рамках категории — «отложил на отпуск»).
- **Дефицит** (`remaining < 0`): перерасход переносится отрицательным `carryover`, урезая следующий период (дисциплина — «перетратил в мае, в июне меньше»).

Перенос — не автоматический фоновый процесс, а **Rollover-флоу** (§3.5): пользователь видит предложение AI и подтверждает (можно обнулить carryover для конкретной категории — «новый месяц с чистого листа»). До подтверждения `carryover` нового конверта = `0` (дефолт схемы, 01-architecture §3.5).

### §2.7 Planned-расходы — слайс 2

`orbis/financial.planned=true` (01-architecture §3.3) — запланированная, ещё не совершённая операция. Её ожидаемая дата хранится в `occurred_on`; для покупки-задачи она совпадает с ожидаемой датой покупки, а не с `created_at`.

- **НЕ входит в `spent`** конверта (формула §2.2 фильтрует `planned=false`) и в баланс периода (§2.5).
- **Входит в прогнозы**: секция Planned на Overview (§3.1) и расчёт «могу позволить?» (§4) учитывают planned как предстоящий отток.
- **Завершение покупки** (UI-флоу, согласован с 02-core-os §5 и §6): покупка-задача часто несёт `orbis/task` + `orbis/financial` (planned). При закрытии задачи Budget предлагает **перевести план в факт**: карточка «Покупка совершена? 8000 ₽ → Одежда» с датой факта (default сегодня, редактируема). Подтверждение одним batch ставит `planned=false`, обновляет `occurred_on` на фактическую дату и заново выбирает конверт (§2.3), после чего расход входит в `spent`. Отклонение оставляет расход planned. Действие обратимо целиком.

```
┌────────────────────────────────────────────┐
│  ✅ Купить кроссовки — задача закрыта        │
│  ┌──────────────────────────────────────┐  │
│  │ Покупка совершена?                    │  │
│  │ 👟 8000 ₽ → Одежда (planned → факт)  │  │
│  │ [Перевести в факт]  [Оставить план]   │  │
│  └──────────────────────────────────────┘  │
└────────────────────────────────────────────┘
```

### §2.8 Recurring-операции — слайс 2

Регулярные операции (подписки, аренда, зарплата) — `orbis/financial.recurring=true` на сущности-шаблоне с `orbis/schedule.recurrence` (01-architecture §3.3 инвариант recurring, §3.1). Инстансы порождаются связью `derived_from` с детерминированными ID и материализуются лениво **на горизонт 14 дней** (01-architecture §5.4). Каждый получает `occurred_on` даты инстанса и до этой даты имеет `planned=true`. Budget — потребитель этой материализации:

- Секция **Coming up** (§3.1) показывает материализованные инстансы на 14 дней вперёд.
- При первом открытии приложения или финансовом запросе в локальный день `occurred_on` системный идемпотентный batch снимает `planned`, авто-привязывает инстанс к конверту (§2.3) и только тогда включает его в `spent`; фоновой wake-up не требуется. До перехода, включая все будущие даты, инстанс остаётся исключительно прогнозом. Чтобы пропустить ожидаемую операцию, пользователь архивирует инстанс до даты; уже выполненный transition можно отменить обычным Undo.
- Шаблоны (сущности с `recurrence`) сами в `spent` не входят и в списках транзакций скрыты — как в Agenda (02-core-os §4.1); виден инстанс, шаблон открывается с его detail-экрана.

### §2.9 Произвольные периоды

Конверт не обязан совпадать с календарным месяцем — `period_start`/`period_end` произвольны (01-architecture §3.5). Это покрывает разовые бюджеты: конверт «Отпуск в Грузии» категории «Путешествия» с периодом `2026-08-10..2026-08-24`. Авто-привязка (§2.3) и `spent` работают одинаково — критерий «дата транзакции в `[period_start; period_end]`» не различает месяц и две недели. Произвольный конверт виден на Overview наравне с месячными; в Rollover (§3.5) он не участвует, пока не закрылся (его период не привязан к смене месяца).

**Карточка конверта, когда «сегодня» вне его периода.** `daily_pace` (§2.4) имеет смысл только внутри активного периода, поэтому отображение карточки конверта (на Overview §3.1 и на экране категории §3.2) зависит от положения «сегодня» относительно `[period_start; period_end]`:

- **(а) период ещё НЕ начался** (`сегодня < period_start`): карточка показывает `spent = 0`, прогресс-бар пустой и нейтрального цвета (пороги подсветки §3.1 не применяются); вместо `daily_pace` — подпись «начнётся DD.MM» (= `period_start`).
- **(б) период уже ЗАКРЫТ** (`сегодня > period_end`): карточка показывает итоговые `spent`/`effective_limit` (с цветом по итоговому порогу §3.1), вместо `daily_pace` — подпись «завершён», сама карточка приглушена (визуально притушена).
- **(в) активный период** (`period_start ≤ сегодня ≤ period_end`): карточка как обычно — прогресс-бар, пороги, `remaining` и `daily_pace`.

`daily_pace` вычисляется **только** в случае (в); в случаях (а) и (б) формула `daily_pace` (§2.4) не считается. Это правило одинаково для месячных и произвольных конвертов.

### §2.10 Иерархия категорий

Категории образуют дерево через relation `parent` (01-architecture §3.6): например, «Развлечения» → «Кино», «Бар». В отображении **родительская категория агрегирует детей**:

- На Overview родительская категория показывает суммарный `spent` и суммарный `effective_limit` своих конвертов и конвертов всех дочерних категорий (рекурсивно).
- Конверт можно завести как на родительскую категорию, так и на дочернюю; транзакция привязывается к конверту своей конкретной `category_ref`.
- Агрегация — отображение (вычисление обходом дерева категорий), не отдельная сущность. Стартовый набор категорий (02-core-os §7.1) плоский; иерархию пользователь строит сам.

## §3. Экраны

Шесть экранов. Корень вкладки Budget — Overview; экраны категории, транзакций, импорта и rollover — push поверх корня (паттерн навигации — 02-core-os §1.1). Quick-add — бар, не отдельный экран. Тап по транзакции/категории/конверту пушит detail-экран сущности (02-core-os §3.5) — единый для всех вкладок. Вкладка Budget целиком — **слайс 2** (00-product §9); все шесть экранов и quick-add-бар помечены индивидуально ниже.

- **Заголовок периода** — текущий месяц с переключателем `[◀ месяц ▶]`; по умолчанию открывается текущий месяц (в таймзоне пользователя). Произвольные конверты (§2.9) показываются в месяцах, на которые попадает их период.
- **Баланс** — `balance(период)` (§2.5): доход − расход, цвет по знаку.
- **Карточка-конверт** — имя и иконка категории (`orbis/category.icon`/`color`, 01-architecture §3.6); **прогресс-бар** заполнен на `spent / effective_limit`; пороги подсветки: **< 60%** — нормальный цвет категории, **60–85%** — внимание (жёлтый), **85–100%** — тревога (⚠, оранжевый), **≥ 100%** — перерасход (🔴, красный). Под баром — `remaining` (ост.) и `daily_pace` (~₽/день; при `remaining < 0` — «—/день»). **Carryover-бейдж** `↩ +1 200` / `↩ −800` — при ненулевом `carryover`.
- **`[+ конверт]`** — создать конверт: выбор категории (из 02-core-os §7.1 или своей) + лимит + период (по умолчанию текущий месяц).
- **Coming up** — материализованные recurring-инстансы на 14 дней вперёд (§2.8, горизонт 01-architecture §5.4): дата, title, сумма со знаком/цветом, иконка 🔁. Это прогноз, в `spent` ещё не входит (инстансы будущих дат).
- **Planned** — запланированные покупки (`planned=true`, §2.7): сумма и целевая категория. Прогноз оттока, в `spent` не входит.
- **Unbudgeted** — категории с фактическими тратами периода, но без конверта (§2.3, шаг 4): суммарный расход по категории + кнопка `[+ конверт]` (§5). Так траты «не теряются»: даже без бюджета они на виду.

Полный список финансовых сущностей с фильтрами и поиском — то, что Browser делает сырым списком (§1.3), но с финансово-специфичными фильтрами.

```
┌──────────────────────────────────────────────┐
│ ←   Транзакции          [🔍 поиск] [фильтры ▾]│
│  Фильтры: Июнь · Все категории · Расходы       │
│  ──────────────────────────────────────────── │
│  13.06  Перекрёсток      −2 340 ₽       🍔    │
│  13.06  Такси            −420 ₽         🚕    │
│  12.06  Netflix     🔁   −599 ₽         🎉    │
│  12.06  Кофе             −280 ₽         🍔    │
│  10.06  Зарплата       +165 000 ₽       💰    │
│  …                                             │
│  ◀ свайп: рекатегоризация · пометить 🔁 ▶     │
└──────────────────────────────────────────────┘
```

- **Строка** — дата, `title`, маркер recurring (🔁, у recurring-инстансов), сумма (± цвет по `direction` — расход красным, доход зелёным; 02-core-os §3.6), бейдж категории (`icon`/`color`). Маркер recurring и бейдж категории **визуально различимы**: 🔁 стоит в отдельной позиции строки (между `title` и суммой), а бейдж категории — в крайней правой колонке; они не примыкают друг к другу. В мокапе выше строка Netflix показывает 🔁 слева от суммы, а бейдж категории 🎉 — справа, поэтому recurring-маркер и категория не сливаются.
- **Фильтры** (потребитель №3 query-движка, 01-architecture §6.3 — выразимы строкой грамматики): категория (`category_ref`), направление (`direction=income|expense`), период (диапазон дат), диапазон сумм (`amount=500..2000`, 01-architecture §6.1), флаг planned (`planned=true|false`).
- **Поиск** — `search=` по `title` + `body` (по contractor/counterparty в title транзакции).
- **Свайпы:** влево — рекатегоризация (выбор категории → перенос relation `parent`, §5); вправо — пометить операцию recurring (помощь в распознавании регулярных трат: ставит `recurring=true` и предлагает завести шаблон с `recurrence`, 01-architecture §3.1). Оба действия — через executor, в журнале, обратимы.

### §3.4 Импорт CSV — слайс 2

Полный флоу импорта банковской выписки. Доступность гейтится entitlement `import.csv` (01-architecture §8; для plan `'dev'` — разрешено). Импорт инициируется с Overview (`⋮ → Импорт CSV`) или из чата («импортируй выписку»); карточка ревью — тип `import_review` в ленте чата (02-core-os §2.3).

**Шаг 1 — выбор файла.** Пользователь выбирает CSV/выписку. Файл парсится **локально**, в LLM целиком не отправляется (приватность + объём); в AI уходят только образцы строк для распознавания структуры.

**Шаг 2 — AI-парсинг структуры** (01-architecture §7.7): распознавание разделителя и **кодировки** (UTF-8 / Windows-1251 — частое для рус. банков), **колонок** (дата, сумма, описание/counterparty, направление), **формата дат** (`DD.MM.YYYY` / `YYYY-MM-DD` / …), знака суммы (отдельная колонка дебет/кредит или знак в сумме). AI предлагает маппинг колонок; пользователь может поправить, если распознано неверно.

**Шаг 3 — экран ревью.** Таблица распознанных строк со **статусом** каждой и предложенной категорией (категоризация AI по counterparty + правилам категорий из body и memory, 01-architecture §7.4):

```
┌──────────────────────────────────────────────┐
│  Импорт · выписка_май.csv · 47 строк           │
│  ┌──────────────────────────────────────────┐ │
│  │ ✓ новых 38 · ⊘ дубли 6 · ⟳ уже 3          │ │
│  └──────────────────────────────────────────┘ │
│  ──────────────────────────────────────────── │
│  ✓ 03.05 ПЯТЁРОЧКА      −1 890  [🍔 Еда ▾]    │
│  ✓ 04.05 YANDEX.TAXI    −420    [🚕 Транспорт▾]│
│  ⊘ 05.05 ОБЕД           −340    вероятный дубль│
│  ⟳ 06.05 NETFLIX        −599    уже импортир. │
│  ✓ 07.05 OZON           −3 200  [❓ выбрать ▾] │
│  …                                             │
│  ── [Снять все дубли]  [Подтвердить 38] ──────│
└──────────────────────────────────────────────┘
```

Статусы строки:
- **✓ новая** — нет совпадений, будет создана. Категория редактируется inline (выпадашка); неуверенная категоризация — `[❓ выбрать]`.
- **⊘ вероятный дубль** — совпадает по критерию §3.4.1 с любой существующей финансовой сущностью: ручной или ранее импортированной из другого файла. По умолчанию **не создаётся**; для существующей сущности заводится дополнительная строка `entity_origins` (01-architecture §4.8) с `external_id` этой строки — сущность «усыновляет» новый источник. Пользователь может переключить на «создать всё равно» для легитимных одинаковых покупок.
- **⟳ уже импортирована** — `external_id` этой строки уже зарегистрирован в `entity_origins` (уникальность `(owner_id, namespace, external_id)`, 01-architecture §4.8) — повтор того же файла/строки. Не импортируется, переключение недоступно.

**Шаг 4 — массовое подтверждение.** Inline-правка категорий до подтверждения; `[Подтвердить N]` отправляет один `batch_execute` с новым `batch_id`: создание строк, запись `entity_origins` и авто-привязки валидируются заранее и применяются одной транзакцией (01-architecture §7.8, §9.2). Ошибка любой строки не оставляет частичный импорт; повтор того же batch идемпотентен. В журнале одна группа; Undo откатывает весь импорт, включая физическое удаление origins (§3.4.1).

Кнопка **`[Снять все дубли]`** — исключить из импорта все строки со статусом ⊘. По умолчанию они и так не создаются (§3.4.1: для существующей записи атомарно заводится дополнительная строка `entity_origins`), и любую можно индивидуально переключить на «создать всё равно». Кнопка возвращает всю группу к дефолту.

**Шаг 5 — итог-карточка.** «Импортировано 38, пропущено 9 (6 дублей, 3 повтор). Без конверта: 3 — Образование, Прочее». Ссылки ведут на затронутые экраны (Unbudgeted, конкретные категории).

#### §3.4.1 Дедуп-критерий (однозначный)

Транзакция считается дублем существующей сущности, если совпадают **все три**:

1. **сумма** — `amount` и `direction` совпадают точно;
2. **дата** — `occurred_on` различается не более чем на **1 календарный день** (банк проводит операцию на день позже ручного ввода);
3. **counterparty** — похожесть строк counterparty/title выше порога (нормализованное сравнение: регистр, пробелы, типовые префиксы вроде `SBOL`/`PAYMENT` игнорируются).

Нормализация и порог byte-точны: Unicode NFKC → lowercase → `ё` заменяется на `е` → пунктуация заменяется пробелами → повторные пробелы схлопываются → leading service-токены `sbol`, `payment`, `card`, `purchase`, `oplata` удаляются. Затем считается `max(normalized Levenshtein similarity, token Jaccard)`; совпадение выполняется при score `≥ 0.85`. При наличии одинакового стабильного bank transaction ID пункт 3 считается выполненным независимо от текста. Порог и список service-токенов — shared-константы SQL/JS, покрытые общими фикстурами.

**Provenance строки** хранится не в самой сущности, а отдельной строкой таблицы `entity_origins` (01-architecture §4.8): `namespace = "csv:<источник>"` (выписка/банк), а `external_id` идентифицирует конкретную строку конкретного источника, а не саму экономическую операцию:

```
external_id = sha256(file_content_hash + ":" + zero_based_row_index + ":" + normalized_row)
```

Повторное чтение byte-identical файла даёт те же `external_id`. Разные, в том числе пересекающиеся, выписки дают другой `external_id` (или другой `namespace`), поэтому дополнительно всегда применяется содержательный критерий (1)+(2)+(3) — он не заменяется таблицей происхождения, а дополняет её. Если банк предоставляет стабильный transaction ID, он включается в `normalized_row` и получает максимальный вес при сравнении. **Жёсткая уникальность на уровне БД** — `UNIQUE (owner_id, namespace, external_id)` (01-architecture §4.8) — гарантирует, что одна и та же строка выписки не может быть импортирована дважды: повторная вставка того же `(namespace, external_id)` отклоняется физически, ещё до прикладной проверки.

Логика:

- Перед созданием строки считается её `external_id`.
- Если в `entity_origins` уже есть строка с этим `(owner_id, namespace, external_id)` → статус **⟳ уже импортирована** (проверка — поиск по уникальному индексу, не скан массива).
- Иначе дубль ищется по критерию (1)+(2)+(3) среди **всех** финансовых сущностей подходящего окна, независимо от истории импорта → статус **⊘ вероятный дубль**. При подтверждении дефолтного варианта для найденной сущности создаётся **ещё одна** строка `entity_origins` с этим `namespace`/`external_id` — у сущности может быть несколько origins: одна и та же операция бывает видна сразу в двух пересекающихся выписках, не только в ручном вводе плюс одной выписке.
- Иначе → **✓ новая**.

Если пользователь выбирает «создать всё равно», создаётся отдельная транзакция со своей строкой `entity_origins` (`entity_id` указывает на новую сущность); это разрешает две реальные одинаковые покупки. Таким образом, уникальность `external_id` гарантирует идемпотентность одного файла, а содержательный критерий обнаруживает пересечения разных файлов без необратимого автослияния.

**Undo импорта физически удаляет** созданные строки `entity_origins` (01-architecture §4.8), а не архивирует их вместе с транзакциями: после отмены та же самая выписка снова видит все свои строки как новые (`external_id` больше не занят в этом `namespace`) и импортируется заново без ложных ⟳.

### §3.5 Rollover — слайс 2

Перенос остатков и создание конвертов нового периода (carryover-логика §2.6).

**Триггер** — первое открытие Budget в новом месяце (предыдущий месяц закрылся, есть конверты прошлого периода без конвертов-преемников в текущем). Также вызывается вручную («настрой бюджеты на июль», §4) и из `⋮ → Rollover`. Поскольку конверты появляются только в слайсе 2 (§3.1), первый Rollover в жизни продукта естественно случается на границе первого календарного месяца после запуска Budget — раньше нет конвертов-предшественников, которые можно переносить; это и есть случай «Первый месяц без истории» ниже.

```
┌──────────────────────────────────────────────┐
│  Новый месяц: Июль 2026                        │
│  AI предложил бюджеты по истории. Проверьте.   │
│  ──────────────────────────────────────────── │
│  Категория    │ Факт июня │ Carryover│ Лимит  │
│  ─────────────┼───────────┼──────────┼─────── │
│  🍔 Еда       │  28 800   │  +1 200  │[30 000▾]│
│  🚕 Транспорт │  10 100   │  −1 100  │[ 9 000▾]│
│  🏠 Жильё     │  45 000   │   0      │[45 000▾]│
│  🎉 Развлеч.  │   3 800   │  +6 200  │[10 000▾]│
│  …                                             │
│  ── [Обнулить переносы]  [Создать 12 конв.] ──│
└──────────────────────────────────────────────┘
```

- **Таблица** по категориям: факт прошлого месяца (`spent` закрытого конверта), `carryover` (= `remaining` прошлого, §2.6), предложенный AI лимит (по истории трат, 01-architecture §7.7), поле правки лимита. Carryover можно обнулить покатегорийно (тап по значению) или целиком (`[Обнулить переносы]`).
- **Первый месяц без истории** — AI нечего предлагать; вместо предложения спрашивает доход и оценки (§5, edge case).
- **`[Создать N конв.]`** — один `batch_execute` создаёт конверты нового периода с подтверждёнными лимитами и carryover. `batch_id` обеспечивает идемпотентность, транзакция исключает частичный rollover; одна запись журнала и Undo откатывают всю группу. Произвольные конверты (§2.9) в rollover не участвуют.

### §3.6 Quick-add бар — слайс 2

Быстрый ввод транзакции без чата — бар внизу Overview и экрана категории. Отличается от quick-capture Browser (02-core-os §3.7, без интерпретации): quick-add Budget **структурирует** ввод в `orbis/financial`.

```
┌──────────────────────────────────────────────┐
│  [−расход] [+доход]      Сумма:  3 4 0         │
│  🍔 Еда  🚕 Транспорт  🎉 Развлеч.  🏠 Жильё  │
│  [title (опц.)…]                    [Записать] │
│  ┌─────────┬─────────┬─────────┐               │
│  │  1 2 3  │  4 5 6  │  7 8 9  │  ⌫  · 0      │
│  └─────────┴─────────┴─────────┘               │
└──────────────────────────────────────────────┘
```

- **Сумма-клавиатура** — числовой ввод (целые/десятичные, как fast-path 01-architecture §7.5).
- **4–5 последних категорий** — пилюли недавно использованных категорий (быстрый выбор без списка); тап по пилюле задаёт `category_ref`. Полный выбор — раскрытием.
- **Title** — опционален (counterparty / комментарий); пусто → title из имени категории + суммы.
- **Переключатель income/expense** — задаёт `direction`.
- **`[Записать]`** — создаёт сущность с `orbis/financial`, `occurred_on` = сегодня в таймзоне пользователя, затем выполняет авто-привязку к конверту (§2.3); результат — `entity_card` с Undo. На экране категории `category_ref` предзадан. Правка даты доступна inline в карточке/detail и повторно запускает выбор конверта атомарным batch'ем.

## §4. AI-сценарии

Эталонные сценарии «реплика → поведение». Механика тулов — 01-architecture §9.2, журнала — §7.8, fast-path — §7.5, модель и транспорт чата — §7.7. Все правки обратимы через журнал (Undo). Принадлежность к слайсам (00-product §9): сценарии **1, 6 и 7** доступны со **слайса 1** — им не нужен Budget-view (1 — fast-path; 6 и 7 — агрегация чатом через `entity_query`, без конвертов и Overview); сценарии **2–5** читают или правят конверты и Rollover, то есть предполагают Budget-view — **слайс 2**.

**1. «обед 340»** (fast-path, 01-architecture §7.5)
→ Мгновенная `entity_card` «⚡ без AI» (02-core-os §2.5): `orbis/financial`, `expense`, `amount="340.00"`, `occurred_on` = сегодня, `category_ref` = «Еда» (резолв по alias). Авто-привязка к конверту «Еда» текущего месяца (§2.3). Карточка показывает **остаток конверта еды** после записи: «Обед 340 ₽ → Еда · осталось 8 060 ₽» (в слайсе 1 — без остатка конверта, 00-product §7). Работает офлайн, LLM не вызывается.

**2. «получил 150к»** / «+150000 зарплата»
→ `orbis/financial`, `income`, `amount="150000.00"`, `occurred_on` = сегодня, категория «Зарплата» (alias). Поскольку это доход в начале периода, AI добавляет **suggestion-чип / предложение распределить**: карточка «Распределить 150 000 ₽ по конвертам?» открывает Rollover-подобный экран аллокации (§3.5) с предложением лимитов по истории. Распределение — подтверждаемое, не автоматическое.

**3. «могу позволить кроссовки за 8000?»**
→ Вопросительная форма — fast-path уступает LLM (01-architecture §7.5). AI считает запас по дискреционным категориям:

```
discretionary_remaining = SUM(remaining конвертов,
                              category.spend_class = discretionary)
future_outflows = множество уникальных financial-сущностей,
                  где direction=expense, planned=true,
                  сегодня < occurred_on <= period_end
доступно = discretionary_remaining − SUM(future_outflows.amount)
ответ: да, если доступно ≥ 8000; иначе нет, с числом
```

`future_outflows` уже включает будущие recurring-инстансы, потому что они материализуются с `planned=true` (§2.8): отдельное суммирование recurring запрещено, чтобы не получить двойной вычет. Категории без `spend_class` не включаются молча — карточка просит сначала классифицировать их как fixed/discretionary. Карточка `query_result` (02-core-os §2.3): «Да — после кроссовок останется 12 400 ₽ свободных» или «Нет — свободно 5 200 ₽, не хватает 2 800 ₽; перенести из Развлечений?».

**4. «переведи 3000 с транспорта на еду»**
→ Один `batch_execute` атомарно правит `limit` **двух** конвертов текущего периода: «Транспорт» (`limit −= 3000`) и «Еда» (`limit += 3000`). `spent` не трогается. Ошибка одной правки откатывает обе; карточка и Undo относятся ко всей группе.

**5. «настрой бюджеты на июль»**
→ Запускает Rollover-флоу (§3.5) **из чата**: AI отвечает картой-таблицей предложенных конвертов (факт / carryover / лимит); подтверждение `[Создать]` выполняет один атомарный `batch_execute`. Если истории нет — спрашивает доход и оценки (§5).

**6. «сколько трачу на фитнес вообще?»** (cross-aspect)
→ Выходит за один конверт: AI вызывает `entity_query` по тегу/категории, относящейся к фитнесу (`aspect=orbis/financial, tags=fitness` или по `category_ref` фитнес-категории, если заведена), агрегирует `amount` за запрошенный период и отвечает `query_result`. Это связность графа (00-product §4): траты на фитнес могут лежать в разных категориях (Здоровье, Подписки, Одежда) — тег/cross-aspect-запрос собирает их вместе, чего конверт одной категории не покрывает. Если запрос «за всё время» — глубина ограничена entitlement `history.depth_days` (для `'dev'` — без ограничения, 01-architecture §8).

**7. «что по бюджету?»** (слайс 1 — до появления Budget-view)
→ Overview (§3.1) в слайсе 1 ещё нет, но вопрос уже осмысленный: AI вызывает `entity_query` по `aspect=orbis/financial, occurred_on=<текущий месяц>` (без конвертов — их тоже пока нет), группирует найденные суммы по `category_ref` и отвечает `query_result` короткой агрегацией: «В этом месяце: Еда −18 400 ₽, Транспорт −6 200 ₽, …». Со слайса 2 тот же вопрос получает более точный ответ карточкой конверта — остаток, темп, прогресс (§3.1); данные не меняются, меняется только линза (§1.3).

## §5. Edge cases

| Ситуация | Поведение |
|---|---|
| **Транзакция в категории без конверта** | Транзакция валидна (несёт `category_ref`), но без relation `parent` к конверту → попадает в секцию **Unbudgeted** на Overview (§3.1) с кнопкой `[+ конверт]`. Создание конверта на этот период авто-привязывает (§2.3) накопленные транзакции категории, и они входят в `spent`. Траты не теряются, но и не искажают несуществующий бюджет |
| **Возврат средств** (рефанд по покупке) | Создаётся `orbis/financial` `direction=income` с тем же `category_ref` и привязкой `parent` к тому же конверту. **Дефолт:** при создании такого income Budget **НЕ трогает исходную expense автоматически** — income отражается только в балансе периода (§2.5, income растёт сразу), а `spent` конверта не уменьшается (формула §2.2 учитывает только `expense`). Чтобы возврат «вернул деньги в конверт», нужно **сторнирование** — уменьшение/архивация исходной expense-транзакции (правка/рекатегоризация, в журнале); это **явное действие пользователя**. AI может его **предложить** (карточка «зачесть возврат в счёт конверта?»), но не выполняет молча. MVP: для точного «возврата в конверт» правится исходная транзакция; чистый income-возврат виден в балансе |
| **Рекатегоризация** | Один `batch_execute` обновляет `category_ref`, удаляет старый budget-parent и при наличии создаёт новый. Инвариант «не более одного budget-parent» проверяется внутри транзакции; частичное состояние невозможно, Undo откатывает всю группу. **Оба конверта пересчитываются** немедленно. Если новой категории нет конверта на `occurred_on`, транзакция уходит в Unbudgeted |
| **Первый месяц без истории** | AI нечего предлагать по истории (нет прошлых конвертов и трат). Rollover/«настрой бюджеты» (§3.5, §4) вместо предложения **спрашивает**: ожидаемый месячный доход и оценки по ключевым категориям (диалог в чате или поля в форме). Из ответов создаются стартовые конверты. Дальше история накапливается, и следующий rollover предлагает по факту |
| **Повторный импорт того же CSV** | Полный дедуп по `entity_origins` (§3.4.1, 01-architecture §4.8): `external_id` каждой строки для этого `namespace` уже зарегистрирован → все строки получают статус **⟳ уже импортирована**, ничего не создаётся. Пересекающаяся выписка из другого файла дополнительно сверяется со всеми ранее импортированными операциями и получает ⊘, а не создаёт молчаливые дубли |
| **Чужая валюта** (операция не в валюте периода) | MVP: транзакция **сохраняется как есть** (своя `currency`, 01-architecture §3.3) и видна в списках транзакций (§3.3) с **бейджем-предупреждением** валюты. В агрегаты периода (`spent` конверта §2.2, баланс §2.5) она **НЕ входит** — обе формулы фильтруют по валюте периода. Пересчёт по курсу — Future (multi-currency, 00-product §10). Так чужая валюта не искажает бюджет и не теряется из учёта |
| **Удаление категории с историей** | Прямое удаление **запрещено** (у категории есть конверты и транзакции по relation `parent`, ссылки `category_ref`). Вместо удаления — **архивация с переносом**: AI/UI предлагает перенести транзакции и конверты в другую категорию (массовая рекатегоризация — перенос `parent` и обновление `category_ref`, в журнале) ИЛИ оставить их в Unbudgeted (отвязать от конвертов). После переноса категория архивируется (`archived=true`, мягкое удаление — 01-architecture §2.1); история сохранена под новой категорией или как Unbudgeted, ничего не потеряно |

## §6. Бейджи и интеграция

### §6.1 Бейдж вкладки Budget — со слайса 2

Бейдж вкладки Budget (notification policy — 02-core-os §1.5, который делегирует значение сюда) = **число конвертов текущего периода в зоне тревоги и перерасхода**: `spent > 85% × effective_limit` — то есть оранжевые (⚠, тревога 85–100%) и красные (🔴, перерасход ≥ 100%) конверты вместе по порогам §3.1. Нет таких конвертов — нет бейджа. Значение пересчитывается клиентским движком при изменении транзакций (производное, не хранится).

«Со слайса 2» здесь — нижняя граница: сам бейдж не может появиться раньше конвертов (§3.1), которых до Overview не существует. Визуальная полировка бейджей вкладок как отдельная сквозная фича (анимации, единый стиль across вкладок) — слайс 3 по роадмапу (00-product §9); это разные грани одной фичи на разных слайсах, а не противоречие.

### §6.2 Интеграция с ядром

Budget — линза поверх графа, поэтому его данные живут и в других views без дублирования:

- **Agenda** (02-core-os §4): финансовые сущности видны в дневных секциях только при наличии `orbis/schedule`; одного `orbis/financial.occurred_on` или task-`due_date` недостаточно. Покупка-задача с одним `due_date` живёт в Daily Planning/Upcoming, а после добавления `orbis/schedule` появляется и в Agenda. Одна сущность, разные линзы без скрытого преобразования дат.
- **Browser** (02-core-os §3): все финансовые сущности видны в Browser сырым списком (native-рендеринг §3.6), фильтруются и ищутся наравне с задачами и заметками. Категории-сущности и конверты — тоже обычные сущности в Browser.
- **Chat** (02-core-os §2): создание и правка транзакций, импорт (`import_review`), вопросы-агрегации (`query_result`) — всё через чат; карточки с Undo.
- **Обратимость**: каждая правка Budget (создание транзакции/конверта, привязка, рекатегоризация, перенос лимита, импорт, rollover) проходит через executor и пишется в **журнал действий** (01-architecture §7.8) — «отмени последнее» работает из чата для любого из этих действий, включая групповые (импорт, rollover) и действия внешних MCP-агентов (01-architecture §9.3).

## §7. Приёмочные проверки финансового контура — со слайса 2

Все шесть проверок ниже предполагают полный финансовый контур — конверты, Overview, CSV-импорт, recurring/Coming up, planned→fact — весь набор приходит в **слайсе 2** (00-product §9); до слайса 2 из графа проверяем только сам факт fast-path-накопления истории (§1.2).

1. **Исторический импорт:** выписка с `occurred_on=2026-05-31`, импортированная 13 июня 2026 года, входит в майский баланс/конверт и не входит в июньский; `created_at` на результат не влияет.
2. **Recurring:** инстанс на завтра материализован с `planned=true` и не входит в `spent`; в дату операции transition выполняется один раз даже с двух устройств, после чего операция входит в факт. Архивированный заранее инстанс не постится.
3. **Пересечение конвертов:** месячный и отпускной конверты одной категории созданы в разном порядке. В обоих вариантах транзакция внутри отпуска имеет ровно один parent — более узкий конверт; после его архивации атомарно возвращается месячному.
4. **Импорт пересекающихся файлов:** повтор byte-identical файла даёт только статусы ⟳; та же операция из другого файла определяется как ⊘ среди ранее импортированных записей. Выбор «создать всё равно» сохраняет две реальные одинаковые покупки и остаётся идемпотентным при повторе файла.
5. **Affordability:** будущий recurring-инстанс присутствует в `future_outflows` один раз, хотя имеет и `recurring=true`, и `planned=true`; расходная категория без `spend_class` приводит к явному запросу классификации, а не к молчаливому ответу.
6. **План → факт:** подтверждение покупки обновляет `planned=false` и `occurred_on` на выбранную фактическую дату одним batch; конверт выбирается по этой дате, Undo восстанавливает план и прежнюю привязку целиком.
