### §1.2 Граница документа

PRD фиксирует **контракты и поведение** — схемы данных, протоколы, грамматику запросов, реестр тулов, инварианты, — но **не реализацию**. Структура кода, выбор библиотек, сигнатуры процедур, организация модулей живут в implementation-планах фаз и в самом коде. Если деталь не влияет на наблюдаемое поведение системы или на совместимость данных — её здесь нет намеренно.

### §1.3 Сквозные принципы

**Принцип 1: данные существуют до views.** Сущности и их связи — первичны; views (Budget, Agenda, Browser, будущие Fitness и Habits) — линзы поверх одного графа. Удаление или замена view не теряет ни одной записи: расход, заведённый через Budget, останется сущностью с `orbis/financial` и будет виден в Browser и в любых будущих views. Следствие для проектирования: ни один view не имеет собственного хранилища данных — только настройки отображения (`user_settings.viewPreferences`).

**Принцип 2: одна грамматика — шесть потребителей.** Одна грамматика запросов (§6), один движок с одним SQL-бэкендом (Postgres) и шесть потребителей — от `{{query:...}}`-блоков в body до AI-тула `entity_query`; любая новая фильтрующая возможность добавляется в грамматику один раз и автоматически доступна всем потребителям. Полный список потребителей — §6.3.

**Принцип 3: правила слоёв сущности.**

- **meta — только AI-извлечённое** до структурирования в аспекты. Никакого application state: координаты досок, настройки отображения, кэши — не сюда. Исключений нет: исходная фраза ввода живёт в `chat_messages.content` (§4.6), а не в meta сущности; provenance импорта — отдельная таблица `entity_origins` (§4.8), а не зарезервированный ключ meta.
- **body — текст и живые запросы.** Markdown, ссылки на сущности, `{{query:...}}`-блоки. Smart lists — это сущности с query-блоками в body, а не отдельный механизм.
- **aspects — всё типизированное**, включая данные views: всё, что имеет схему в реестре, живёт в aspects и валидируется по этой схеме.

## §2. Модель данных: сущность

### §2.1 Четыре слоя

Сущность — универсальная единица графа. Все домены (задачи, финансы, заметки, бюджеты, память AI) выражаются одной структурой из четырёх слоёв:

```
┌──────────────────────────────────────────────────────────────────┐
│  СУЩНОСТЬ                                                        │
│                                                                  │
│  ┌ 1. Core ────────────────────────────────────────────────────┐ │
│  │  id (UUIDv7, генерируется клиентом), title, emoji,          │ │
│  │  created_at, updated_at, archived                           │ │
│  └─────────────────────────────────────────────────────────────┘ │
│  ┌ 2. Body ────────────────────────────────────────────────────┐ │
│  │  markdown + [[entity:uuid|text]] + {{query:...}}            │ │
│  │  body_refs[] — UUID, извлечённые из body при сохранении     │ │
│  └─────────────────────────────────────────────────────────────┘ │
│  ┌ 3. Tags + Meta ─────────────────────────────────────────────┐ │
│  │  tags[] — нормализованные ярлыки (от AI и пользователя)     │ │
│  │  meta{} — AI-извлечённые ключ-значения до структурирования  │ │
│  └─────────────────────────────────────────────────────────────┘ │
│  ┌ 4. Aspects ─────────────────────────────────────────────────┐ │
│  │  { "orbis/task": {...}, "orbis/financial": {...} }          │ │
│  │  типизированные данные по схемам из реестра аспектов        │ │
│  └─────────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────────┘
```

- **Core** — минимум, который есть у всего: `id` (UUIDv7, генерируется на клиенте — время в префиксе даёт естественную хронологическую сортировку и нужно retry-буферу fast-path-ввода, §5.3, для идемпотентной досылки ещё не подтверждённых create-мутаций), `title`, опциональный `emoji`, таймстемпы, флаг `archived` (мягкое удаление; жёсткого удаления сущностей в MVP нет).
- **Body** — markdown с двумя расширениями: inline-ссылка `[[entity:uuid|текст]]` и живой query-блок `{{query:...}}` (§6). Из body при каждом сохранении извлекается `body_refs[]` — массив UUID упомянутых сущностей, для быстрых backlink-запросов без парсинга markdown.
- **Tags + Meta** — прото-аспектный слой. Пишется всегда, даже когда данные уже структурированы в аспекты: теги дают сквозную фильтрацию, meta — сырьё для ретроактивной миграции (§3.10).
- **Aspects** — JSONB-карта `aspect_id → данные`, валидируемая по JSON-схемам из реестра (§3). Одна сущность может иметь несколько аспектов: «купить кроссовки 8000₽ к субботе» = одна сущность с `orbis/task` + `orbis/financial` + `orbis/schedule`.

### §2.2 Что является сущностью, а что нет

| Сущности (живут в `entities`) | Не сущности (отдельные таблицы) |
|---|---|
| Задачи | `user_settings` |
| Расходы и доходы (транзакции) | `aspect_definitions` (реестр аспектов) |
| Заметки | История чатов (`chat_threads`, `chat_messages`) |
| Проекты | Метеринг (`ai_usage`) |
| Конверты бюджета | Provenance импорта (`entity_origins`) |
| Категории | |
| Memory-факты и правила AI | |
| Smart lists (query-блоки в body) | |
| Recurring-шаблоны и их инстансы | |
| Чекины привычек (future) | |

### §2.3 Правило идентичности

**Сущность — то, на что пользователь ссылается как на вещь с собственной судьбой.** У неё есть имя, её можно открыть, связать с другими, заархивировать, обсудить в её треде. Категория «Еда» — сущность: пользователь переименовывает её, пишет в её body правила категоризации, смотрит её транзакции. Настройка `weekStartDay` — не сущность: это параметр поведения системы, у него нет собственной жизни в графе. Этим правилом проверяется каждый новый тип данных: если объект нужно показывать в Browser, связывать relations или обсуждать — это сущность; если это конфигурация или журнал — это таблица.

### §2.4 Сквозной пример

Сущность из ввода «купить кроссовки 8000₽ к субботе» — все четыре слоя в одном JSON (служебный столбец `owner_id` опущен; поля аспектов — по схемам §3.2 и §3.3):

```json
{
  "id": "019ea8b1-4778-7f3d-9a5c-6a521fa1cc24",
  "title": "Купить кроссовки",
  "emoji": "👟",
  "body": "К субботней пробежке. Модель выбрана в [[entity:019e4466-1000-7e07-b5d4-64be9721da51|Wishlist: бег]].",
  "body_refs": ["019e4466-1000-7e07-b5d4-64be9721da51"],
  "tags": ["task", "expense", "running"],
  "meta": {},
  "aspects": {
    "orbis/task": {
      "status": "planned",
      "priority": "medium",
      "due_date": "2026-06-13"
    },
    "orbis/schedule": {
      "start_at": "2026-06-13T00:00:00+03:00",
      "all_day": true,
      "timezone": "Europe/Moscow"
    },
    "orbis/financial": {
      "amount": "8000.00",
      "currency": "RUB",
      "direction": "expense",
      "category_ref": "019d48ea-2e00-7a52-876a-c301529b0456",
      "occurred_on": "2026-06-13",
      "planned": true
    }
  },
  "created_at": "2026-06-08T19:24:11Z",
  "updated_at": "2026-06-08T19:24:11Z",
  "archived": false
}
```

Здесь: `body_refs` извлечён из `[[entity:...]]`-ссылки в body; `category_ref` указывает на категорию-сущность «Одежда»; `orbis/schedule` делает покупку видимой в Agenda на субботу; `occurred_on` задаёт ожидаемую дату финансового события; `planned: true` не даёт списать сумму до подтверждения факта (§3.3, §3.5). `meta` пуст: всё, что AI извлёк из фразы, уже структурировано в аспекты; исходная фраза «купить кроссовки 8000₽ к субботе» сохранена как `content` инициировавшего создание сообщения чата (§4.6), а не в meta сущности.

## §3. Аспекты MVP

Семь встроенных аспектов. Схема каждого хранится в реестре `aspect_definitions` (§4.3) как JSON Schema; ниже — нормативное содержание этих схем. Все семь активны для каждого пользователя с первого дня: passive-состояния в v3 нет.

Условные обозначения: ✓ в колонке Req — поле обязательно при наличии аспекта на сущности.

### §3.1 `orbis/schedule` — время

| Поле | Тип | Req | Описание |
|---|---|---|---|
| `start_at` | timestamp (ISO 8601) | ✓ | Начало события / момент привязки ко времени |
| `end_at` | timestamp (ISO 8601) | | Конец события; если задан — должен быть ≥ `start_at` |
| `duration_min` | integer | | Длительность в минутах (альтернатива `end_at`; если заданы оба — приоритет у `end_at`) |
| `all_day` | boolean | | Событие на весь день (default `false`) |
| `recurrence` | object | | Правило повторения: `{freq: daily\|weekly\|monthly, interval: int, byweekday?: string[], until?: date}`. Наличие = сущность является recurring-шаблоном |
| `location` | string | | Место |
| `timezone` | string (IANA) | | Таймзона события; по умолчанию — таймзона пользователя |

`tag_mappings`: `["schedule", "event", "meeting", "appointment"]`

Recurrence живёт только здесь — единый источник истины «когда повторяется» для любых доменов (платежи, тренировки, привычки). Инстансы порождаются от шаблона связью `derived_from` с детерминированными ID (§5.4).

### §3.2 `orbis/task` — задача

| Поле | Тип | Req | Описание |
|---|---|---|---|
| `status` | enum: `inbox` \| `planned` \| `in_progress` \| `waiting` \| `done` \| `cancelled` | ✓ | Состояние задачи; default при создании — `inbox` |
| `priority` | enum: `low` \| `medium` \| `high` | | Приоритет |
| `due_date` | date (ISO 8601) | | Срок |
| `completed_at` | timestamp | | Момент завершения; проставляется при переходе в `done`, при откате задачи из `done` — очищается |
| `effort_min` | integer | | Оценка трудозатрат в минутах |
| `waiting_for` | string | | Кого/чего ждём (для `status=waiting`) |

`tag_mappings`: `["task", "todo", "action", "deadline"]`

### §3.3 `orbis/financial` — финансовая операция

| Поле | Тип | Req | Описание |
|---|---|---|---|
| `amount` | decimal-string > 0 | ✓ | Сумма в единицах валюты, канонический формат base-10 без экспоненты (`"340.00"`, `"127.50"`). Всегда положительная; знак задаёт `direction` |
| `currency` | string (ISO 4217) | | Валюта; default — `user_settings.defaultCurrency` |
| `direction` | enum: `income` \| `expense` | ✓ | Направление операции |
| `category_ref` | uuid | ✓ | Ссылка на **категорию-сущность** (с аспектом `orbis/category`) |
| `occurred_on` | date (ISO 8601) | условно ✓ | Дата фактической или ожидаемой операции в таймзоне пользователя. Обязательна для фактов, planned-операций и recurring-инстансов; отсутствует только у recurring-шаблона |
| `planned` | boolean | | `true` = запланированная операция (ещё не факт): видна в прогнозах и секции planned, не списывается из конверта как фактический расход. Default `false` |
| `recurring` | boolean | | `true` = сущность является шаблоном регулярной операции (период — в `orbis/schedule.recurrence` той же сущности); инстансы связываются `derived_from` |
| `payment_method` | string | | Способ оплаты (карта, наличные, счёт) |
| `counterparty` | string | | Контрагент (магазин, работодатель, человек) |

`tag_mappings`: `["expense", "income", "payment", "cost"]`

Категория — ссылка на сущность (`category_ref`), а не строка: переименование категории не требует массового обновления транзакций, категория несёт иерархию (через `parent`), body-правила и собственный тред.

**Инвариант recurring.** Канонический источник повторяемости — `orbis/schedule.recurrence` на сущности-шаблоне (§3.1). `orbis/financial.recurring=true` — информационный маркер для прогнозов (Coming up, прогноз баланса), валидный только на шаблоне с `recurrence` или на его инстансах (связь `derived_from`). Шаблон не имеет `occurred_on`; каждый инстанс получает `occurred_on` из даты инстанса. Будущий инстанс создаётся с `planned=true`; в дату `occurred_on` локальный системный transition атомарно меняет неархивный инстанс на `planned=false` и выполняет авто-привязку к конверту. Его `batch_id` детерминирован: `uuidv5(ORBIS_NAMESPACE, "post-financial:<instance_id>")`, поэтому конкурентные выполнения перехода не создают повторных действий. Несогласованность — `recurring=true` без `recurrence` и без `derived_from`, либо не-шаблон без `occurred_on` — отклоняется валидацией при записи.

**Денежная арифметика.** `amount` и все денежные поля других аспектов хранятся JSON decimal-строками со схемой `{type: "string", format: "decimal"}` и вычисляются библиотекой точной base-10 арифметики; преобразование в IEEE-754 `number` запрещено. Масштаб при сохранении нормализуется по minor-unit валюты (для RUB/USD/EUR — 2 знака), правила округления — half-away-from-zero только на границе ввода/импорта. Query-движок принимает числовые литералы как decimal и сравнивает их без потери точности.

### §3.4 `orbis/note` — заметка (маркер)

| Поле | Тип | Req | Описание |
|---|---|---|---|
| `content_type` | enum: `markdown` \| `plain` \| `checklist` | | Тип содержимого; default `markdown` |
| `pinned` | boolean | | Закреплена в списках заметок; default `false` |

`tag_mappings`: `["note", "thought", "idea", "journal"]`

Содержимое заметки живёт в `body` (как у любой сущности); аспект — маркер «главное назначение этой сущности — текст», делающий её фильтруемой как заметку. Задача с богатым body не обязана иметь `orbis/note`.

### §3.5 `orbis/budget` — конверт бюджета

| Поле | Тип | Req | Описание |
|---|---|---|---|
| `category_ref` | uuid | ✓ | Категория-сущность, к которой относится конверт |
| `limit` | decimal-string ≥ 0 | ✓ | Лимит конверта на период; денежная арифметика — по правилу §3.3 |
| `currency` | string (ISO 4217) | | Валюта; default — `user_settings.defaultCurrency` |
| `period_start` | date (ISO 8601) | ✓ | Начало периода (включительно) |
| `period_end` | date (ISO 8601) | ✓ | Конец периода (включительно) |
| `carryover` | decimal-string | | Остаток, перенесённый из предыдущего периода (может быть отрицательным — перерасход). Default `"0.00"`. Доступно в периоде = `limit + carryover − spent` |

`tag_mappings`: `["budget", "envelope", "limit"]`

**`spent` не хранится** — потраченное вычисляется из транзакций (сущностей с `orbis/financial`, `planned=false`, `occurred_on ≤ сегодня`, в валюте конверта), привязанных к конверту связью `parent` (конверт — родитель транзакции). Это единственный источник истины: правка, перенос или архивация транзакции немедленно и без рассинхрона меняет состояние конверта. Любое отображаемое значение `spent` — вычисление на лету или кэш по правилу 3 (§10).

Пример пары «конверт + транзакция» (показаны значимые слои; остальные поля — как в §2.4):

```json
{
  "id": "019e81fb-7180-7306-ad8f-842ad8efb58d",
  "title": "Еда — июнь 2026",
  "tags": ["budget"],
  "aspects": {
    "orbis/budget": {
      "category_ref": "019d48ea-4188-765d-8e96-93a0ad9c262a",
      "limit": "30000.00",
      "currency": "RUB",
      "period_start": "2026-06-01",
      "period_end": "2026-06-30",
      "carryover": "-1200.00"
    }
  }
}
```

```json
{
  "id": "019eac7a-5980-7fa8-8425-8b14dfcbba25",
  "title": "Обед",
  "tags": ["expense", "food"],
  "aspects": {
    "orbis/financial": {
      "amount": "340.00",
      "currency": "RUB",
      "direction": "expense",
      "category_ref": "019d48ea-4188-765d-8e96-93a0ad9c262a",
      "occurred_on": "2026-06-13",
      "planned": false
    }
  }
}
```

Их связывает relation `parent` (§4.2): `source_id` = конверт `019e81fb-…`, `target_id` = транзакция `019eac7a-…` — конверт является родителем транзакции. Оба `category_ref` указывают на одну категорию-сущность «Еда». `spent` на лету = сумма `amount` фактических транзакций-детей в валюте конверта, дата которых уже наступила; доступно = `30000.00 + (−1200.00) − spent`.

### §3.6 `orbis/category` — категория

| Поле | Тип | Req | Описание |
|---|---|---|---|
| `icon` | string | | Иконка (имя иконки или emoji) |
| `color` | string (hex) | | Цвет категории — используется конвертами и графиками |
| `aliases` | string[] | | Синонимы для резолва ввода, включая русские. Пример: `["еда", "продукты", "обед", "food", "groceries", "lunch"]` |
| `spend_class` | enum: `fixed` \| `discretionary` | | Класс расходной категории для расчёта доступного свободного бюджета. Для доходных категорий не задаётся; отсутствие у расходной категории означает «не классифицирована» и требует выбора пользователя перед включением в расчёт |

`tag_mappings`: `["category"]`

### §3.9 Соглашение об именах meta-ключей

Правило для AI-инструкций (фиксируется в system prompt и в `ai_instructions` аспектов): **имена ключей в `meta` обязаны совпадать с именами полей аспектов**. AI, извлекая структуру из «потратил 500 на такси» до/помимо аспекта, пишет `meta: {amount: "500.00", direction: "expense"}` — те же имена и типы, что в схеме `orbis/financial`. Это делает ретроактивную миграцию (§3.10) механической операцией, а не угадыванием.

### §3.10 Ретроактивная миграция

Механизм подключения будущих аспектов к уже накопленным данным. Когда в реестре появляется новый аспект (например, `orbis/fitness`), исторические сущности, описывавшие этот домен тегами и meta, подключаются к нему так:

1. **Поиск кандидатов** — запрос по `tag_mappings` нового аспекта: все сущности, у которых есть хотя бы один из маппинг-тегов и нет самого аспекта.
2. **Маппинг meta → поля аспекта** — благодаря соглашению §3.9 перенос прямой: `meta.amount → aspect.amount`. Ключи meta, не входящие в схему аспекта, остаются в meta.
3. **Валидация по схеме** — собранный объект аспекта проверяется JSON-схемой из реестра (обязательные поля, типы, enum-значения).
4. **Attach** — валидные сущности получают аспект; перенесённые ключи удаляются из meta.
5. **Невалидные** — сущности, не прошедшие валидацию (нет обязательного поля, неверный тип), получают флаг ручного разбора (тег `needs-review`). Пользователь видит их в **Browser через фильтр «требуют разбора»** и дозаполняет вручную или через чат («у этих пяти тренировок не хватает даты — расставь по описанию»).

Миграция идемпотентна: повторный запуск пропускает сущности, у которых аспект уже есть.

| Столбец | Тип | Описание |
|---|---|---|
| `id` | uuid PK | Генерируется клиентом |
| `source_id` | uuid NOT NULL | FK → `entities.id`, ON DELETE CASCADE |
| `target_id` | uuid NOT NULL | FK → `entities.id`, ON DELETE CASCADE |
| `relation_type` | text NOT NULL | `parent` \| `blocks` \| `related_to` \| `derived_from` |

Семантика типов: `parent` — source является родителем target (проект → задача, конверт → транзакция, категория → подкатегория); `blocks` — source блокирует target; `related_to` — мягкая связь (явная или автоматическая из body_refs); `derived_from` — target порождён от source (recurring-шаблон → инстанс).

Constraints: `CHECK (source_id != target_id)` (запрет связи с самим собой); unique index `UNIQUE (source_id, target_id, relation_type)` — глобальная уникальность связи среди всех строк таблицы. Удаление связи — обычный `DELETE`; пересоздание удалённой связи — новая строка с новым `id`, как и любая другая вставка.

**Доменный инвариант финансов.** У сущности с `orbis/financial` может быть не более одного `parent`, чей source имеет `orbis/budget`: одна транзакция списывается максимум из одного конверта. `relation_create` проверяет этот инвариант атомарно и отклоняет вторую связь; перенос выполняется batch'ем «удалить старую + создать новую» (§9.2). Другие `parent`-связи финансовой сущности, например к проекту, инвариантом не запрещены.

**Инвариант ацикличности `blocks`.** Граф `blocks`-связей обязан оставаться ацикличным: сервер отклоняет `relation_create` (§9.2), замыкающую цикл, со структурированной ошибкой, содержащей путь цикла («A → B → C → A»). Проверка — обход графа `blocks`-связей при создании; алгоритм обхода — деталь реализации и здесь не специфицируется. Поведение UI при отказе — 02-core-os.md §6.

### §4.10 Row Level Security

Принцип один: **каждая строка принадлежит ровно одному пользователю, и видит/меняет её только владелец**. Политика-шаблон для всех таблиц с `owner_id`:

```sql
CREATE POLICY owner_owns_row ON <table>
  FOR ALL
  USING (owner_id = auth.uid())
  WITH CHECK (owner_id = auth.uid());
```

Особые случаи:

- `relations` (нет своего `owner_id`) — чтение и изменение разрешены, только если **обе** связанные сущности принадлежат `auth.uid()`; это одновременно запрещает межпользовательские связи;
- `chat_messages` (нет своего `owner_id`) — доступ только через `chat_threads.owner_id = auth.uid()` по `thread_id`; вставка дополнительно проверяет принадлежность треда текущему пользователю;
- `aspect_definitions` — встроенные записи (`owner_id IS NULL`) читаемы всеми, изменяемы только service-role; кастомные — по шаблону владельца.

Все внешние ключи, которые косвенно задают владельца (`chat_messages.thread_id`, `relations.source_id/target_id`), покрыты индексами, чтобы RLS-проверка не превращала чтение в полный scan. Политики генерируются из одного шаблона — будущий переход owner→membership правится в одном месте (D11).

### §4.11 Реестр тегов — вычислимый

Отдельной таблицы тегов нет. Реестр тегов (для автокомплита, фильтров, цветов) — агрегация `SELECT unnest(tags), count(*) FROM entities GROUP BY 1` плюс клиентский кэш (TanStack Query, §5.1), обновляемый при записи сущностей. Цвета тегов — `user_settings.tagColors`.

### §4.12 Клиент vs сервер

Клиент — тонкий: вся модель данных §4.1–§4.9 существует только на сервере (PostgreSQL), собственной базы данных у клиента нет. Server-state-кэш (TanStack Query) — деталь реализации фронтенда, а не хранилище: он держит в памяти последние прочитанные данные для мгновенного рендера и инвалидируется после мутаций (§5.1), но не является источником истины и не переживает предназначенный ему жизненный цикл вкладки. Единственные персистентные клиентские данные — retry-буфер неотправленных create-мутаций fast-path-ввода (§5.3): небольшая очередь, переживающая перезагрузку страницы до подтверждения сервером. Любые другие данные, видимые в UI без сети, — это то, что успел закэшировать TanStack Query до потери соединения, а не гарантированный офлайн-доступ.

## §5. Персистентность и конкурентность

### §5.1 Источник истины и кэш

PostgreSQL (Supabase) — единственный источник истины: все чтения и записи в конечном счёте идут через tRPC в Postgres, собственной базы данных у клиента нет (§4.12). Мгновенность UX обеспечивают два механизма фронтенда, оба — детали клиентской реализации, не протокол: **server-state-кэш** (TanStack Query) — данные, однажды прочитанные с сервера, рендерятся мгновенно при повторной навигации и **инвалидируются после любой мутации**, вызывая перечитывание с сервера; и **оптимистичные обновления** — UI показывает результат мутации сразу, не дожидаясь ответа сервера, и откатывает изменение, если сервер вернул ошибку. Сервер не знает и не обязан знать о состоянии кэша конкретного клиента.

### §5.2 Конкурентность

Разрешение конфликтов — **last-write-wins (LWW) на уровне API**: из двух конкурентных мутаций одного ресурса побеждает та, что сервер обработал последней. Для правок `body` — поля, где тихая перезапись стирала бы набранный пользователем текст, — действует **optimistic-check по `updated_at`**: мутация обязана передать `updated_at` версии, которую клиент прочитал перед правкой; если серверное значение уже другое, правка **отклоняется структурированной ошибкой**, а не применяется поверх более новых данных. UI на отказ предлагает перезагрузить сущность и повторить правку вручную — автоматического merge нет. Известный компромисс: две вкладки или устройства, одновременно правящие один и тот же `body`, могут получить отказ — это осознанное ограничение (§12).

### §5.3 Retry-буфер ввода

Офлайн в Orbis — не режим работы, а буфер на входе. Буферизуются **только** create-операции fast-path-ввода (§7.5): каждая создаётся с client-generated UUIDv7 (§2.1) и складывается в локальную очередь, если запрос не смог уйти на сервер. Контракт буфера:

- **Идемпотентность по client-UUID обязательна и онлайн.** Повтор браузера, упавшая вкладка, разрыв соединения после отправки запроса, но до получения ответа — сервер обязан принимать повторный create с тем же UUID как один и тот же результат, а не создавать дубль. Это не офлайн-специфичное свойство.
- **Буфер различает transport failure и business rejection.** Обрыв соединения, таймаут — ретраится автоматически. Отказ сервера по бизнес-правилу (валидация, доменный инвариант, entitlement) — не ретраится, показывается пользователю как ошибка.
- **Удаление из очереди — только по подтверждению сервера.** Запись покидает буфер, когда сервер вернул успех или окончательный business-отказ, показанный пользователю; потеря сетевого пакета в процессе — не повод считать мутацию отправленной.
- **Корректность не зависит от service workers / Background Sync.** Буфер — обычное персистентное клиентское состояние (единственное, §4.12); он досылает накопленное при следующем открытии приложения с сетью, но не полагается на фоновые браузерные API, недоступные на всех платформах.

Офлайн-**правки** существующих сущностей недоступны: буфер работает только для create fast-path, не для update/delete и не для LLM-пути.

### §5.4 Детерминированные ID порождаемых сущностей

Все сущности, которые система порождает по расписанию от шаблона, получают ID не случайно, а детерминированно. Формула byte-точная:

```
id = uuidv5(ORBIS_NAMESPACE, name)

ORBIS_NAMESPACE = "cb339e97-82d7-4d16-91c6-942d42df7054"   // фиксированная константа проекта
name            = "<template_id>:<YYYY-MM-DD>"
```

где `template_id` — UUID шаблона **в lowercase**, `<YYYY-MM-DD>` — дата инстанса, вычисленная в таймзоне `orbis/schedule.timezone` шаблона, а при её отсутствии — в таймзоне пользователя из `user_settings.timezone`.

Пример вычисления: шаблон `019ded47-d100-717a-8307-a5b7a5be722f`, инстанс на 1 июля 2026 →

```
name = "019ded47-d100-717a-8307-a5b7a5be722f:2026-07-01"
id   = uuidv5("cb339e97-82d7-4d16-91c6-942d42df7054", name)
     = "e7d0bfa4-f62a-59c1-b560-1c17cb32e89f"
```

**Порождает только сервер.** uuidv5-инстансы вычисляет исключительно сервер при материализации; клиент их не порождает. Конкурентные запросы материализации одного шаблона на одну дату (несколько вкладок, повторные вызовы) сходятся к одному и тому же ID — дубль невозможен по построению, независимо от того, сколько раз материализация была запрошена.

**Материализация инстансов.** Инстансы recurring-шаблонов создаются лениво: любой запрос диапазона дат потребителем query-движка (Agenda, query-блоки, Budget) материализует инстансы видимого диапазона, но не дальше горизонта **14 дней вперёд** от сегодня. Финансовый инстанс получает `occurred_on=<дата инстанса>` и `planned=true`, пока дата не наступила; переход в факт определён в §3.3. Идемпотентность гарантирована детерминированными ID: уже созданные инстансы не пересоздаются и не перезаписываются.

Применения: инстансы recurring-операций и событий (`orbis/financial.recurring` + `orbis/schedule.recurrence`), чекины привычек (future, §11). Это правило 2 проектирования аспектов (§10).

**Формулы, включающие идентификатор пользователя.** Помимо инстансов от шаблона, детерминированные ID использует ещё пара сущностей — глобальный тред и тред сущности (§4.5), обе построены от `owner_id`. При введении workspace'ов (D11) эти формулы становятся **workspace-scoped**: пространство имён переезжает на `workspace_id`; формула для инстансов от шаблона (выше) уже не завязана на пользователя и не меняется.

## §6. Query-движок

Единая грамматика фильтрации сущностей. Запрос — строка из конструкций, разделённых запятыми; все условия соединяются логическим AND (внутри значения `|` даёт OR).

### §6.1 Грамматика

| Возможность | Синтаксис | Пример / семантика |
|---|---|---|
| Теги (OR внутри) | `tags=a\|b` | Сущности с тегом `work` ИЛИ `personal` |
| Исключение тегов | `excludeTags=x` | Исключить сущности с тегом |
| Наличие аспекта | `aspect=orbis/task` | Аспект должен присутствовать |
| Поле аспекта (OR) | `<поле>=v1\|v2` | `status=planned\|in_progress` |
| Отрицание | `<поле>=!v1&!v2` | `status=!done&!cancelled` — НЕ эти значения |
| Относительные даты | `<date-поле>=today\|overdue\|next_7d\|after_7d` | Токены применимы к **любому** полю типа date/timestamp: `due_date=today`, `start_at=next_7d`. `today` — сегодня; `overdue` — строго «значение поля < сегодня»; `next_7d` — диапазон [сегодня; сегодня + 7 дней], **включительно с обеих сторон**; `after_7d` — строго после этого диапазона: значение поля > сегодня + 7 дней |
| Числовое сравнение | `amount>1000` | Поле аспекта больше значения (аналогично `<`) |
| Числовой диапазон | `amount=500..2000` | Между значениями включительно |
| Дети сущности | `children_of=<uuid\|this>` | Вернуть детей X — сущности, у которых X — родитель (по `parent`). Так конверт показывает свои транзакции, проект — задачи |
| Родители сущности | `parents_of=<uuid\|this>` | Вернуть родителей X (по `parent`) |
| Исключение заблокированных | `excludeBlocked=true` | Исключить сущности, у которых есть входящая `blocks`-relation от сущности со статусом НЕ в `done\|cancelled` |
| Архивные | `archived=true\|any` | Токен отсутствует — только неархивные; `archived=true` — только архивные; `archived=any` — все |
| Сортировка multi-field | `sortBy=priority:desc\|due_date:asc` | Упорядоченный список полей с направлением |
| Полнотекстовый поиск | `search=API` | По `title` + `body` |
| Лимит | `limit=20` | Максимум результатов |
| Режим отображения | `display=compact\|list\|table` | Подсказка рендереру query-блока |
| Заголовок | `title=My Tasks` | Заголовок над результатами query-блока |

Лексика и экранирование:

- Значения, содержащие `,`, `|`, `&` или пробелы по краям, заключаются в двойные кавычки: `title="My Tasks, важное"`. Внутри кавычек `"` экранируется как `\"`. Переводы строк внутри `{{query:...}}`-блока эквивалентны пробелам.
- Числовые сравнения и диапазоны применимы к JSON Schema `number`/`integer` и к полям формата `decimal-string`; в последнем случае бэкенд (Postgres) приводит значение к `numeric` через ту же decimal-арифметику, что денежные операции (§3.3). Лексикографическое сравнение строк запрещено.
- Смешивание `|` и `&` в одном значении (`status=a|b&!c`) — ошибка парсинга (§6.4).
- `due` — документированный алиас для `orbis/task.due_date`.
- `overdue` — чистое сравнение дат, **без доменной логики задач** («и не закрыто» в семантику токена не входит): для «просроченных и незакрытых» задач условия комбинируются явно — `due_date=overdue, status=!done&!cancelled`.
- Относительные date-токены комбинируются через `|` как обычные значения: `due_date=today|overdue` — срок сегодня ИЛИ просрочен.
- Сортировка enum-полей — по **порядку объявления значений в схеме аспекта** (`aspect_definitions.schema`), не по алфавиту. NULL-значения — всегда в конце, независимо от направления.

Правила резолва имени поля: сначала зарезервированные ключи грамматики из таблицы выше (`tags`, `aspect`, `sortBy`, `title`…), затем **core-поля сущности** `created_at`, `updated_at`, `title` (см. §4.1), затем имена полей аспектов (`status`, `amount`, `due_date`…) по схемам реестра. Core-поля доступны для фильтрации и сортировки наравне с полями аспектов: `created_at=today`, `sortBy=updated_at:desc`. Исключение — `title`: в позиции фильтра ключ `title=` занят параметром заголовка, поэтому core-поле `title` участвует только в `sortBy`, а отбор по заголовку выполняется через `search=`. При неоднозначности имени поля аспекта (одно имя в двух аспектах) запрос обязан содержать `aspect=`; имя, не разрешившееся ни одним шагом резолва, — неизвестное поле (ошибка парсинга, §6.4). `display` и `title` — параметры представления, на отбор не влияют.

Пример полного запроса в body проекта:

```
{{query: aspect=orbis/task, status=!done&!cancelled, excludeBlocked=true,
         children_of=this, sortBy=priority:desc|due_date:asc, limit=20,
         display=compact, title=Активные задачи проекта}}
```

Иллюстрация семантики (псевдо-SQL):

```sql
SELECT * FROM entities
WHERE owner_id = :uid AND NOT archived
  AND aspects ? 'orbis/task'
  AND aspects->'orbis/task'->>'status' NOT IN ('done', 'cancelled')
  AND id IN (SELECT target_id FROM relations
             WHERE source_id = :this AND relation_type = 'parent')
  -- excludeBlocked: NOT EXISTS входящей blocks-связи
  --                 от сущности со status NOT IN ('done', 'cancelled')
ORDER BY CASE aspects->'orbis/task'->>'priority'   -- порядок объявления enum в схеме
           WHEN 'low' THEN 0 WHEN 'medium' THEN 1 WHEN 'high' THEN 2
         END DESC NULLS LAST,
         (aspects->'orbis/task'->>'due_date') ASC NULLS LAST
LIMIT 20;
```

### §6.2 Один бэкенд и golden-тесты

Единственный SQL-бэкенд движка — **PostgreSQL**. Спецификация грамматики — одна (эта глава); парсер транслирует грамматику §6 в SQL для одного диалекта, без слоя абстракции «диалект A / диалект B».

**Приёмочный критерий — golden-тесты.** Корректность компиляции грамматика → SQL проверяется юнит-тестами на golden-фикстурах: набор пар «запрос §6 → ожидаемый SQL» плюс эталонный датасет сущностей/relations, на котором запрос обязан вернуть ожидаемый по составу и порядку результат. Любая новая конструкция грамматики добавляется в набор фикстур одновременно с реализацией. Результат запроса детерминирован одной реализацией на одном движке БД — сверять больше не с чем.

### §6.3 Шесть потребителей

Все шесть потребителей компилируют грамматику §6 в один и тот же SQL и исполняют его на одном бэкенде — PostgreSQL, через tRPC (§9.1); отдельного клиентского движка нет.

| Потребитель | Где живёт запрос |
|---|---|
| Body-блоки `{{query:...}}` | body любой сущности |
| Smart lists | body pinned-сущностей (Daily Planning, Upcoming, All Tasks) |
| Фильтры views | состояние UI Browser/Budget/Agenda |
| Прогресс целей (future) | `orbis/goal.progress_source.query` |
| Авто-чекины привычек (future) | `orbis/habit.auto_source.query` |
| AI-тул `entity_query` | вызовы LLM и MCP-агентов |

### §6.4 Поведение при ошибках

- **Невалидный запрос** (синтаксическая ошибка, неизвестное поле, неоднозначное поле без `aspect=`) — это **ошибка парсинга, отображаемая пользователю**: query-блок рендерится с сообщением об ошибке и указанием места; AI-тул получает структурированную ошибку. Запрос с ошибкой никогда не выполняется частично.
- **Пустой результат ≠ ошибка**: валидный запрос без совпадений рендерится как пустой список («ничего не найдено»). Различие принципиально: тихая пустота вместо сообщения об ошибке — главный способ потерять доверие к живым документам.

## §7. AI-слой

### §7.1 Пятислойная модель контекста

Каждый LLM-вызов собирает контекст из пяти слоёв:

```
┌ 1. System prompt ── статика: правила, соглашения, аспектные      ┐
│                     инструкции из реестра                        │ глобальный
├ 2. Память ───────── активные orbis/memory-сущности, кап ~50      ┤ глобальная
├ 3. Рабочий контекст─ активный view, открытая сущность            ┤ ситуативный
├ 4. История диалога ─ ТОЛЬКО текущий тред (rolling + summary)     ┤ скоупнутая
└ 5. Доступ к графу ── через тулы (§9.2), по необходимости         ┘ весь граф
```

| Слой | Содержимое | Токен-бюджет (ориентир) |
|---|---|---|
| 1. System prompt | Правила поведения, соглашение meta-ключей (§3.9), инструкции активных аспектов | ~800 |
| 2. Память | До 50 активных memory-фактов и правил (с учётом `scope`) | ~1–2K |
| 3. Рабочий контекст | Текущий view, открытая сущность (title, tags, аспекты, превью body) | ~500 |
| 4. История треда | Последние сообщения текущего треда + summary более старых | ~4K |
| 5. Доступ к графу | Через тулы (§9.2): в контекст входят полные определения всех активных тулов (§7.6), сами данные графа приходят ответами тулов. Токен-бюджет строки — стоимость определений тулов | ~1.5K |

Бюджеты — целевые значения для контроля стоимости, не жёсткие константы; точные числа живут в конфиге и метрятся через `ai_usage`.

### §7.2 Диалоги эфемерны, знания живут в графе

Принцип: **история тредов — рабочая поверхность, а не хранилище знаний**. То, что должно пережить разговор, переносится в граф — тремя путями:

1. **Memory-факт** — AI предлагает сохранить факт/правило, пользователь подтверждает → создаётся `orbis/memory`-сущность (§7.4).
2. **Дозапись в body якорной сущности** — вывод разговора в треде сущности дописывается в её body («добавь это резюме в проект»).
3. **Note-сущность** — отдельная заметка с relation `related_to` к обсуждавшейся сущности.

По умолчанию — изоляция: содержимое треда никуда не утекает без явного действия.

### §7.3 Треды

- **Глобальный тред** (`chat_threads.entity_id = NULL`) — мессенджер-модель: один бесконечный диалог, rolling window + summary. Это главная вкладка Chat.
- **Тред сущности** — персистентный диалог, привязанный к сущности; живёт вкладкой на её детальном экране. Контекст слоя 3 автоматически содержит эту сущность.
- **Тулы не скоупятся.** Из любого треда доступен весь набор тулов и весь граф: **скоупится разговор, а не способности**. В треде категории «Еда» можно спросить про задачи — AI дотянется через `entity_query`.

### §7.4 Память

Память AI — это `orbis/memory`-сущности (§3.7), а не скрытое состояние:

- **Видимы и редактируемы в Browser**: пользователь курирует память напрямую — правит формулировки, архивирует устаревшее.
- **Инжекция**: активные (не archived) memory-сущности попадают в слой 2; правила с `scope` — приоритетно в контексте своего домена. Кап ~50 фактов; при превышении — приоритет правилам (`kind=rule`) и недавно использованным.
- **Пополнение**: пользователь просит запомнить; AI предлагает сохранить факт из разговора (по подтверждению); эскалация исправлений (§7.8) порождает правила.

### §7.5 Fast-path: детерминированный парсер

Клиентский парсер на грамматике (без LLM) перехватывает частотные паттерны ввода до отправки в LLM. Мгновенно, бесплатно, **работает офлайн**: сам разбор не требует сети, а результат уходит на сервер через retry-буфер (§5.3), если сети сейчас нет. Пользователь не выбирает «режим» — fast-path прозрачен: если парсер уверен, карточка появляется немедленно; нет — ввод молча уходит LLM-путём.

Паттерны MVP:

| Паттерн | Пример | Результат |
|---|---|---|
| `<текст> <число>` | «обед 340» | Сущность с `orbis/financial`: `direction=expense`, `amount="340.00"`, `occurred_on` = сегодня в таймзоне пользователя, валюта по умолчанию; `category_ref` — резолв «обед» по `aliases` категорий-сущностей |
| `+<число> <текст>` | «+150000 зарплата» | `direction=income`, `amount="150000.00"`, `occurred_on` = сегодня; категория — по aliases |
| `<текст> <число> <валюта>` | «кофе 4 usd» | Как expense, но с явной `currency` |

Числа — целые и десятичные с точкой или запятой («340», «127.50», «99,90»). Валютные токены MVP: `₽`, `руб`, `р`, `$`, `€`, `USD`, `EUR`, `RUB` (без учёта регистра); без явного токена — валюта пользователя (`user_settings.defaultCurrency`).

Правила передачи в LLM (парсер обязан уступить при любой неуверенности):

- **Неизвестная категория** — текст не разрезолвился ни в один alias ни одной категории;
- **Множественные суммы** — в строке больше одного числа («обед 340 и такси 200»);
- **Вопросительная форма** — `?`, вопросительные слова («сколько я потратил на еду?») — это запрос, а не ввод.

Fast-path **применяет correction-правила из памяти** (`orbis/memory`, `kind=rule`, `scope=orbis/financial`): правило «бар → Развлечения» работает и в детерминированном пути, без LLM. Карточка результата fast-path идентична AI-карточке: inline-правка и Undo работают одинаково (§7.8).

### §7.6 Динамические тулы из реестра

Набор тулов LLM генерируется из реестра аспектов: **core-набор** (§9.2) + по одному тулу `attach_<aspect>` на каждый активный аспект (схема input = JSON Schema аспекта, описание = `ai_instructions`). Новый аспект в реестре автоматически даёт AI способность работать с ним — без изменения кода AI-слоя.

**В MVP тулы загружаются всегда, полным составом**: core-тулы и `attach_<aspect>`-тулы для каждого активного аспекта передаются в каждый LLM-вызов с полной input-схемой. Условная частичная загрузка схем по релевантности контекста разговора — Future, актуальна при заметном росте числа кастомных аспектов.

### §7.7 Модель и транспорт чата

**Модель.** В MVP — одна модель, Claude по умолчанию, без роутинга по типу операции. Разделение на классы моделей (дешёвая — для нормализации тегов, категоризации транзакций, summary тредов; основная — для создания сущностей из свободного текста, cross-aspect рассуждений, диалога) остаётся архитектурно возможным через `LLMProvider`, но не реализуется в MVP — Future Directions. Имя конкретной модели в PRD не фиксируется — это конфиг. Каждый вызов метрится в `ai_usage` с указанием модели.

**Транспорт.** Чат — обычная tRPC-мутация: клиент отправляет сообщение, ждёт и получает ответ целиком одним пакетом. Ни `useChat`, ни SSE, ни любой другой потоковый транспорт в MVP не используется; UI показывает состояние ожидания (индикатор набора текста), а не токен-за-токеном рендер.

**Реализация LLM-слоя.** Интерфейс `LLMProvider` реализуется поверх **Vercel AI SDK**, но **типы AI SDK не протекают через интерфейс**: наружу возвращаются только наши типы, AI SDK остаётся заменяемой деталью реализации. Стриминг и мульти-провайдерность доступны через AI SDK — это **шов**, оставленный для будущего перехода на потоковый транспорт, — но в MVP не используются: ответ приходит целиком, провайдер один (Claude).

### §7.8 Corrections & Undo

- **Карточка действия.** Каждое AI- или fast-path-действие порождает карточку в чате с inline-редактированием полей (поменять категорию, сумму, дату — прямо в карточке) и кнопкой Undo.
- **Журнал действий.** Каждый одиночный вызов executor'а или атомарный batch записывает один action в `chat_messages.metadata.actions`. Формат элемента:

```json
{
  "id": "0190a3c2-8e51-7b04-b21f-3d9e7a44c615",
  "type": "entity_created",
  "entity_id": "018f...",
  "operations": [{ "op": "entity_create", "payload": { "id": "018f..." } }],
  "inverse": [{ "op": "entity_archive", "payload": { "id": "018f..." } }]
}
```

  Для одиночного действия массивы содержат один элемент. Для импорта, rollover, рекатегоризации и переноса лимитов — все операции группы; `inverse` хранится в обратном порядке исполнения. Обратные операции: создание → архивация; обновление → восстановление старых значений (`payload` = прежние поля); создание relation → её удаление. Журнал общий для внутреннего чата, fast-path и MCP-агентов (§9.3).
- **Атомарность группы.** `batch_execute` (§9.2) валидирует все операции до записи и применяет их одной PostgreSQL-транзакцией. Ошибка любого элемента откатывает всю группу; карточка и журнал появляются только после полного успеха. Action получает `id=batch_id`, а системное audit-сообщение — детерминированный `uuidv5(ORBIS_NAMESPACE, "batch:<owner_id>:<batch_id>")`; уникальность PK сообщения делает повтор batch проверяемым: повторный вызов с тем же `batch_id` конфликтует по PK вместо создания дубля и возвращает сохранённый результат, ничего не применяя повторно.
- **Состояние Undo при append-only журнале.** `chat_messages.metadata` неизменяема (§4.6), поэтому отмена не правит записанное действие, а добавляет **новое системное сообщение** `{type: "undo", undoes: "<action_id>"}` (где `action_id` — `id` элемента журнала). Действие считается отменённым, если в журнале существует undo-запись с его `id`; повторная отмена отклоняется по этой же проверке. «Отмени последнее» сканирует журнал с конца, пропуская уже отменённые действия, и применяет `inverse` первого неотменённого.
- **Эскалация в правило.** После **двух одинаковых исправлений** (например, пользователь дважды переложил «бар» из «Еды» в «Развлечения») AI предлагает создать memory-правило; по подтверждению появляется `orbis/memory` (`kind=rule`, `scope=orbis/financial`), которое применяется и LLM-путём, и fast-path-парсером. Счётчик одинаковых исправлений не хранится отдельным состоянием — он вычисляется сканом журнала действий за последние 30 дней.
- **Retention журнала** (carried-решение RET-02). Полные записи действий (`chat_messages.metadata.actions` и связанные системные сообщения) хранятся **90 дней**, после чего вычищаются; диагностика отклонённых команд (структурированные ошибки валидации и бизнес-отказов) — **30 дней**; идемпотентные receipts команд — минимальная запись «эта мутация с этим client-UUID/`batch_id` уже выполнена и вот её результат», переживающая полную запись действия, — **180 дней**, чтобы повторная отправка (§5.3, идемпотентность по client-UUID) оставалась безопасной дольше срока жизни полной записи.

### §7.9 Деградация

| Состояние | Работает | Не работает |
|---|---|---|
| Офлайн | Fast-path-ввод (уходит в retry-буфер, §5.3, досылается при восстановлении сети); чтение уже загруженных данных из server-state-кэша (§5.1) | Правки существующих сущностей; LLM-путь; данные, не попавшие в кэш до потери сети |
| Сеть есть, AI недоступен (сбой провайдера, исчерпан лимит) | Всё, кроме LLM-пути, работает как обычно | LLM-путь — явная ошибка с указанием причины и кнопкой «повторить»; запрос не встаёт в очередь |

Инвариант изменился по сравнению с полным offline-first: Orbis больше не остаётся полнофункциональным инструментом без сети — офлайн честно покрыт только вводом (буфер) и тем, что уже в кэше; это осознанная цена онлайн-первого разреза (§12). AI при этом не становится точкой отказа для остального продукта: сбой провайдера или исчерпанный лимит — явная ошибка с возможностью повторить, не тихая деградация и не очередь.

### §7.10 Политика подтверждений AI-действий

Каждый tool-call — внутреннего чата или MCP-агента (§9.3) — после структурной валидации input'а (§9.2) получает от **политики Orbis**, не от модели, один из четырёх уровней подтверждения:

| Уровень | Поведение |
|---|---|
| `execute` | Выполняется немедленно; карточка и запись в журнале (§7.8) появляются постфактум |
| `preview` | Действие исполняется без явного клика пользователя, но результат сопровождается предпросмотром (например, diff изменяемых полей) — предпросмотр информационный, не блокирующий |
| `explicit-confirmation` | Действие не исполняется, пока пользователь не подтвердил его на карточке-запросе; до подтверждения ничего не записано ни в граф, ни в журнал |
| `forbidden` | Tool-call отклоняется до исполнения структурированной ошибкой; ни модель, ни агент не могут обойти запрет переформулировкой вызова |

**Уровень определяет политика после валидации, не модель.** LLM не решает, насколько его действие рискованно, — решает серверный слой политики, которому на вход идут типизированные признаки вызова, а не текст рассуждения модели. Входы классификации:

- **явность намерения** — прямая команда пользователя («удали задачу X») классифицируется мягче, чем самостоятельная инициатива модели в рамках более широкой просьбы;
- **обратимость** — есть ли у операции `inverse` в журнале (§7.8) и восстанавливает ли он состояние полностью;
- **масштаб** — `single` (одна сущность), `bounded` (заранее известный конечный список, например batch с явным списком ID), `bulk` (открытый диапазон — «архивируй все задачи с тегом X»);
- **внешние эффекты** — затрагивает ли действие что-то за пределами графа Orbis (в MVP таких тулов нет, но политика на них рассчитана заранее);
- **чувствительность** — типы данных и операций с повышенной ценой ошибки (например, безвозвратные операции с финансовыми записями).

**Одобрение исполняет сохранённый payload, не повторяет вызов модели.** Когда пользователь подтверждает действие уровня `explicit-confirmation`, executor применяет тот же неизменяемый payload, который был провалидирован в момент запроса подтверждения, — после ревалидации текущего состояния (данные могли измениться за время ожидания), но без обращения к LLM повторно. Подтверждение — это разрешение исполнить уже собранный план, не новый раунд рассуждения.

**Правила едины для внутреннего AI и MCP-агентов.** Один и тот же классификатор уровня применяется к вызовам из чата и к вызовам через MCP (§9.3): внешний агент не может получить более широкие права, обойдя политику через другой транспорт.

## §8. Entitlements & metering

Шов для будущей монетизации, закладываемый сейчас, чтобы потом не переписывать проверки по всему коду.

- **Entitlements-резолвер** — один серверный модуль, отвечающий на вопрос «что можно этому пользователю и с какими лимитами» по строковым ключам фич. Резолвер **принимает субъекта параметром**, а не подразумевает единственного пользователя системы — это фиксирует workspace-ready границу (D11) на будущий переход owner→membership. **Все** проверки лимитов в системе идут только через него.

| Ключ | Что ограничивает | Значение для plan `'dev'` |
|---|---|---|
| `ai.requests_per_day` | LLM-запросов в день | не ограничено |
| `ai.tokens_per_day` | Токенов (input+output) в день | не ограничено |
| `aspects.custom_max` | Кастомных аспектов в реестре | не ограничено |
| `import.csv` | Доступность CSV-импорта | разрешено |
| `history.depth_days` | Глубина истории в запросах | не ограничено |
| `agents.requests_per_day` | Запросов внешних агентов (MCP) в день | не ограничено |

- **`user_settings.plan`** — строка (сейчас у всех `'dev'`); маппинг план → значения ключей живёт в серверном конфиге, не в БД.
- **`ai_usage`** (§4.7) — метеринг с первого дня: резолвер сравнивает дневные счётчики с лимитами; владельцу метеринг полезен и лично (сколько стоит AI-слой).
- **Rate limits** (частотные ограничения AI- и agent-запросов) — тоже через резолвер, по тем же ключам.
- **НЕ делаем** в этой итерации: платёжную интеграцию, состояния подписки, тарифную сетку, paywall-экраны. Когда понадобятся — добавится конфиг планов и UI, но ни одна проверка в коде не изменится.

## §9. API и внешние агенты

### §9.1 Внутренний API (tRPC)

Транспорт клиент ↔ сервер — tRPC-роутеры:

| Роутер | Назначение |
|---|---|
| `entity` | CRUD сущностей, выполнение запросов движка (§6) |
| `relation` | Создание/удаление связей, чтение графа связей |
| `aspect` | Чтение реестра аспектов, CRUD кастомных аспектов, ретроактивная миграция (§3.10) |
| `user` | `user_settings`, онбординг-сидирование, экспорт (§9.4) |
| `ai` | LLM-диалог: сборка контекста (§7.1), исполнение тулов, журнал действий |
| `chat` | Треды и сообщения (`chat_threads`, `chat_messages`) |

Отдельного роутера категорий **нет** — категории суть сущности и ходят через `entity`.

Принципы: входы/выходы валидируются Zod-схемами из shared-пакета (одни типы у клиента и сервера, end-to-end типизация); все мутации — `protectedProcedure` (аутентифицированный пользователь); мутации клиента — оптимистичные (§5.1): UI применяет результат сразу, откатывает при ошибке сервера; создание сущностей и fast-path-create **идемпотентно по client-generated `id`** (UUIDv7) — повторный запрос с тем же UUID не создаёт дубль и возвращает исходный результат (контракт retry-буфера, §5.3).

**Один путь мутаций.** PWA никогда не обращается к Supabase Data API напрямую — все мутации идут только через tRPC → executor (§9.2). Прямой доступ к БД в обход executor'а исключает журнал, Undo и валидацию инвариантов, поэтому архитектурно запрещён на уровне того, какие клиенты и ключи вообще имеют доступ к Data API.

**Min-compatible-version.** Сервер объявляет минимальную совместимую версию клиента; запрос от клиента старше этой версии получает структурированную ошибку с требованием обновиться, а не пытается работать в неопределённо совместимом режиме. Это защищает от разъезда контрактов схем/грамматики между устаревшей закэшированной PWA и текущим сервером.

Точные сигнатуры процедур в PRD не фиксируются — этот контракт живёт в коде и проверяется типами. Единственное место, где API специфицируется как внешний контракт, — реестр тулов (§9.2).

### §9.2 Реестр тулов — единый публичный контракт

Один реестр тулов с одним tool-executor'ом и **двумя потребителями**: внутренний чат-LLM (§7) и внешние агенты через MCP (§9.3). Любое действие любого потребителя проходит через executor → попадает в журнал действий (§7.8) → подлежит Undo.

Core-тулы (нотация input-схем: `*` после типа — обязательное поле, `?` после имени — опциональное):

| Тул | Назначение | Input-схема |
|---|---|---|
| `entity_query` | Поиск/фильтрация сущностей грамматикой §6 | `{ query: string* }` — строка в грамматике §6 (включая `sortBy`, `limit`). Возвращает список сущностей (core + tags + aspects) |
| `entity_get` | Полное чтение одной сущности | `{ id: uuid*, include?: ("body" \| "relations" \| "backlinks" \| "thread")[] }` — по умолчанию `body` + `relations`; `backlinks` — кто ссылается через `body_refs`; `thread` — сообщения треда сущности |
| `entity_create` | Создание сущности | `{ id?: uuid, title: string*, emoji?: string, body?: string, tags: string[]*, meta?: object, aspects?: object }` — `id` опционален (идемпотентность повторов); `aspects` валидируются схемами реестра |
| `entity_update` | Частичное обновление | `{ id: uuid*, title?, emoji?, body?, tags?, meta?, aspects?: { [aspect_id]: object \| null }, archived? }` — передаются только изменяемые поля; `aspects` мержится по aspect-id, а внутри аспекта — по полям (shallow merge: переданы только `{status, completed_at}` → остальные поля аспекта сохраняются; поле со значением `null` удаляется); значение `null` вместо объекта аспекта **снимает аспект** целиком (detach), остальные данные сущности не затрагиваются |
| `relation_create` | Создание связи | `{ source_id: uuid*, target_id: uuid*, relation_type: ("parent" \| "blocks" \| "related_to" \| "derived_from")* }` |
| `relation_delete` | Удаление связи | `{ source_id: uuid*, target_id: uuid*, relation_type: ("parent" \| "blocks" \| "related_to" \| "derived_from")* }` — тот же enum, что в `relation_create` |
| `batch_execute` | Атомарная группа мутаций с единым Undo | `{ batch_id: uuid*, operations: { tool: string*, input: object* }[]* }`. Допустимы мутирующие core- и `attach_*`-тулы, кроме самого `batch_execute`; порядок значим, весь batch валидируется до начала и выполняется одной транзакцией (§7.8) |

Внутренний хелпер `user_query` (вопрос-агрегация «сколько потрачено на еду за месяц») в публичный реестр **не входит** — это внутренний инструмент чат-LLM; MCP-агенты используют `entity_query` и агрегируют результаты сами.

Динамические тулы: **`attach_<aspect>`** — по одному на каждый активный аспект реестра, генерируются автоматически (§7.6):

| Тул | Input-схема |
|---|---|
| `attach_orbis_task`, `attach_orbis_financial`, … | `{ entity_id: uuid*, data: object* }` — `data` валидируется JSON-схемой соответствующего аспекта (§3) |

**Конвейер executor'а.** Любая мутация — от одиночного тула, batch'а (§7.8), fast-path-ввода или MCP-вызова (§9.3) — проходит один и тот же семистадийный конвейер:

1. **validate envelope** — структурная валидация входного payload по Zod-схеме тула/операции;
2. **validate aspects** — валидация значений `aspects`-полей по JSON-схемам реестра (§3, §4.3);
3. **load state** — чтение текущего состояния затронутых сущностей/связей из Postgres;
4. **validate all before first write** — проверка доменных инвариантов (§4.2, §3.3 и др.) над загруженным состоянием; при любой ошибке — отказ до единственной записи в БД;
5. **apply in transaction** — применение всех операций одной PostgreSQL-транзакцией;
6. **inverse ops + cards** — вычисление обратных операций для Undo и данных для карточки чата (§7.8);
7. **audit** — запись одного action в журнал (§7.8).

Стадии 1–4 гарантируют «всё или ничего» до первой записи: мутация, провалившая структурную валидацию, доменный инвариант или проверку состояния, не оставляет частичного следа в БД — ни для одиночного тула, ни для элемента batch'а.

Контрактные гарантии сверх самого конвейера: ошибки на любой стадии — структурированные (код + сообщение); `batch_execute` идемпотентен по `batch_id` (повтор конфликтует по PK и возвращает сохранённый результат, §7.8); лимиты — через entitlements-резолвер (§8); уровень подтверждения (§7.10) применяется до стадии 5 — `explicit-confirmation`-действие не доходит до apply без подтверждения пользователя.

### §9.3 MCP-сервер

**Статус: MVP, слайс 1.** MCP-сервер — **тонкий адаптер** над тем же tool-executor'ом, что у внутреннего чата: транслирует MCP-вызовы в тулы §9.2 один в один, без собственной бизнес-логики. Внешний агент (Claude Code и др.) получает ровно те же способности и ограничения, что внутренний AI.

- **Аутентификация: PAT-токен**, отдельный от Supabase JWT. MVP: один токен полного доступа на пользователя. Контракт хранения — **hash-only**: сервер хранит только хеш токена, сравнение при аутентификации — constant-time (защита от timing-атак); сырой токен показывается пользователю **один раз**, в момент выдачи, и не восстановим повторно. Выдача — через CLI/env; экран управления в настройках — Future Directions (00-product.md). Туда же — скоупы, несколько токенов, отзыв через UI.
- **Аудит**: действия агентов пишутся в тот же журнал действий (§7.8) — пользователь видит карточки агентских действий, inline-правка и **Undo работают и для агентов**; политика подтверждений (§7.10) применяется одинаково к внутреннему AI и MCP-агентам.
- **Лимиты**: `agents.requests_per_day` и rate limits через entitlements-резолвер (§8).

Первый эталонный сценарий — «найди среди моих задач X и выполни»:

1. `entity_query({ query: "aspect=orbis/task, status=!done&!cancelled, search=X" })` → агент получает кандидатов, выбирает нужную задачу;
2. `entity_get({ id, include: ["body", "relations", "thread"] })` → агент читает постановку из body, контекст из связей (блокеры, родительский проект) и обсуждение из треда;
3. агент выполняет работу своими средствами (вне Orbis);
4. `entity_update({ id, aspects: { "orbis/task": { status: "done", completed_at: "<now>" } } })` → задача закрыта; действие — в журнале, пользователь видит карточку и может отменить.

Второй эталонный сценарий — polling-петля агента **«что нового»** (без нового серверного механизма, только комбинация существующих тулов):

1. `entity_query({ query: "updated_at>{{cursor}}" })` — фильтр по core-полю `updated_at` (§6.1) находит задачи и другие сущности, изменённые с момента прошлого опроса;
2. `entity_get({ id, include: ["thread"] })` — агент читает новые инструкции, оставленные владельцем в треде каждой изменённой сущности;
3. курсор — «момент прошлого опроса» — агент хранит **у себя** (не на сервере Orbis); следующий цикл передаёт в `entity_query` актуальный курсор.

Оба сценария используют один и тот же реестр тулов (§9.2); специального push-канала к агенту в MVP нет — агентная петля работает на polling, инициируемом самим агентом.

### §9.4 Экспорт данных

JSON-дамп всего графа пользователя — одна ручка (`user`-роутер, §9.1) и кнопка в настройках (02-core-os §1.6). Состав дампа: сущности (`entities`), связи (`relations`), треды и сообщения чатов (`chat_threads`, `chat_messages`), настройки (`user_settings`), кастомные аспекты (`aspect_definitions` с непустым `owner_id`) — встроенные аспекты не экспортируются, это не пользовательские данные. Формат — единый JSON-файл, пригодный и для чтения человеком, и для будущего повторного импорта. При любом будущем CSV-экспорте (Future) обязателен **formula-injection escaping**: значения, начинающиеся с `=`, `+`, `-`, `@`, экранируются, чтобы открытие файла в Excel/Sheets не выполняло произвольную формулу. Экспорт доступен с первого дня слайса 1 — система, претендующая на роль единственного хранилища жизни владельца, обязана иметь выход без ожидания более зрелых итераций.

## §10. Правила проектирования аспектов

Три правила, обязательные для всех аспектов — встроенных и будущих. Каждое выведено из конкретной ошибки v2.2.

**Правило 1: ограниченные массивы.** Массив внутри аспекта допустим, только если он ограничен по размеру и пишется в рамках одной сессии (сеты одной тренировки: ~5–30 элементов, write-once). Неограниченно растущие во времени ряды — это child-сущности через relation `parent`, не массив.

*Обоснование:* JSONB-аспект пишется атомарно — каждое добавление элемента переписывает весь объект (write-amplification), а LWW на уровне API (§5.2) при конкурентной записи выбирает одну версию целиком: параллельные дописывания в массив с двух вкладок/устройств теряют более раннюю из двух правок. *Нарушение в v2.2:* `check_ins[]` внутри аспекта привычки — годовой ряд чекинов в одном JSONB: каждый чекин переписывал растущий объект, а чекины, поставленные офлайн на втором устройстве, стирались merge'ем.

**Правило 2: детерминированные ID порождаемых сущностей.** Всё, что система порождает от шаблона по расписанию, получает `id = uuidv5(ORBIS_NAMESPACE, "<template_id>:<дата>")` (точная формула — §5.4).

*Обоснование:* материализация вызывается многократно и конкурентно (открытие Agenda и Budget одновременно, повторный запрос диапазона дат) — при случайных UUID конкурентные вызовы создали бы два разных объекта «один и тот же инстанс на одну дату». Детерминированный ID делает вставку идемпотентной — дубль невозможен по построению, без дедупликации и эвристик, независимо от того, сколько раз материализация была запрошена. *Нарушение в v2.2:* recurring-инстансы со случайными UUID, порождаемые на клиенте офлайн, — дубли платежей и событий после офлайн-работы на двух устройствах.

**Правило 3: производные значения — кэш, не истина.** Стрики привычек, прогресс целей, `spent` конвертов вычисляются из графа (транзакции, чекины, query-агрегации). Хранить их в аспекте можно только как кэш, пересчитываемый при записи затрагивающих сущностей.

*Обоснование:* производное значение, хранимое как истина, неизбежно расходится с источником — правка задним числом, Undo (§7.8) или ретроактивная миграция (§3.10) меняют исходные данные в обход кода, который когда-то вычислил кэш. Обязательный пересчёт при каждой записи, затрагивающей источник, держит кэш в согласии с графом. *Нарушение в v2.2:* `streak` хранился в аспекте привычки как авторитетное значение — после sync-merge чекинов на двух устройствах стрик молча показывал неверное число. В v3.1 тот же принцип закрывает `spent` конвертов (§3.5): хранили бы — разъезжался бы с транзакциями при каждой правке задним числом.

## §11. Приложение: проверка масштабируемости

Три будущих домена прогнаны через модель данных v3.1 — ядро (слои сущности, relations, query-движок, персистентность и конкурентность §5) не потребовало изменений ни в одном случае.

### §11.1 Фитнес

```
Программа «База 3×/нед»            (entity + orbis/fitness-program)
 ├─ parent ──→ День A: грудь/трицепс   (orbis/workout-plan:
 │              упражнения — ограниченный массив, правило 1)
 │              └─ derived_from ──→ Тренировка 2026-06-09
 │                                  (orbis/fitness: фактические сеты
 │                                   + orbis/schedule: когда была)
 ├─ parent ──→ День B: спина/бицепс
 │              └─ derived_from ──→ Тренировка 2026-06-11 ...
 └─ parent ──→ День C: ноги
```

Сеты тренировки — массив внутри `orbis/fitness`: ограничен (десятки элементов) и пишется в одной сессии — правило 1 соблюдено. Каталог упражнений — клиентский JSON + кастомные упражнения как сущности. Ядру нужны только записи в реестре (`orbis/fitness-program`, `orbis/workout-plan`, `orbis/fitness`) и view.

### §11.2 Привычки

```
Привычка «Зал 3×/нед»   (orbis/habit: тип, frequency,
 │                       streak — КЭШ (правило 3),
 │                       auto_source: {query})
 ├─ parent ──→ Чекин 2026-06-09   id = uuidv5(NS, habit_id + ":2026-06-09")
 ├─ parent ──→ Чекин 2026-06-11   id = uuidv5(NS, habit_id + ":2026-06-11")
 └─ parent ──→ Чекин 2026-06-13   ...
```

Чекины — child-сущности (неограниченный ряд — правило 1), с детерминированными ID (правило 2), стрик — кэш, пересчитываемый из чекинов (правило 3). **Авто-чекин из графа**: `auto_source: {query}` — запрос движка §6 (например `aspect=orbis/fitness, start_at=today`); непустой результат автоматически зачитывает привычку за день — тренировка закрывает «зал 3×/нед» без ручного чекина. Это потребитель №5 единого query-движка.

### §11.3 Цели

```
Цель «Накопить 300 000 ₽»   (orbis/goal:
 │                           progress_source: {
 │                             query: "aspect=orbis/financial, direction=income,
 │                                     tags=savings",
 │                             aggregate: "sum",
 │                             field: "amount"
 │                           },
 │                           current_value — КЭШ (правило 3),
 │                           target_value: 300000)
 └─ related_to ──→ сущности-вехи, заметки
```

Прогресс цели = запрос движка §6 + агрегация: `aggregate: "sum" | "count" | "latest"`, агрегируемое поле — отдельным ключом `field` (для `sum` и `latest`; `count` поля не требует). «Прочитать 24 книги» — `aggregate: "count"`; «вес 80 кг» — `aggregate: "latest", field: "weight"`. `current_value` — кэш по правилу 3. Честное ограничение: агрегация `max` **внутрь** JSONB-массивов (цель «жим 100 кг» по сетам внутри тренировок) — за скоупом этого механизма (§12); такой цели потребуется либо вынос рекордов на уровень полей аспекта, либо расширение движка.

### §11.4 Вывод

**Ядро не меняется: новый домен = записи в реестре аспектов + view.** Слои сущности, типы relations, детерминированные ID (§5.4), грамматика запросов и контракт тулов покрывают все три домена в текущем виде; три правила §10 предотвращают известные по v2.2 ошибки на этапе проектирования аспекта.

## §12. Известные ограничения

Фиксируются честно — это осознанные компромиссы, а не упущения.

1. **Тяжёлая аналитика поверх JSONB** — не сильная сторона модели: сложные агрегации по полям аспектов (по месяцам × категориям × годам) упираются в JSONB-извлечение. Клапан: декларация `aggregations` в реестре аспектов (§4.3) как точка будущей материализации, без изменения модели данных.
2. **Офлайн-правки недоступны.** Офлайн покрывает только буферизованный ввод (retry-буфер fast-path-create, §5.3) и кэш чтения (server-state-кэш, §5.1); открытие уже загруженных данных работает, правка существующих сущностей и LLM-путь — нет.
3. **Две вкладки/устройства, правящие один и тот же `body`, — optimistic-check с отказом, не merge** (§5.2): проигравшая правка не теряется молча — она отклоняется структурированной ошибкой и требует перезагрузки и повторного ввода вручную.
4. **Полный offline-first отложен осознанно** (см. `04-decision-log.md`, D1) — сознательный компромисс ради быстрого выхода в продакшен под реальную среду владельца (телефон + ноут, почти всегда онлайн).
5. **Семантический поиск отсутствует** — только full-text (`search=`, GIN по title + body). «Найди заметку, где я думал про смену работы» работает по словам, не по смыслу — до добавления embeddings (Future Directions, 00-product.md).
6. **`max`-агрегации внутрь JSONB-массивов** (личные рекорды из сетов внутри тренировок) — вне MVP-механизма целей (§11.3): `progress_source` агрегирует поля сущностей, но не элементы массивов внутри аспектов.

## §13. Приёмочные проверки критических инвариантов

1. **Optimistic-check body:** два клиента читают сущность, оба правят `body`. Поздняя (по прибытии на сервер) правка со stale `updated_at` отклоняется структурированной ошибкой (§5.2); после перезагрузки сущности и повторной правки — правка проходит успешно.
2. **Идемпотентность досылки:** повторная отправка `entity_create` с тем же client-generated UUID (например, из retry-буфера, §5.3, после разрыва соединения на середине ответа) не создаёт дубль и возвращает исходный результат первой успешной попытки.
3. **Один entity-тред:** два клиента одновременно впервые пишут в тред одной и той же сущности (например, владелец с телефона и повторный запрос упавшей вкладки). Существует ровно одна строка `chat_threads` с детерминированным ID (§4.5, §5.4) — оба сообщения оказываются в ней; конфликт гасится уникальным индексом на уровне БД, а не логикой клиента.
4. **Атомарный batch:** в batch из трёх операций вторая не проходит валидацию. Не применяется ни одна операция и не создаётся action. Повтор успешного batch с тем же `batch_id` не меняет данные второй раз и возвращает исходный результат.
5. **RLS чатов и связей:** пользователь B не может прочитать или вставить `chat_messages` в тред пользователя A и не может создать relation, где хотя бы одна сущность принадлежит A; запросы отклоняются политикой БД, а не только API.
6. **Decimal:** операции `0.10 + 0.20`, carryover `-0.10` и сравнение `amount=0.30` дают точный результат `0.30` в PostgreSQL; точность проверяется в БД и в persisted JSON — IEEE-754 значения там отсутствуют.
7. **Один budget-parent:** конкурентные попытки привязать финансовую сущность к двум конвертам приводят максимум к одной живой budget-parent relation; проигравший вызов получает структурированную ошибку или выполняет атомарный перенос, частичного состояния нет.
