## Решения v3.1 (2026-07-02)

### D1. Онлайн-первый разрез

- **Решение:** PostgreSQL (Supabase) — единственный источник истины; клиент тонкий, мгновенность UX — оптимистичные обновления и server-state-кэш. Из MVP вырезаются: PowerSync, клиентский SQLite/WASM/OPFS, матрица конфликтов, `server_revision`, `sync_log`, tombstone-связи, конфликтные копии body, второй SQLite-бэкенд query-движка и тест-сьют эквивалентности бэкендов.
- **Статус:** принято 2026-07-02
- **Обоснование:** реальная среда владельца (телефон + ноут) почти всегда онлайн; цена честного offline-first (PowerSync + SQLite + матрица конфликтов + двухбэкендный движок) выше его ценности для этой среды. Полный offline-first — Future; при возврате отправная точка — готовый sync-движок класса PowerSync, а не рукописный протокол репликации (обе прошлые итерации сошлись: самописная синхронизация — худший вариант).
- **Заменяет:** «полный offline-first» спеки v3 целиком; PROD-01 леджера
- **Детали:** 01-architecture §5.1, §12

### D2. Офлайн = retry-буфер ввода, а не режим работы

- **Решение:** Буферизуются только create-операции fast-path (client-generated UUIDv7, идемпотентная досылка); офлайн-правки существующих сущностей недоступны. Контракт буфера — три пункта: (1) серверная идемпотентность по client-UUID обязательна и онлайн (повторы браузера, упавшие вкладки, разрыв после отправки до ответа — не офлайн-специфичное свойство); (2) буфер различает transport failure (ретраится автоматически) и business rejection (не ретраится, показывается пользователю как ошибка); (3) удаление записи из очереди — только после подтверждения сервера (успех или показанный пользователю business-отказ), а не по факту отправки пакета. Корректность не зависит от service workers / Background Sync.
- **Статус:** принято 2026-07-02
- **Обоснование:** полный offline write-back снят вместе с D1, но потеря набранного пользователем текста при разрыве связи недопустима — узкий буфер только на входе закрывает главный риск дёшево, без протокола репликации.
- **Заменяет:** офлайн-режим полного доступа на запись, предполагавшийся спекой v3
- **Детали:** 01-architecture §5.3

### D3. Конкурентность: LWW + optimistic-check

- **Решение:** Last-write-wins на уровне API для большинства полей; для правок `body` — optimistic-check по `updated_at` (правка с устаревшей версией отклоняется структурированной ошибкой, UI предлагает перезагрузить сущность и повторить правку вручную — автоматического merge нет).
- **Статус:** принято 2026-07-02
- **Обоснование:** без CRDT нужен предсказуемый и дешёвый в реализации механизм; известный компромисс — две одновременно открытые вкладки/устройства, правящие один `body`, могут получить отказ — фиксируется как осознанное ограничение.
- **Заменяет:** Yjs-решение леджера §18 (CRDT для совместного редактирования body)
- **Детали:** 01-architecture §5.2

### D4. Модель данных: JSONB v3 сохраняется + черри-пик из леджера

- **Решение:** Четыре слоя сущности и aspects-JSONB остаются без изменений — мотивация нормализации была в основном PowerSync-специфичной и умерла вместе с ним (DATA-01…08 → superseded). Из леджера переносятся: **`entity_origins`** — таблица provenance импорта с уникальностью `(owner, namespace, external_id)` вместо `meta.import_ids` (Undo импорта физически удаляет строки origins, повторный импорт возможен); **`raw_input`** живёт в `content` сообщения чата, не в `meta` сущности. Реестр тегов остаётся вычислимым.
- **Статус:** принято 2026-07-02
- **Обоснование:** причина нормализации (DATA-01…08) — совместимость с PowerSync-репликацией — отпала вместе с D1; переписывать модель данных без этой причины избыточная работа без ценности.
- **Заменяет:** DATA-01…08 леджера (нормализация модели под PowerSync)
- **Детали:** 01-architecture §2, §4.8

### D5. MCP возвращается в MVP — слайс 1

- **Решение:** Агентная петля — якорный сценарий продукта, MCP — её транспорт. MVP: один PAT полного доступа (hash-only хранение, constant-time сравнение, сырой токен показывается один раз), тот же реестр тулов, действия — через executor в журнал. Дополнение контракта — паттерн **«что нового»** для polling-петли агента, без нового серверного механизма, только комбинация существующих тулов: `entity_query` с фильтром по `updated_at` (core-поле, уже в грамматике) находит сущности, изменённые с момента прошлого опроса; `entity_get(include: ["thread"])` читает новые инструкции, оставленные владельцем в тредах изменённых сущностей; курсор «момент прошлого опроса» агент хранит у себя, не на сервере Orbis.
- **Статус:** принято 2026-07-02 (отменяет отсрочку MCP до фазы 2, принятую ранее в этой же переработке; подтверждает исходное решение спеки v3 новым обоснованием)
- **Обоснование:** без продакшена и MCP агентная петля не запускается день в день с владельцем; второй якорный сценарий (человек + агенты) требует MCP с первого слайса, а не фазы 2.
- **Заменяет:** промежуточную отсрочку MCP до фазы 2, принятую в начале этой переработки
- **Детали:** 01-architecture §9.3

### D6. Политика подтверждений AI-действий — включается, обязательна

- **Решение:** Уровень `execute / preview / explicit-confirmation / forbidden` определяет серверная политика Orbis после структурной валидации tool-call, не модель. Входы классификации — пять: явность намерения (прямая команда vs самостоятельная инициатива модели), обратимость (есть ли у операции `inverse` и восстанавливает ли он состояние полностью), масштаб (`single` / `bounded` / `bulk`), внешние эффекты (за пределами графа Orbis), чувствительность данных и операций. Одобрение действия уровня `explicit-confirmation` исполняет сохранённый неизменяемый payload, провалидированный в момент запроса подтверждения, — после ревалидации текущего состояния, но без повторного вызова модели. Правила едины для внутреннего AI и MCP-агентов.
- **Статус:** принято 2026-07-02 (перенос AI-09…14 леджера в PRD), обязательна к реализации в слайсе 1
- **Обоснование:** агентная петля (D5) увеличивает автономность действий без владельца рядом; без явной серверной политики риск непредсказуемых bulk- или необратимых операций растёт вместе с автономностью агентов.
- **Заменяет:** AI-09…14 леджера как неформализованный (вне PRD) статус — политика становится частью PRD, а не только внутренним соглашением
- **Детали:** 01-architecture §7.10

### D7. Транспорт чата

- **Решение:** Обычная tRPC-мутация — клиент отправляет сообщение, ждёт и получает ответ целиком одним пакетом. Ни `useChat`, ни SSE, ни другой потоковый транспорт в MVP не используется. Шов для будущего стриминга остаётся в интерфейсе `LLMProvider` (реализован поверх Vercel AI SDK, но типы SDK наружу не протекают).
- **Статус:** принято 2026-07-02 (закрывает открытый вопрос AI-08 леджера)
- **Обоснование:** соответствует контракту v3 «стриминг — Future»; простая мутация дешевле в реализации, чем потоковый транспорт, а UX-цена (typing-индикатор вместо токен-за-токеном рендера) приемлема для MVP.
- **Заменяет:** открытый вопрос AI-08 леджера (транспорт чата не был выбран)
- **Детали:** 01-architecture §7.7

### D8. Экспорт данных — в слайс 1

- **Решение:** JSON-дамп всего графа пользователя (сущности, relations, треды и сообщения чатов, настройки, кастомные аспекты) — одна ручка в `user`-роутере и кнопка в настройках. При любом будущем CSV-экспорте обязателен formula-injection escaping значений, начинающихся с `=`, `+`, `-`, `@`.
- **Статус:** принято 2026-07-02
- **Обоснование:** система, претендующая на роль единственного хранилища жизни владельца, обязана иметь выход с первого дня — не ждать более зрелых итераций продукта.
- **Заменяет:** отсутствие экспорта в спеке v3
- **Детали:** 01-architecture §9.4

### D9. Треды сущностей — слайс 1; Agenda-lite — слайс 2

- **Решение:** Заметки агента по задаче и обсуждение задачи с ИИ живут в треде сущности — треды сущностей переносятся в слайс 1. Чат-компонент строится переиспользуемым (лента + ввод + карточки); тред сущности на detail-экране — тот же компонент, что рендерит глобальный тред, с другим `thread_id` и включённым контекст-якорем. Agenda-lite остаётся в слайсе 2: она дёшева, но её ценность держится на recurring-инстансах, которые приходят вместе с ней.
- **Статус:** принято 2026-07-02
- **Обоснование:** агентная петля (D5) требует места для переписки владелец↔агент по конкретной задаче с первого слайса; Agenda-lite без recurring-инстансов малоценна, переносить её раньше времени незачем.
- **Заменяет:** прежний порядок слайсов спеки v3 (треды сущностей позже Agenda-lite)
- **Детали:** 02-core-os §2.2

### D10. Консолидация источников истины

- **Решение:** `docs/implementation_old/00-decision-ledger.md`, объявивший себя source of truth и конфликтовавший с PRD, получает шапку «superseded»; `docs/implementation_old/` целиком помечается архивом. Создаётся компактный `docs/prd/04-decision-log.md` (этот документ) — единственный действующий журнал решений: D1–D11 плюс carried-решения леджера.
- **Статус:** принято 2026-07-02
- **Обоснование:** два конкурирующих source of truth (леджер и PRD) создают риск рассинхронизации при любой будущей правке; один документ с явным статусом каждого решения устраняет двусмысленность.
- **Заменяет:** `docs/implementation_old/00-decision-ledger.md` как source of truth
- **Детали:** этот файл (04-decision-log.md)

### D11. Multi-user: граница без инфраструктуры

- **Решение:** Схема остаётся per-owner; workspace'ы, шаринг и роли — Future (владелец подтвердил: реализуем обязательно, но позже). Каждое решение MVP обязано сохранять границу — workspace-ready чек-лист, все пять пунктов: (1) нейминг `owner_id` / `actor_user_id` — двусмысленный `user_id` запрещён; (2) identity течёт только через request-контекст, бизнес-логика нигде не предполагает «единственного пользователя системы»; (3) RLS-политики генерируются из одного шаблона — будущий переход owner→membership правится в одном месте; (4) детерминированные uuidv5-формулы, включающие `user_id` (глобальный тред, entity-треды), помечаются «workspace-scoped при введении workspace'ов»; (5) entitlements-резолвер принимает субъекта параметром.
- **Статус:** принято 2026-07-02
- **Обоснование:** принцип границы — формулировка леджера: «Deferral means implementation must preserve the boundary, not add placeholder infrastructure» — откладывать инфраструктуру можно, но не ценой архитектуры, которую потом придётся переписывать. Отвергнутые альтернативы: `workspace_id` с первого дня (налог на каждый инвариант/тест/спайк при нулевой сегодняшней ценности); полные роли в MVP (та же преждевременная платформенность, что породила всю переработку, §1 спеки).
- **Заменяет:** расширяет AUTH-04 леджера (общий тезис «будущий sharing не меняет владение» без чек-листа) в явный workspace-ready чек-лист
- **Детали:** 01-architecture §4.10, §5.4
