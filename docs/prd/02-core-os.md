## §1. Навигация

Детализация принципов 00-product §6.1.

### §1.1 Карта экранов

Четыре вкладки нижней панели; поверх каждой — собственный стек push-экранов:

```
нижняя панель:   [ Chat ]      [ Browser ]      [ Agenda ]      [ Budget ]
                    │               │                │               │
корень вкладки:  глобальный     список сущностей  список дней    Overview
                 тред           + сайдбар pinned  (сегодня → +7)  (03-budget.md)
                    │               │                │               │
push-стек        detail          detail           detail          detail
(внутри своей       │               │                │               │
 вкладки):       detail          detail           detail          detail
                    ⋮               ⋮                ⋮               ⋮
```

Вертикальные связи под корнем — push-стек **внутри своей вкладки** (общий паттерн: detail-экран сущности пушится поверх, из него можно перейти в следующий detail), а не переходы между вкладками: между колонками стрелок нет, вкладки переключаются только нижней панелью.

- Detail-экран сущности (§3.5) — единый для всех вкладок: тап по сущности в любом списке (чат-карточка, Browser, Agenda, Budget) пушит его поверх текущего экрана той же вкладки.
- Каждая вкладка хранит **свой** стек: переключение вкладок не сбрасывает стеки — вернувшись на Browser, пользователь видит тот же открытый detail.
- Повторный тап по активной вкладке сворачивает её стек до корня.

### §1.2 Правила back

- Системный «назад» (жест или кнопка) закрывает верхний push-экран активной вкладки — «назад» всегда возвращает туда, откуда пришёл (00-product §6.1).
- На **корне вкладки** back НЕ переключает вкладки: жест отдаётся системе — стандартное платформенное поведение выхода/сворачивания PWA. Переключение вкладок — только тапом по нижней панели.

### §1.3 Deep links

Любую вещь в Orbis можно открыть по ссылке (00-product §6.1). Канонический формат:

| Ссылка | Поведение |
|---|---|
| `orbis://entity/<id>` | Активируется вкладка Browser, поверх корня пушится detail-экран сущности |
| `orbis://thread/<id>` | `<id>` — это `chat_threads.id` (01-architecture §4.5). Тред сущности → Browser + detail-экран с открытой вкладкой «Тред»; глобальный тред → вкладка Chat |
| `orbis://budget/category/<id>` | Вкладка Budget + экран категории (03-budget.md) |

- Открытие deep link **сбрасывает стек целевой вкладки** и пушит целевой экран поверх корня — прежний стек этой вкладки не сохраняется.
- Источник ссылок — пункт **«Скопировать ссылку»** в меню ⋮ detail-экрана (§3.5).
- Несуществующий или чужой `id` → экран «не найдено» с возвратом на корень вкладки; частичное открытие невозможно.
- `orbis://` — каноническая нотация документов; фактический URL-формат PWA (https-маршруты) — деталь реализации, поведение идентично.

### §1.4 Восстановление состояния

После перезапуска PWA восстанавливаются **активная вкладка и стеки всех вкладок**. Состояние навигации хранится локально на устройстве и не синхронизируется между устройствами (это не данные графа). Если сущность из сохранённого стека недоступна (не найдена в локальной базе), стек схлопывается до ближайшего валидного экрана; архивные сущности открываются нормально (мягкое удаление — 01-architecture §2.1).

### §1.5 Бейджи вкладок

Пассивная сигнализация по notification policy (00-product §6.3) — бейджи на вкладках нижней панели:

| Вкладка | Бейдж |
|---|---|
| Chat | Число LLM-запросов в офлайн-очереди (§2.6); нет очереди — нет бейджа |
| Browser | Без бейджа |
| Agenda | Число элементов в секции «Просроченное» (§4.2) |
| Budget | Определяется в 03-budget.md |

### §1.6 Экран настроек

Точка входа — иконка ⚙ в хедере корневого экрана каждой вкладки. Разделы MVP:

| Раздел | Содержимое |
|---|---|
| Общие | `timezone`, `defaultCurrency`, `weekStartDay` (хранение — `user_settings`, 01-architecture §4.4) |
| Память AI | Экран памяти (§2.7) |
| Аспекты | Список реестра встроенных и кастомных аспектов (все active в MVP — 01-architecture §3.8) |
| Views | Установленные views (`installedViews` — 01-architecture §4.4) |

PAT-токен для внешних агентов в UI настроек **отсутствует** — генерация через CLI/env (01-architecture §9.3).

## §2. Chat

### §2.1 Глобальный тред

Главная вкладка Chat — мессенджер-UI: один бесконечный диалог с AI, поле ввода внизу, история — скроллом вверх.

- **История не теряется.** Все сообщения сохраняются полностью и доступны скроллом (append-only хранение — 01-architecture §4.6). Rolling-окно и сжатие старых сообщений в summary касаются только контекста, передаваемого модели (01-architecture §7.1, слой 4; §7.3) — в UI ничего не «схлопывается» и не исчезает.
- Содержимое треда по умолчанию никуда не утекает: знания переносятся в граф только явными действиями — memory-факт, дозапись в body, note-сущность (01-architecture §7.2).
- **Состояние «AI отвечает».** Пока модель формирует ответ, в ленте показывается typing-индикатор; поле ввода остаётся активным. Сообщения, отправленные в это время, встают в очередь за текущим и обрабатываются последовательно — та же механика, что у офлайн-очереди (§2.6), но с индикатором «обрабатывается» вместо «ждут сети». Fast-path-вводы (§2.5) исполняются немедленно даже пока LLM думает. В MVP ответ приходит целиком; стриминг токенов — Future.

```
┌──────────────────────────────────────────┐
│  …история (скролл вверх)…                │
│                                          │
│              обед 340  ◄── пользователь  │
│  ┌────────────────────────────────────┐  │
│  │ ⚡ без AI                          │  │
│  │ 🍔 Обед — 340 ₽ → Еда             │  │
│  │ [категория ▾] [сумма] [Undo]      │  │
│  │ не то? → разобрать с AI           │  │
│  └────────────────────────────────────┘  │
│ ┌──────────────────────────────────────┐ │
│ │ ⏳ ждут сети: 2                      │ │  ← офлайн-очередь (§2.6)
│ └──────────────────────────────────────┘ │
│  [ поле ввода…                    ➤ ]    │
└──────────────────────────────────────────┘
```

### §2.2 Тред сущности

У любой сущности есть персистентный тред — вкладка «Тред» на её detail-экране (§3.5).

- Заголовок треда = `title` сущности; при переименовании сущности заголовок следует за ней.
- Сущность — **контекст-якорь**: она автоматически входит в рабочий контекст модели (01-architecture §7.1, слой 3; §7.3). «Обсудим детали» в треде сущности «План питания» — про план питания, без уточнений.
- Тулы не скоупятся: из треда сущности доступен весь граф (01-architecture §7.3) — скоупится разговор, не способности.
- Тред создаётся лениво — при первом отправленном сообщении; до этого вкладка показывает пустое состояние с полем ввода. ID детерминирован как `uuidv5(ORBIS_NAMESPACE, "<user_id>:entity-thread:<entity_id>")`, поэтому одновременное офлайн-создание на двух устройствах не порождает два треда (01-architecture §4.5).

### §2.3 Типы карточек MVP

Карточки — структурированные блоки в ленте чата; привязаны к сообщениям (хранение — 01-architecture §4.6, формат действий и Undo — §7.8).

| Тип | Когда появляется | Содержимое и действия |
|---|---|---|
| `entity_card` | Сущность создана или обновлена — AI-путём, fast-path или внешним агентом (01-architecture §9.3) | `title` + иконки аспектов + ключевые поля (для financial — сумма и категория, для task — статус и срок). **Inline-правка полей** прямо в карточке; кнопка **Undo**. Тап по заголовку → detail-экран |
| `entity_list` | Ответ на запрос-перечисление («покажи задачи на неделю») | Заголовок + список сущностей в native-рендеринге (§3.6); тап по строке → detail-экран |
| `query_result` | Ответ-агрегация: сумма, остаток, количество | Число + подпись («Еда в июне: 12 430 ₽ из 30 000 ₽»); действие «показать список» разворачивает в `entity_list` |
| `import_review` | CSV-импорт выписки | Содержимое и флоу — 03-budget.md |
| `error_card` | Действие не удалось | Что не получилось и почему (ошибка валидации, лимит, ошибка парсинга запроса — 01-architecture §6.4, §9.2) + предложение следующего шага |

Поведение карточек:

- Inline-правка поля в `entity_card` — то же действие, что любая правка: проходит через executor и пишется в журнал (01-architecture §7.8), то есть само подлежит Undo.
- После Undo карточка переходит в состояние «отменено» (визуально погашена, поля заблокированы); повторная отмена недоступна (01-architecture §7.8).
- Действия внешних MCP-агентов отображаются в **глобальном** треде системным сообщением («🤖 агент: закрыл задачу X») с обычной `entity_card` и Undo; записываются они тем же журналом действий (01-architecture §7.8, §9.3).

### §2.4 Suggestion chips

Под последним ответом ассистента — **2–4 контекстных продолжения**, сгенерированных AI вместе с ответом: после записи расхода — «что по бюджету?», после создания задачи — «поставить срок». Тап по чипу отправляет его текст как обычное сообщение пользователя в текущий тред. Чипы привязаны к последнему ответу и исчезают после следующего отправленного сообщения. У fast-path-карточек чипов нет (LLM не вызывался).

### §2.5 Fast-path UX

Механика парсера — 01-architecture §7.5; здесь — наблюдаемое поведение.

- Уверенный паттерн («обед 340») → `entity_card` появляется **мгновенно**, с пометкой **«⚡ без AI»**. Карточка идентична AI-карточке: те же inline-правка и Undo.
- Рядом с карточкой — действие **«не то? → разобрать с AI»**: тап автоматически отменяет fast-path-действие (обратная операция из журнала — 01-architecture §7.8) и отправляет исходную строку LLM-путём. Одна строка не порождает две сущности.
- Если парсер не уверен — строка молча уходит LLM-путём; пользователь не выбирает режим (00-product §5).

### §2.6 Очередь офлайн

Поведение деградации — 01-architecture §7.9; UI очереди:

- Над полем ввода — видимый блок **«ждут сети: N»** (N = число LLM-запросов в очереди). Блок появляется при первом запросе в очередь и исчезает, когда она пуста.
- Тап по блоку → список ожидающих запросов (текст + время); каждый можно **отменить** — запрос удаляется из очереди и не исполняется; в треде от него не остаётся следов.
- **Очередь — не лента.** До исполнения текст запроса живёт только в очереди и в ленту треда не пишется. При появлении сети запросы исполняются по порядку постановки — только тогда пара «сообщение пользователя + ответ» появляется в своём треде, поэтому append-only-хранение (01-architecture §4.6) не нарушается.
- Если сеть есть, но AI недоступен (сбой провайдера, исчерпан лимит) — та же очередь плюс явное сообщение о причине (01-architecture §7.9).
- Fast-path-ввод в очередь не встаёт — он работает офлайн (01-architecture §7.5).

### §2.7 UI памяти

Экран «Память AI» в настройках — это **Browser, отфильтрованный по аспекту `orbis/memory`** (§3.8), с кратким поясняющим текстом сверху: что AI помнит, как факты и правила влияют на ответы и как их редактировать. Отдельного хранилища и отдельного редактора нет:

- правка памяти = правка обычной сущности (title, поля аспекта, body);
- архивация memory-сущности исключает её из инжекции в контекст (01-architecture §7.4);
- сюда же попадают правила, рождённые из повторных исправлений (01-architecture §7.8).

## §3. Entity Browser

### §3.1 Структура экрана

```
┌─────────────┬────────────────────────────────────┐
│ САЙДБАР     │  [ поиск… ]  [ фильтры ▾ ]         │
│             │                                    │
│ 📌 Pinned   │  ◉ Закончить API   ‹high› 13.06   │
│ ☀️ Daily…(4)│  ◯ Купить кроссовки 👟  сб        │
│ 🗓 Upcoming…│  🔒 Задеплоить       ‹ждёт API›    │
│ 📋 All Tasks│  📄 Идея про отпуск               │
│             │  …                                 │
│ (drag ⇅)    ├────────────────────────────────────┤
│             │  [ + быстрая запись…           ➤ ] │  ← quick-capture (§3.7)
└─────────────┴────────────────────────────────────┘
```

На узких экранах сайдбар — выдвижная панель. Корень вкладки — список всех сущностей (по умолчанию неархивные, без служебных — §3.9) с поиском и фильтрами; тап по строке пушит detail-экран (§3.5). Сортировка по умолчанию — `updated_at:desc`; список подгружается порциями по 50 по мере скролла (infinite scroll); фильтры §3.8 применяются поверх.

### §3.2 Сайдбар: pinned-сущности

- Сайдбар показывает закреплённые сущности в пользовательском порядке (хранение порядка — `pinnedEntities`, 01-architecture §4.4).
- **Бейдж** у каждой pinned-сущности — число результатов **первого** query-блока её body: у Daily Planning это размер Inbox, у Upcoming — «Ближайшие 7 дней», у All Tasks — все незакрытые. Считается count-запросом **без учёта `limit`** (`limit` — параметр отображения, не подсчёта); при значении больше 99 отображается «99+». Бейдж пересчитывается клиентским движком при изменении данных; если первого query-блока нет или он невалиден — бейдж не показывается.
- **Drag-перестановка** строк сайдбара меняет порядок.
- Действие **«закрепить»** — в меню detail-экрана любой сущности (§3.5); повторное действие — «открепить». Закрепить можно любую сущность, не только smart lists.

### §3.3 Преднастроенные сущности

Три pinned-сущности создаются при онбординге (§7). Это обычные сущности с query-блоками в body — пользователь может править их, открепить или удалить; система их не «защищает». Все query-блоки — строго по грамматике 01-architecture §6.1.

**Daily Planning** — title `Daily Planning`, emoji `☀️`, tags `["smart-list"]`. Body:

```markdown
Утренний обзор: разобрать Inbox, пройтись по списку «Сегодня».

{{query: aspect=orbis/task, status=inbox,
         sortBy=created_at:desc, display=list, title=Inbox}}

{{query: aspect=orbis/task, due_date=today|overdue, status=!done&!cancelled&!waiting,
         excludeBlocked=true, sortBy=priority:desc|due_date:asc,
         display=list, title=Сегодня}}

{{query: aspect=orbis/task, status=waiting,
         sortBy=updated_at:asc, display=compact, title=Ожидание}}
```

Семантика секций: Inbox — неразобранное, свежее сверху; «Сегодня» — фокус-список: срок сегодня или просрочен, незакрытые и не ожидающие, **заблокированные исключены** (`excludeBlocked=true`); «Ожидание» — делегированное и ждущее, давно ждущие сверху. Пересечение секций намеренно: inbox-задача со сроком сегодня видна и в Inbox, и в «Сегодня» — это две линзы на одну сущность, а не дубль.

**Upcoming** — title `Upcoming`, emoji `🗓️`, tags `["smart-list"]`. Body:

```markdown
Горизонт планирования: неделя и дальше.

{{query: aspect=orbis/task, due_date=next_7d, status=!done&!cancelled,
         sortBy=due_date:asc|priority:desc, display=list, title=Ближайшие 7 дней}}

{{query: aspect=orbis/task, due_date=after_7d, status=!done&!cancelled,
         sortBy=due_date:asc, limit=30, display=compact, title=Позже}}
```

В обоих блоках **намеренно нет** `excludeBlocked=true`: Upcoming — горизонт планирования, заблокированные задачи здесь остаются видимыми с lock-иконкой (§3.6), чтобы не выпасть из поля зрения. Фокус-фильтрация — задача списка «Сегодня», а не горизонта (см. §6, случай 3).

**All Tasks** — title `All Tasks`, emoji `📋`, tags `["smart-list"]`. Body:

```markdown
{{query: aspect=orbis/task, status=!done&!cancelled,
         sortBy=updated_at:desc, display=list, title=Все незакрытые задачи}}
```

Страховочный список: всё незакрытое, недавно тронутое сверху. Задача, не попавшая ни в один smart list, всегда находится здесь.

### §3.4 Query-builder

Принцип «визуальный builder в обе стороны»: query-блок ↔ форма, без второго хранилища — строка блока в body и есть состояние.

- **Рендеринг.** Каждый `{{query:...}}`-блок в body рендерится виджетом: заголовок (из `title=`; нет параметра — без заголовка), список результатов в native-рендеринге (§3.6), счётчик, кнопка «настроить». Пустой результат валидного запроса — «ничего не найдено» (01-architecture §6.4).
- **Редактор-форма.** «Настроить» парсит строку и открывает форму по полям грамматики (01-architecture §6.1): теги (включение/исключение), аспекты, поля выбранных аспектов (значения, отрицания, сравнения, диапазоны), даты (относительные токены и конкретные значения), relation-фильтры (`children_of`/`parents_of` — выбор сущности или `this`), `excludeBlocked`, `archived`, сортировка (упорядоченный список «поле + направление»), лимит, режим отображения, заголовок.
- **Сериализация.** Сохранение формы сериализует её обратно в строку блока (включая правила кавычек и экранирования грамматики) и записывает в body. Body после этого — обычная правка сущности со всеми следствиями (sync и пр.).
- **Ошибки парсинга.** Невалидный блок рендерится **красной плашкой с текстом ошибки и указанием места** — никогда не пустым списком (различие нормативно — 01-architecture §6.4). У невалидного блока «настроить» открывает строковый редактор: форма требует валидного парса.
- **Ручное редактирование.** Режим «редактировать как текст» доступен из формы — для продвинутых, опционален; знание синтаксиса не требуется для базовых сценариев.

### §3.5 Detail-экран

```
┌────────────────────────────────────────────┐
│ ←   👟 Купить кроссовки               ⋮    │  ⋮ = меню действий (ниже)
│ [ Сущность ]  [ Тред ]                     │  ← вкладки
│  #task  #expense  #running                 │
│ ────────────────────────────────────────── │
│  body: markdown, [[entity:…]]-ссылки,      │
│  {{query:…}}-блоки как живые виджеты       │
│ ────────────────────────────────────────── │
│  Аспекты:  [Task ▾] [Financial ▾]  [+аспект]│
│  Подзадачи (2/5):  ◉ … ◯ … [+ подзадача]  │
│  Блокировки:  блокирует · заблокирована    │
│  Связанное (backlinks):  …                 │
├────────────────────────────────────────────┤
│  [ + быстрая запись…                  ➤ ]  │
└────────────────────────────────────────────┘
```

Меню ⋮: **«закрепить»** / «открепить» (§3.2); **«архивировать»** — у архивной сущности вместо него **«разархивировать»**; **«Скопировать ссылку»** — deep link `orbis://entity/<id>` (§1.3).

Порядок секций вкладки «Сущность» (пустые секции скрываются):

1. **Title + emoji** — редактируются на месте.
2. **Теги** — добавление/снятие чипами.
3. **Body** — markdown-редактор; query-блоки рендерятся виджетами (§3.4) и в режиме просмотра, и как блоки в редакторе.
4. **Карточки аспектов** — по карточке на каждый аспект сущности: поля по схеме реестра, inline-правка. На карточке — действие **«снять аспект»** (detach: `null` вместо объекта аспекта в `entity_update` — 01-architecture §9.2). Кнопка **«+аспект»** — добавление аспекта из реестра (выбор по имени/иконке/описанию — 01-architecture §4.3).
5. **Подзадачи** — дети по relation `parent` (01-architecture §4.2): дети-задачи — чеклистом (native-рендеринг §3.6), прочие дети — обычными строками; счётчик «2/5» — закрытые/все среди детей-задач; «+ подзадача» создаёт child прямо из секции.
6. **Блокировки** — два списка: «блокирует» (исходящие `blocks`) и «заблокирована» (входящие `blocks` от незакрытых задач); кнопка **«+»** создаёт `blocks`-связь с выбором сущности.
7. **Связанное (backlinks)** — объединённая секция: явные `related_to`-связи и автоматические backlinks по `body_refs` (01-architecture §4.1), с пометкой источника. Здесь же видны конфликтные копии body (01-architecture §5.3) — всегда, независимо от фильтра служебных (§3.9).
8. **Тред** — отдельная вкладка экрана (§2.2).

### §3.6 Native-рендеринг в списках

Применяется во всех списках: Browser, query-виджеты, `entity_list`-карточки чата, Agenda.

**`orbis/task`:**

| Элемент | Поведение |
|---|---|
| Чекбокс | Цвет по `priority` (high — акцентный/красный, medium — жёлтый, low — серый, не задан — нейтральный). Тап → `status=done` с проставлением `completed_at` (01-architecture §3.2); повторный тап откатывает |
| Статус-бейдж | Для статусов кроме дефолтного потока: `inbox`, `waiting` (с `waiting_for`, если задан), `cancelled` |
| Lock-иконка 🔒 | Есть живая входящая `blocks`-связь от незакрытой задачи (семантика — как у `excludeBlocked`, 01-architecture §6.1). Тап по иконке — подсказка «кем заблокирована» |
| Счётчик детей | «2/5» — закрытые/все дети-задачи; показывается при наличии детей-задач |
| Срок | `due_date`; просроченный — красным |

**`orbis/schedule`:** дата/время `start_at` в строке (диапазон, если задан `end_at`; «весь день» при `all_day`); иконка recurrence 🔁 у шаблонов (задано `recurrence` — 01-architecture §3.1).

**`orbis/financial`:** сумма со знаком и цветом по `direction` (расход — `−340 ₽`, красным; доход — `+150 000 ₽`, зелёным) + бейдж категории по `category_ref` — те же ключевые поля, что у `entity_card` в чате (§2.3).

**Generic-карточка** (остальные аспекты, включая будущие): строка/карточка «ключ — значение» по 2–3 ключевым полям аспекта (подсказки — `view_config` реестра, 01-architecture §4.3) + действие **«Открыть в [view]»**, если для аспекта установлен view (`installedViews` — 01-architecture §4.4). Новый аспект получает читаемое представление без кода в Browser.

Сущность с несколькими аспектами комбинирует элементы: «Купить кроссовки» = чекбокс + сумма + день недели.

### §3.7 Quick-capture бар

Строка быстрой записи внизу экрана списка и detail-экранов. Создаёт сущность из введённого текста (текст → `title`) **без интерпретации**: это путь ввода №3 (00-product §5), AI и fast-path-грамматика здесь не применяются — интерпретация свободного текста живёт в Chat.

**Smart list** — сущность с тегом `smart-list`: все три преднастроенные сущности §3.3 (Daily Planning, Upcoming, All Tasks) и списки, сохранённые через §3.8. Дискриминатор поведения quick-capture — именно тег, а не наличие `{{query:...}}`-блоков в body: проект с query-блоком — не smart list, его quick-capture создаёт детей.

Поведение контекстное:

| Где вызван | Что создаётся |
|---|---|
| Корень Browser | Сущность без аспектов (заметка/идея) |
| Detail-экран smart list (тег `smart-list`) | Сущность с `orbis/task`, `status=inbox` — **без relation** к smart list: smart list — линза, а не родитель, задачу находят его query-блоки (в Daily Planning — мгновенно в Inbox-блоке; в All Tasks — в общем списке; в Upcoming — после простановки `due_date`) |
| Detail-экран любой другой сущности (проект, категория — любая сущность без тега `smart-list`, даже с `{{query:...}}`-блоками в body) | Child этой сущности через relation `parent`; вызов из секции «Подзадачи» дополнительно вешает `orbis/task` (`status=inbox`) |

### §3.8 Фильтры и smart lists

Панель фильтров над списком: **теги, аспекты, статус, приоритет, диапазон дат** + переключатель «показать служебные» (§3.9). «Диапазон дат» фильтрует по `created_at` (core-поле — 01-architecture §6.1); фильтры по аспектным датам (`due_date`, `start_at`) — через query-builder (§3.4). Фильтры — потребитель №3 единого query-движка (01-architecture §6.3): любое состояние панели выразимо строкой грамматики.

Кнопка **«Сохранить как smart list»**: запрашивает имя → создаётся обычная сущность (title = имя, body = query-блок, сериализованный из текущих фильтров, тег `smart-list`) → автоматически закрепляется в сайдбаре. Дальше это просто сущность: правится query-builder'ом (§3.4), открепляется, удаляется.

### §3.9 Служебные сущности

Сущности, порождаемые системой как побочный артефакт, исключены из списков Browser и результатов фильтров **по умолчанию**:

- **конфликтные копии body** — архивные снапшоты с тегом `conflict-copy` (01-architecture §5.3);
- **чекины привычек** — future, при появлении домена привычек (01-architecture §11.2).

Переключатель **«показать служебные»** в панели фильтров возвращает их в выдачу (включая архивные конфликтные копии). Независимо от переключателя конфликтная копия всегда видна в секции «Связанное» своего оригинала (§3.5), а recurring-**инстансы** служебными не считаются — это полноценные пользовательские сущности.

## §4. Agenda-lite

Один экран, без push-подэкранов собственных — тап по строке пушит обычный detail-экран (§3.5).

```
┌──────────────────────────────────────────┐
│  ⚠ Просроченное (2)            ← красный │
│   ◉ Закончить API        срок был 11.06  │
│   ◉ Подтвердить созвон    был вчера 📅   │
│ ──────────────────────────────────────── │
│  Сегодня · сб, 13 июня                   │
│   весь день   Отпуск: день 1     ✈️      │
│   09:00   Стендап                📅 🔁   │
│   14:00   Врач                   📅 💊   │
│ ──────────────────────────────────────── │
│  Вс, 14 июня                             │
│   — день свободен —                      │
│ ──────────────────────────────────────── │
│  Пн, 15 июня …                           │
└──────────────────────────────────────────┘
```

### §4.1 Состав и группировка

- Горизонт: **Сегодня → +7 дней**, группировка по дням, каждый день — секция с заголовком.
- Дни наполняются сущностями с `orbis/schedule` по `start_at`. Строка = время `start_at` (диапазон при `end_at`) + `title` + иконки аспектов (native-рендеринг — §3.6). `all_day`-сущности — в начале дня, с пометкой «весь день», до строк со временем.
- Задачи без `orbis/schedule` в дневные секции **не попадают** — их место в Daily Planning и Upcoming (§3.3); Agenda показывает время. Единственное исключение — секция «Просроченное» (§4.2). Объединённое представление «срок + время» — задача полного Calendar view (Future).
- **Recurring:** показываются инстансы (связь `derived_from` от шаблона — 01-architecture §3.1, §5.5); сами шаблоны (сущности с заданным `recurrence`) в Agenda скрыты — иначе шаблон ежемесячного платежа висел бы строкой без даты исполнения. Открыть шаблон можно с detail-экрана инстанса.
- Пустой день — строка «день свободен»; секция дня не скрывается, чтобы горизонт читался целиком.

### §4.2 Секция «Просроченное»

Всегда сверху, с красным акцентом. Содержит:

1. **Незакрытые задачи с просроченным сроком** — `orbis/task`, `due_date` раньше сегодня, статус не `done`/`cancelled`. Без ограничения давности: задача закрывается или переносится — и уходит из секции.
2. **Незакрытые scheduled-задачи, время которых прошло** — сущность одновременно имеет `orbis/task` и `orbis/schedule`, `start_at` раньше начала сегодняшнего дня, статус не `done`/`cancelled`. Если у неё также есть `due_date`, в секции она показывается один раз по более ранней из дат.

Чистые события с одним `orbis/schedule` в «Просроченное» **не попадают**: в MVP у события нет статуса посещения, поэтому прошедшее время не означает «пропущено». Такие события остаются историей в Browser/поиске. Статус посещения и отдельный missed-events UX требуют нового аспектного поля и отложены до полного Calendar view.

Внутри секции — сортировка по релевантной дате, старейшие сверху. Элемент покидает секцию при закрытии задачи, переносе даты/времени или архивации. Счётчик секции — бейдж вкладки Agenda (§1.5).

### §4.3 Граница компонента

Agenda-lite — это список, сознательно без календарной механики: **сетка месяца/недели, drag-перенос между днями и слотами — это полный Calendar view**, отнесённый в Future Directions (00-product §10). Данные времени уже в графе — Calendar станет ещё одной линзой без миграций.

## §5. AI-сценарии ядра

Восемь эталонных сценариев «реплика → ожидаемое поведение». Механика тулов — 01-architecture §9.2, журнала — §7.8, fast-path — §7.5.

**1. «купи продукты»** (глобальный тред)
→ Создаётся сущность с `orbis/task`, `status=inbox` (дефолт — 01-architecture §3.2). В чате — `entity_card` с Undo. Задача немедленно появляется в Inbox-блоке Daily Planning (§3.3) — без какого-либо связывания: её находит query.

**2. «закончить API к пятнице»**
→ Задача со `status=planned` и `due_date` = ближайшая будущая пятница (если сегодня пятница — сегодня); дата вычисляется в таймзоне пользователя. `entity_card` показывает срок — ошибка распознавания даты правится inline.

**3. Из треда сущности «План питания»: «обсудим детали»**
→ Диалог идёт во вкладке «Тред» этой сущности; якорная сущность уже в контексте модели (§2.2), уточнять «детали чего» не нужно. Глобальный чат не засоряется: история скоупнута тредом (01-architecture §7.2–§7.3); выводы переносятся в граф только явно — memory-факт, дозапись в body, note-сущность.

**4. «запомни: не ем свинину»**
→ AI предлагает сохранить memory-факт — карточка предложения с кнопками **[Запомнить]** / **[Не надо]**. По [Запомнить] создаётся сущность с `orbis/memory` (`kind=fact`) — видимая и редактируемая в Browser и на экране «Память AI» (§2.7). Факт начинает инжектиться в контекст (01-architecture §7.4) и влияет на будущие предложения (например, рецепты в «Плане питания»).

**5. «отмени последнее»**
→ Отменяется последнее ещё не отменённое действие — механика Undo по журналу действий (01-architecture §7.8). В чате — подтверждение с указанием, что именно отменено; соответствующая карточка гаснет (§2.3). Работает и для действий fast-path и внешних агентов.

**6. «что я сделал на этой неделе?»**
→ AI вызывает `entity_query` (например, `aspect=orbis/task, status=done, sortBy=completed_at:desc, limit=100`), отбирает попавшие в текущую неделю по `completed_at` и отвечает карточкой `entity_list` с кратким резюме. Отбор недели — на стороне AI: относительные токены грамматики (01-architecture §6.1) не включают «прошедшую неделю», и расширять грамматику для этого не требуется.

**7. «обед 340»**
→ Fast-path (01-architecture §7.5): мгновенная `entity_card` с пометкой «⚡ без AI» (§2.5) — сущность с `orbis/financial`, категория «Еда» по alias. Работает офлайн; LLM не вызывается.

**8. «перенеси встречу с Димой на четверг»**
→ `entity_query` (`aspect=orbis/schedule, search=Дима`) — поиск кандидата; при нескольких совпадениях AI уточняет, какую встречу. Затем `entity_update` аспекта `orbis/schedule` (новый `start_at`). В чате — `entity_card` с новым временем и Undo; Agenda отражает перенос немедленно.

## §6. Edge cases

| Ситуация | Поведение |
|---|---|
| **Завершение родителя с активными детьми** (чекбокс на проекте, у которого есть незакрытые дети-задачи) | UI (и AI — уточняющим вопросом до вызова тулов) спрашивает, что сделать с детьми: **завершить вместе с родителем** / **отменить действие** / **завершить только родителя** (дети остаются активными и видимыми в списках). Молчаливого каскада нет |
| **Циклическая блокировка** (создание `blocks`-связи, замыкающей цикл) | Связь не создаётся; пользователь видит **путь цикла** («A → B → C → A») — плашкой в UI или `error_card` в чате. Инвариант ацикличности `blocks`-графа и серверный отказ с путём цикла — 01-architecture §4.2 |
| **Заблокированная задача с `due_date` сегодня** | Скрыта в «Сегодня» Daily Planning — там `excludeBlocked=true` (§3.3); остаётся видимой в Upcoming (срок «сегодня» попадает в `next_7d`: диапазон включает сегодня — 01-architecture §6.1) и в All Tasks, с lock-иконкой (§3.6). Задача не «исчезает из системы» — она уходит только из фокус-списка |
| **Правка инстанса recurring** | Меняется **только инстанс** — это отдельная сущность, связанная `derived_from` с шаблоном (01-architecture §3.1, §5.5). Серия правится отдельным действием **«править серию»** на detail-экране инстанса (открывает шаблон); правка шаблона влияет на будущие инстансы, уже созданные не трогает |
| **Конфликтная копия body** (sync-конфликт двух устройств) | По 01-architecture §5.3: проигравший текст сохраняется архивной сущностью с тегом `conflict-copy` и relation `related_to` к оригиналу. Видна в секции «Связанное» оригинала (§3.5); из общих списков скрыта по умолчанию как служебная (§3.9). Текст не теряется никогда |
| **Сущность без аспектов** | Полностью валидна — заметка, идея, черновик мысли. Видна в Browser обычной строкой (title + теги, без native-элементов §3.6); аспекты добавляются позже через «+аспект» или чат |

## §7. Онбординг и сидирование

При создании пользователя система создаёт стартовый набор данных (онбординг-сидирование — роутер `user`, 01-architecture §9.1). Сидирование выполняется один раз; повторный вызов дублей не создаёт.

### §7.1 Категории

Создаются **12 категорий-сущностей** с аспектом `orbis/category` (схема — 01-architecture §3.6). Aliases включают имя категории в нижнем регистре и синонимы (рус + англ) — по ним fast-path и AI резолвят ввод. Расходным категориям задаётся `spend_class`, используемый воспроизводимым расчётом «могу позволить?» (03-budget §4); пользователь может изменить его. Направление операции категория не задаёт — его несёт `orbis/financial.direction`; у типичных доходных категорий «Зарплата» и «Фриланс» `spend_class` отсутствует.

## §8. Приёмочные проверки Agenda

1. Прошедшая чистая сущность с одним `orbis/schedule` не появляется в «Просроченном» и остаётся доступна в Browser.
2. Незакрытая task-сущность с просроченным `due_date` появляется в «Просроченном» независимо от наличия schedule; после `done`, переноса или архивации исчезает.
3. Task + schedule с прошедшим `start_at` показывается в «Просроченном» один раз, даже если одновременно просрочен `due_date`.
4. Задача с одним `due_date` видна в Daily Planning/Upcoming, но не в дневной секции Agenda; после добавления `orbis/schedule` появляется в соответствующем дне.
