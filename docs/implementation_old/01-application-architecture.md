## 1. Runtime topology

MVP has three deployable entry points but only two runtime technologies:

```text
Browser / installed PWA
  React UI
  Dexie local database
  shared domain + query evaluator
          |
          | HTTPS, Supabase JWT
          v
Bun service
  tRPC API
  sync coordinator
  AI orchestration
  tool executor
  MCP Streamable HTTP endpoint
          |
          | PostgreSQL transaction
          v
Supabase
  Auth
  PostgreSQL + RLS
```

The MCP endpoint is a separate entry point and auth path in the same service, not a separate business service. It may be split later without moving domain code.

The PWA never connects directly to application tables through the Supabase Data API. It uses Supabase in the browser only for Auth. This preserves one mutation path, one audit path, and one sync protocol.

## 2. Repository boundaries

Target workspace shape:

```text
apps/
  web/                 React PWA
  api/                 Bun HTTP service: tRPC, AI, sync, MCP
packages/
  contracts/           Zod wire schemas, errors, IDs, shared constants
  domain/              entities, commands, invariants, money, dates, Undo
  query/               parser, AST, JS evaluator, SQL compiler contract
  client-db/           Dexie schema, repositories, local migrations
  server-db/           Drizzle schema, SQL repositories, DB migrations
  sync/                 diff, merge, protocol envelopes, rebase logic
  ai/                   provider interface, context builder, prompts, tools
  ui/                   reusable presentation components when duplication appears
supabase/
  migrations/          generated/reviewed SQL migration artifacts
  tests/database/      pgTAP tests for RLS and database invariants
```

`packages/ui` starts small and must not become a prerequisite for every screen. Domain-specific components stay in `apps/web` until a repeated stable pattern exists.

## 3. Dependency direction

Allowed dependency flow:

```text
contracts <- domain <- query
      ^          ^        ^
      |          |        |
 client-db     sync    server-db
      ^          ^        ^
      +------ web/api ----+
                 ^
                 |
                 ai
```

Normative rules:

1. `domain` does not import React, Dexie, Drizzle, tRPC, Supabase, or an AI SDK.
2. `query` owns one parser and one AST. SQL and JS are compilers/evaluators of that AST, never separate parsers.
3. `client-db` and `server-db` implement repository ports defined by domain/application modules.
4. `ai` can call the public tool registry but cannot import database repositories directly.
5. UI components do not construct sync diffs or mutate database rows directly. They submit commands.
6. tRPC and MCP adapters cannot implement business rules.

These rules are enforced with workspace import boundaries in ESLint and with package export maps.

## 4. Command and tool execution

Every user-visible mutation is represented as an application command. A command includes:

- `action_id` generated by the initiator;
- actor kind: user, AI, MCP agent, or system;
- ordered operations;
- command timestamp for display only;
- idempotency key;
- optional source message or import identifier.

The executor performs the same stages in both environments:

1. Validate the fixed command envelope with Zod.
2. Resolve and validate aspect payloads with Ajv.
3. Load the minimum state required for invariants.
4. Validate all operations before the first write.
5. Apply operations in a storage transaction.
6. Produce inverse operations and presentation cards.
7. Append the audit action.

Storage adapters differ, but command semantics do not. The local executor writes current state, outbox records, base snapshots where needed, and the local audit projection in one Dexie transaction. The server executor writes PostgreSQL state, revisions, and the permanent audit message in one PostgreSQL transaction.

Server validation is authoritative. Local validation exists to provide immediate behavior and prevent predictable rejection; it is not a security boundary.

## 5. Read paths

### 5.1 Web reads

Screens and widgets read only from Dexie repositories. React subscribes through `useLiveQuery`-based hooks. Repository functions return domain read models rather than raw IndexedDB records where derived values are required.

Examples:

- Browser list: query AST evaluated against local indexes and records;
- Agenda: materialization command, then local query;
- Budget overview: local read model using exact decimal arithmetic;
- chat history: local recent mirror, with explicit remote-history loading when needed.

### 5.2 Server reads

The server query compiler translates the shared AST into parameterized Drizzle SQL. SQL compilation must never concatenate untrusted query values. Full-text and JSONB expressions are isolated in compiler functions with fixture parity against the JS evaluator.

### 5.3 Remote-only operations

AI generation, full historical chat loading, account operations, and sync are remote calls. Their responses may cause a local transaction, but React does not render canonical entities directly from the network response.

## 6. Authentication and database identity

### 6.1 Browser to API

1. The browser obtains a Supabase Auth session.
2. It sends the access token as a bearer token to tRPC.
3. The API verifies signature, issuer, audience, expiry, and subject using Supabase JWKS.
4. The verified subject becomes the request user ID.

No user ID from an input payload is trusted.

### 6.2 API to PostgreSQL

The application connection must not use a role with `BYPASSRLS` for normal requests. Each user transaction:

1. starts on one checked-out connection;
2. establishes transaction-local authenticated role and JWT claim context from the already verified token;
3. executes all Drizzle queries on that transaction object;
4. resets context automatically on commit or rollback.

Migration and administrative connections use separate credentials and never serve requests.

This design requires a phase-zero proof against local Supabase and the hosted project. The proof must show that `auth.uid()` is populated, RLS rejects cross-user reads/writes, advisory locks work, and pooled connections do not leak identity. If the hosted role model prevents this safely, the fallback is a dedicated non-bypass application role with equivalent policies based on a transaction-local application user setting. Using the service role for ordinary requests is not an acceptable fallback.

## 7. API surface

tRPC routers follow PRD 01 section 9.1, with these implementation rules:

- `sync` is the only router that uploads local entity/relation/thread/message changes;
- UI CRUD procedures are allowed for server-only workflows and diagnostics, but ordinary offline-capable UI mutations use local commands plus sync;
- `ai` accepts a user message, runs provider/tool loops, and returns/streams message events;
- `chat` loads history outside the local mirror window;
- all errors use a shared stable code enum and include a request ID;
- transport retries are safe only for procedures marked idempotent by contract.

MCP maps official tool names to the same registry and executor. It has separate authentication, rate limiting, and request logging, but no separate tool implementation.

## 8. AI architecture

The AI package contains five explicit layers:

1. Provider adapter: messages, tool calls, usage, cancellation, normalized errors.
2. Model router: task class to configured provider/model; model identifiers live in environment config.
3. Context builder: system rules, relevant aspects, memory, entity/thread context, query results.
4. Tool loop: validates calls, invokes executor, feeds structured results back to the model.
5. Response renderer: stores assistant content, cards, suggestions, and audit links.

The provider never receives database credentials. It sees only context selected by the application and can mutate data only through tools.

Prompts are versioned text assets with fixture tests. Provider responses are not treated as valid until tool arguments pass schema validation. Free-form JSON extraction from assistant text is prohibited when native tool calling is available.

## 9. PWA and browser lifecycle

The service worker owns app-shell caching and update activation. It does not own canonical state or keep a separate database wrapper. Database changes from the page, worker, or another tab must all go through Dexie so reactive observers are notified.

Update policy:

- download a new app shell in the background;
- show an explicit reload prompt when an update is ready;
- do not force reload while there are unsaved drafts or active imports;
- IndexedDB migrations are forward-only and run before the new UI reads data;
- a failed local migration blocks the app with a recoverable diagnostic screen rather than clearing storage.

## 10. Cross-cutting conventions

- Persist timestamps as UTC ISO strings; preserve IANA time zone where domain semantics require it.
- Persist money as normalized decimal strings and currency separately.
- Generate IDs at the edge that first knows the record: browser for user records, deterministic UUIDv5 for derived records.
- Use structured errors, never parse error messages for control flow.
- Include `request_id`, `action_id`, `device_id`, and `user_id` in server logs where applicable; never log bodies, prompts, tokens, or imported CSV rows by default.
- Add abstractions only at process/storage/provider boundaries or after measured duplication.
