## 1. Контекст и цель

Существующий комплект документации (PRD v2.2 + 6 сопутствующих доков) содержит сильную концепцию, но страдает от: недоказанных продуктовых тезисов («80% ввода через чат»), несостыковок между документами, дыр в спецификациях (sync-конфликты, correction UX, цели) и большого объёма преждевременной платформенности (marketplace, native, iframe-изоляция).

Цель: переписать PRD с нуля, сохранив удачные решения, исправив известные дыры и сузив скоуп до **Core OS + Budget** с архитектурным заделом на будущие домены (fitness, nutrition, habits).

## 4. Концепция продукта (ядро 00-product.md)

**Главный тезис: Orbis продаёт не трекинг, а связность** — ответы на вопросы, требующие данных из нескольких доменов сразу. Это недоступно специализированным конкурентам.

Боли → ответы:

1. **Жизнь размазана по 5–10 приложениям** → один граф: «купить кроссовки 8000₽ к субботе» = одна сущность (task + financial + schedule), видна в задачах, бюджете и агенде без дублирования.
2. **Friction ввода убивает трекинг** → «обед 340» в чат (fast-path, мгновенно, офлайн) + CSV-импорт раз в неделю.
3. **Данные мёртвые** → AI-запросы поверх графа: «могу ли я позволить X?» с учётом конвертов, recurring и planned-расходов.
4. **Инструменты навязывают модель** → aspect registry: свой домен за минуту через чат, без кода.
5. **Смена инструмента = потеря истории** → «данные существуют до views»: views — линзы, данные в графе.

Метрики успеха MVP:
- Median time-to-log расхода < 5 сек (fast-path < 2 сек)
- Доля AI-записей, потребовавших исправления, < 10%
- Ежедневное использование 30+ дней подряд
- ≥ 3 cross-aspect запроса в неделю с полезным результатом
- Расходы не теряются: ручной ввод + импорт покрывают ≥ 95% транзакций банка

## 5. Модель данных (ядро 01-architecture.md)

### 5.1 Сущность: 4 слоя

Core (id UUIDv7 client-generated, title, emoji, timestamps, archived) → Body (markdown + `[[entity:ref]]` + `{{query:...}}`, body_refs[]) → Tags + Meta (прото-аспекты, AI-извлечённое) → Aspects (типизированные данные по схемам реестра).

Правила разделения слоёв:
- **meta — только AI-извлечённое до структурирования.** Никакого application state.
- **body — текст и живые запросы.** Smart lists = сущности с query-блоками.
- **aspects — всё типизированное**, включая данные views.

Что является сущностью: всё, что пользователь создаёт и видит (задачи, расходы, заметки, проекты, конверты, категории, memory-факты, smart lists, recurring-шаблоны и инстансы, чекины). Что НЕ является: user_settings, aspect_definitions, история чатов, метеринг.

### 5.2 Аспекты MVP (7 встроенных)

| Аспект | Ключевые поля |
|---|---|
| orbis/schedule | start_at, end_at, duration_min, all_day, recurrence, location, timezone |
| orbis/task | status (inbox/planned/in_progress/waiting/done/cancelled), priority, due_date, completed_at, effort_min, waiting_for |
| orbis/financial | amount, currency, direction (income\|expense), **category_ref** (uuid категории), **planned** (bool), recurring, payment_method, counterparty |
| orbis/note | content_type, pinned |
| orbis/budget | **новый**: category_ref, limit, currency, period_start, period_end, carryover. `spent` НЕ хранится — вычисляется из транзакций по relation (один источник истины) |
| orbis/category | **новый**: icon, color, aliases[] (вкл. русские) |
| orbis/memory | **новый**: kind (fact\|rule), scope (опционально: aspect-id) |

Будущие аспекты (fitness, habits, goals и т.д.) — записи в реестре + views, без изменений ядра (валидировано, см. §9).

### 5.3 Таблицы (8)

| Таблица | Изменения vs v2.2 |
|---|---|
| entities | без изменений структуры |
| relations | + updated_at, + deleted_at (tombstone) |
| aspect_definitions | + description, + icon (потеряны в старой схеме) |
| user_settings | + plan (строка, сейчас 'dev') |
| chat_threads | новая: id, user_id, entity_id (NULL = глобальный), title, archived |
| chat_messages | новая: id, thread_id, role, content, metadata (actions/cards), created_at; append-only |
| ai_usage | новая: метеринг токенов/запросов per user per day |
| sync_log | без изменений |

Реестр тегов — вычислимый (агрегация по entities.tags + клиентский кэш), таблица не нужна.

### 5.4 Sync-протокол: фиксы дыр v2.2

| Проблема v2.2 | Решение v3 |
|---|---|
| Удаление relations не синкалось | tombstone через deleted_at, sync по updated_at |
| Union-merge тегов воскрешал удалённые | LWW всего массива tags |
| Body atomic LWW — тихая потеря текста | LWW + **конфликтная копия**: проигравшая версия сохраняется archived-снапшотом с пометкой |
| Дубли recurring-инстансов с двух устройств | **Детерминированные ID**: uuid(template_id + date) для всех порождаемых сущностей (инстансы, чекины) |
| (без изменений — работало) | aspects: aspect-level LWW; meta: key-level LWW |

### 5.5 Единый query-движок

Грамматика `{{query:...}}` из v2.2 (теги, аспекты, поля, отрицания, относительные даты, диапазоны, relation-фильтры, сортировка, лимиты) становится спецификацией **одного движка с двумя бэкендами** (SQL на сервере, JS поверх IndexedDB на клиенте) и **шестью потребителями**: body-блоки, smart lists, фильтры views, прогресс целей (future), авто-чекины привычек (future), AI-тул entity_query. Это центральный архитектурный принцип v3.

### 5.6 Три правила проектирования аспектов

1. **Ограниченные массивы**: массив внутри аспекта допустим, если ограничен и пишется в рамках одной сессии (сеты тренировки). Неограниченно растущие ряды (чекины за годы) → child-сущности через parent.
2. **Детерминированные ID порождаемых сущностей**: uuid(родитель + дата) — идемпотентность при офлайн-генерации.
3. **Производные значения — кэш, не истина**: стрики, прогресс целей, spent конвертов вычисляются из графа; кэшированные копии пересчитываются при записи и после sync-merge.

## 6. AI-слой (ядро 01-architecture.md)

### 6.1 Пятислойная модель контекста

```
1. System prompt        — статика: правила, аспектные инструкции (из реестра)
2. Память               — orbis/memory сущности, кап ~50 фактов   ← глобальная
3. Рабочий контекст     — активный view, открытая сущность
4. История диалога      — ТОЛЬКО текущий тред (+ его summary)     ← скоупнутая
5. Граф данных          — через тулы, по необходимости            ← всегда доступен
```

**Принцип: диалоги эфемерны и скоупнуты, знания живут в графе.** Перенос из треда в общее — только через граф: AI предлагает memory-факт, дописывает вывод в body якорной сущности или создаёт note-сущность. По умолчанию — изоляция.

### 6.2 Треды

Глобальный чат — один бесконечный диалог (rolling window + summary). У любой сущности — свой тред (вкладка на детальном экране). Тулы доступны из любого треда: скоупится разговор, не способности.

### 6.3 Память

orbis/memory-сущности: видимы и редактируемы в Browser, пользователь курирует. Сюда же — correction-правила («бар → Развлечения»). Инжектятся в слой 2 (активные, кап по количеству).

### 6.4 Fast-path

Детерминированный парсер (грамматика, не LLM) **на клиенте**: «обед 340» → expense + резолв категории по alias; «+150000 зарплата» → income. Мгновенно, бесплатно, **работает офлайн**. Не уверен → молча передаёт в LLM-путь. Применяет correction-правила из памяти. Пользователь не выбирает «режим».

### 6.5 Тулы и модели

Динамическая генерация тулов из реестра (core-тулы + по тулу на аспект), tiered loading. Роутинг моделей: дешёвая для нормализации/категоризации, основная для создания/cross-aspect. Всё метрится в ai_usage.

### 6.6 Corrections & Undo

- Каждое AI-действие → карточка с inline-редактированием и Undo.
- **Журнал действий**: каждый tool call пишет обратную операцию (created→archive, updated→старые значения, relation→удаление). «Отмени последнее» работает из чата.
- Повторное одинаковое исправление → AI предлагает правило → memory-сущность → применяется и LLM-путём, и fast-path-парсером.

### 6.7 Деградация

Без сети и без AI работает всё, кроме LLM-пути: fast-path ввод, Browser, Agenda, Budget, правки. LLM-запросы из офлайна — в видимую очередь.

### 6.8 Внешние агенты (MCP)

Реестр AI-тулов — **единый публичный контракт** с двумя потребителями: внутренний чат-LLM и внешние агенты (Claude Code и др.) через MCP-сервер. MCP-сервер — тонкий адаптер над тем же tool-executor'ом, что у чата. Сценарий: «найди среди моих задач X и выполни» = entity_query → entity_get (body, relations, тред) → entity_update.

- Аутентификация: PAT-токен, отдельный от Supabase JWT. MVP: один токен полного доступа, генерация без UI (CLI/env). Скоупы и управление токенами — Future.
- Аудит: действия агентов пишутся в тот же журнал действий → inline-правка и Undo работают и для них.
- Лимиты: ключи entitlements `agents.requests_per_day` и т.п.
- Спецификация тулов с input-схемами — в 01-architecture (это единственное место, где API специфицируется как контракт; tRPC-процедуры — внутренний транспорт, в документах не фиксируются).

## 7. Entitlements & metering

- **Entitlements-резолвер**: один серверный модуль, отвечающий «что можно пользователю и с какими лимитами» по строковым ключам фич (ai.requests_per_day, aspects.custom_max, import.csv...). Все проверки лимитов — только через него. Сейчас возвращает «всё разрешено».
- **user_settings.plan**: строка ('dev'), маппинг план→entitlements в серверном конфиге.
- **ai_usage**: метеринг с первого дня (биллингу нужен, лично полезен).
- Rate limits — через тот же резолвер.
- НЕ делаем: платёжка, состояния подписки, тарифы, paywall.

## 8. Компоненты (ядро 02-core-os.md и 03-budget.md)

### 8.1 Навигация

Нижняя панель: **Chat / Browser / Agenda / Budget**. Views — обычные экраны (не модалки), детальный экран сущности — push поверх. Простой back-стек, deep links. При 3+ views вкладка Budget → «Views» со списком (путь к Hub заложен). Status strip отложен — бейджи на вкладках навигации.

### 8.2 Chat

Глобальный тред + вкладка-тред на детальном экране сущности. Карточки действий (inline-правка, Undo), suggestion chips. Fast-path — мгновенная карточка. Очередь LLM-запросов в офлайне — видимая. UI памяти в настройках.

### 8.3 Entity Browser

- Сайдбар: pinned-сущности (преднастроенные Daily Planning / Upcoming / All Tasks — обычные сущности с query-блоками).
- **Визуальный query-builder в обе стороны**: query-блок рендерится виджетом, «настроить» парсит строку → форма → сериализует обратно. Невалидный запрос показывает ошибку, не пустоту. Синтаксис руками — опционален.
- Детальный экран: title/emoji/теги/body/карточки аспектов/relations/тред/«+аспект».
- Quick-capture бар. Фильтры. Служебные сущности (чекины) исключены из списков по умолчанию.
- Native-рендеринг orbis/task (чекбоксы, приоритеты, блокировки) — как в v2.2.

### 8.4 Agenda-lite

Список «Сегодня / Неделя» по start_at, группировка по дням, просроченное сверху, тап → детальный экран. Без сетки (полный календарь — future view).

### 8.5 Budget

Экраны:
- **Overview**: баланс, сетка конвертов (цвета категорий-сущностей), Coming up (recurring), секция planned-расходов.
- **Категория**: конверт, carryover, транзакции, тренд. Категория — сущность: body с правилами, свой тред, иерархия через parent.
- **Транзакции**: список + фильтры + поиск.
- **Импорт CSV**: загрузка выписки → AI-парсинг → экран ревью (категоризация, дедупликация против ручных записей) → подтверждение пачкой.
- **Rollover**: AI предлагает конверты нового месяца с carryover, подтверждение одним экраном.
- Quick-add бар с последними категориями.

Модель: категория (вне времени) → parent → конверты по периодам; транзакции ссылаются category_ref; spent вычисляется только из транзакций, привязанных relation к конверту.

### 8.6 Онбординг и сидирование

При создании пользователя: ~12 категорий-сущностей с alias'ами (рус+англ), 3 pinned-сущности, дефолтные настройки (timezone, currency, plan='dev').

## 9. Приложение: проверка масштабируемости (войдёт в 01-architecture.md)

Три будущих домена прогнаны через модель — ядро не потребовало изменений:

**Фитнес**: программа (entity + orbis/fitness-program) → parent → дни-шаблоны (orbis/workout-plan с ограниченным массивом упражнений) → derived_from → фактические тренировки (orbis/fitness с фактическими сетами + orbis/schedule). Сеты — массив в аспекте (правило 1: ограничен, write-once). Каталог упражнений — клиентский JSON + кастомные сущности.

**Привычки**: привычка (orbis/habit: тип, frequency, кэш стриков) → parent → чекины-сущности (правило 1: неограниченный ряд; правило 2: детерминированные ID uuid(habit_id+date); правило 3: стрик — кэш). **Авто-чекин из графа**: auto_source: {query} — тренировка за сегодня автоматически зачитывает привычку «зал 3р/нед».

**Цели**: orbis/goal с progress_source: {query, aggregate: sum|count|latest} — прогресс = запрос+агрегация по графу (механизм, отсутствовавший в v2.2). current_value — кэш (правило 3). max внутрь массивов — за скоупом механизма, честно отмечено.

## 11. Известные ограничения (фиксируются честно)

- Тяжёлая аналитика поверх JSONB — не сильная сторона; клапан: aggregations в реестре аспектов (материализация).
- LWW-sync рассчитан на одного пользователя с 2–3 устройствами; family-сценарии потребуют пересмотра.
- Семантический поиск — только full-text до добавления embeddings.
- max-агрегации внутрь JSONB-массивов (цель «жим 100кг») — вне MVP-механизма целей.

## 12. Порядок написания документов

1. `01-architecture.md` — фундамент, на него ссылаются остальные
2. `00-product.md` — концепция, UX-принципы, метрики, роадмап
3. `02-core-os.md` — компоненты ядра
4. `03-budget.md` — первый view

Правило: каждый факт живёт ровно в одном документе; пересечения — только ссылками. Схемы аспектов — единственный источник в 01; экраны — в 02/03; боли и метрики — в 00.
