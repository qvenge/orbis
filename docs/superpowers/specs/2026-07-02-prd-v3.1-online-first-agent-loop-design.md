## 1. Контекст и цель

**Цель итерации меняется:** не «личный MVP когда-нибудь», а **максимально быстрый продакшен**, в котором ежедневно работают двое: владелец (с телефона и ноута) и внешние AI-агенты (Claude Code через MCP). Первый проект в системе — разработка самого Orbis: агент создаёт проект, переносит документацию, ведёт задачи со статусами и заметками; владелец наблюдает, отвечает в тредах и даёт инструкции с любого устройства.

**Диагноз, из которого выросла переработка.** Спека v3 назвала главной болезнью v2.2 «преждевременную платформенность». PRD v3 повторил ту же ошибку слоем ниже — **преждевременная инфраструктурность**: полный offline-first (PowerSync + клиентский SQLite + матрица конфликтов + двухбэкендный query-движок с тест-сьютом эквивалентности) встал на критический путь до первого дня жизни в системе. Реальная среда владельца — телефон + ноут, почти всегда онлайн; цена честного offline-first оказалась выше его ценности для этой среды.

**Два якорных сценария продукта** (оба — ядро, не Future):

1. **Личная ОС** — задачи, финансы, заметки, планирование дня/недели/жизни с ИИ, обладающим большим контекстом жизни владельца (граф + память + треды).
2. **Рабочее пространство человек + агенты** — внешние агенты работают внутри Orbis над задачами проектов; их действия видимы, атрибутированы, обратимы и управляемы с телефона.

Архитектура v3 второй сценарий **выдерживает без пересмотра**: реестр тулов проектировался как единый публичный контракт (MCP — тонкий адаптер), журнал действий атрибутирует и откатывает действия агентов, модель сущностей нативно выражает проекты/задачи/заметки/доки. Онлайн-первый разрез сценарию помогает: один источник истины с API — ровно то, что нужно агенту.

## 2. Решения переработки (decision log)

Каждое решение получит запись в новом `docs/prd/04-decision-log.md` (см. §5).

**D1. Онлайн-первый разрез (отменяет «полный offline-first» спеки v3 и PROD-01 леджера).**
PostgreSQL (Supabase) — единственный источник истины; клиент тонкий, мгновенность UX — оптимистичные обновления. Вырезаются из MVP: PowerSync, клиентский SQLite/WASM/OPFS, матрица конфликтов §5.2, `server_revision`, `sync_log`, tombstones relations, конфликтные копии body, второй SQLite-бэкенд query-движка и тест-сьют эквивалентности. Полный offline-first — Future; при возврате отправная точка — PowerSync-дизайн леджера, не рукописный протокол (обе прошлые итерации сошлись: рукописная репликация — худший вариант).

**D2. Офлайн = retry-буфер ввода, а не режим работы.**
Буферизуются только create-операции fast-path (client-generated UUIDv7, идемпотентная досылка); офлайн-правки недоступны. Контракт буфера: серверная идемпотентность по client-UUID обязательна даже онлайн (повторы браузера, упавшие вкладки); буфер различает transport failure (ретраить) и business rejection (не ретраить, показать ошибку); удаление из очереди — только после подтверждения сервера. Корректность не зависит от service workers / Background Sync.

**D3. Конкурентность: LWW + optimistic-check (отменяет Yjs-решение леджера §18).**
Last-write-wins на уровне API; для правок `body` — optimistic-check по `updated_at` (правка с устаревшей версией отклоняется, UI предлагает перезагрузить). Известный компромисс: две одновременно открытые вкладки могут получить отказ правки — фиксируется в PRD §12 как осознанное ограничение.

**D4. Модель данных: JSONB v3 сохраняется + черри-пик из леджера (DATA-01…08 → superseded).**
Четыре слоя сущности и aspects-JSONB остаются (мотивация нормализации была в основном PowerSync-специфичной и умерла вместе с ним). Из леджера переносятся: **`entity_origins`** — таблица provenance импорта с уникальностью `(owner, namespace, external_id)` вместо `meta.import_ids`; Undo импорта физически удаляет origins (повторный импорт возможен); **`raw_input`** живёт в контексте сообщения чата, не в meta сущности. Реестр тегов остаётся вычислимым (v3 §4.11).

**D5. MCP возвращается в MVP — слайс 1 (отменяет отсрочку, принятую ранее в этой переработке, и подтверждает исходное решение спеки v3 новым обоснованием).**
Агентная петля — якорный сценарий, MCP — её транспорт. MVP: один PAT полного доступа (hash-only хранение, constant-time сравнение, показ один раз), тот же реестр тулов, действия — через executor в журнал. Дополнение контракта: паттерн **«что нового»** для polling-петли агента — документированная комбинация существующих тулов, без нового механизма: `entity_query` с фильтром по `updated_at` (core-поле уже в грамматике) находит задачи, изменённые с прошлого опроса, `entity_get(include: ["thread"])` читает новые инструкции в их тредах; курсор «момент прошлого опроса» агент хранит у себя. Фиксируется в 01 §9 как эталонный сценарий рядом с существующим «найди задачу и выполни».

**D6. Политика подтверждений AI-действий — включается (перенос AI-09…14 леджера в PRD), обязательна.**
Уровень — `execute / preview / explicit-confirmation / forbidden` — определяет политика Orbis после валидации tool-call, не модель. Входы: явность намерения, обратимость, масштаб (single/bounded/bulk), внешние эффекты, чувствительность. Одобрение исполняет сохранённый неизменяемый payload с ревалидацией состояния — не повторный вызов модели. Единые правила для внутреннего AI и MCP-агентов.

**D7. Транспорт чата (закрывает открытый вопрос AI-08 леджера).**
Обычная tRPC-мутация, ответ целиком — соответствует контракту v3 «стриминг — Future». Ни `useChat`, ни SSE в MVP; шов для стриминга остаётся в `LLMProvider`.

**D8. Экспорт данных — в слайс 1.**
JSON-дамп (сущности, relations, треды, настройки): одна ручка + кнопка в настройках. Formula-injection escaping при любом будущем CSV-экспорте. Система, претендующая на роль единственного хранилища жизни, обязана иметь выход с первого дня.

**D9. Треды сущностей — слайс 1; Agenda-lite — слайс 2.**
Пересмотр под агентную петлю: заметки агента по задаче и обсуждение задачи с ИИ живут в треде сущности. Чат-компонент строится переиспользуемым (лента + ввод + карточки), тред сущности — тот же компонент на detail-экране. Agenda-lite дёшева, но её ценность держится на recurring-инстансах — приходит вместе с ними в слайсе 2.

**D10. Консолидация источников истины.**
`00-decision-ledger.md` объявляет себя действующим source of truth и конфликтует с PRD. Создаётся компактный `docs/prd/04-decision-log.md` (решения D1–D10 + carried-решения леджера); леджер получает шапку «superseded», `docs/implementation_old/` целиком помечается архивом.

**D11. Multi-user: граница без инфраструктуры.**
Схема остаётся per-owner; workspace'ы, шаринг и роли — Future (владелец подтвердил: реализуем обязательно, но позже). Каждое решение MVP обязано сохранять границу — workspace-ready чек-лист фиксируется в 04-decision-log:
- нейминг `owner_id` / `actor_user_id`; двусмысленный `user_id` запрещён;
- identity течёт только через request-контекст; бизнес-логика нигде не предполагает «единственного пользователя системы»;
- RLS-политики генерируются из одного шаблона — будущий переход owner→membership правится в одном месте;
- детерминированные uuidv5-формулы, включающие `user_id` (глобальный тред, entity-треды), помечаются «workspace-scoped при введении workspace'ов»;
- entitlements-резолвер принимает субъекта параметром.
Принцип границы — формулировка леджера: «Deferral means implementation must preserve the boundary, not add placeholder infrastructure». Отвергнутые альтернативы: `workspace_id` с первого дня (налог на каждый инвариант/тест/спайк при нулевой сегодняшней ценности), полные роли в MVP (преждевременная платформенность).

**Carried-решения леджера** (переносятся в 04-decision-log со статусом carried, детали — в implementation-планах): механика RLS через Bun API (транзакционно-локальная identity, роль без BYPASSRLS, service-role не fallback) + SPIKE-01 как блокирующий спайк фазы 0; принцип «PWA никогда не ходит в Supabase Data API напрямую — один путь мутаций»; 7-стадийный конвейер executor'а (validate envelope → validate aspects → load state → validate all before first write → apply in transaction → inverse ops + cards → audit); нейминг `owner_id` / `actor_user_id` (запрет двусмысленного `user_id`); retention журнала 90/180/30 дней; модель тредов CHAT-01…08 (детерминированный тред на owner+scope; экран/период/фильтры — runtime-контекст сообщения, не идентичность треда); версионированные промпты с fixture-тестами; запрет free-form JSON extraction при доступном native tool calling; лимиты multi-step-циклов LLM — границы Orbis, не дефолты SDK; pgTAP-тесты RLS в CI; правила миграций (no schema push в прод, forward-only, expand/contract); min-compatible-version для клиента; перф-бюджеты (fast-path < 500 мс на референс-устройстве); инъекция времени/рандома/юзер-настроек в тестах; LLM-вызовы вне детерминированного CI.

## 3. Архитектурный базис: сводка

| Слой | Было (v3) | Становится (v3.1) |
|---|---|---|
| Источник истины | SQLite на клиенте (PowerSync), Postgres — копия | **PostgreSQL, единственный** |
| Клиент | Локальная БД + write-back | Тонкий: tRPC + server-state-кэш (TanStack Query) + оптимистичные обновления |
| Офлайн | Всё работает офлайн | Retry-буфер fast-path-create; чтение — из кэша |
| Конфликты | Матрица §5.2, конфликтные копии | LWW + optimistic-check `updated_at` для body |
| Query-движок | Одна грамматика, **два** SQL-бэкенда | Одна грамматика, **один** бэкенд (Postgres). Принцип: «одна грамматика — шесть потребителей» |
| Модель сущности | 4 слоя, aspects-JSONB | Без изменений + `entity_origins`, `raw_input` в сообщении |
| Журнал/Undo | Есть | Без изменений + политика подтверждений (D6) |
| Fast-path | Есть, офлайн | Есть; офлайн — через retry-буфер |
| Внешние агенты | MCP в фазе 2 | **MCP в слайсе 1** + паттерн «что нового» |
| Треды | Глобальный + сущностей | Без изменений; чат-компонент переиспользуемый |
| ID | UUIDv7 клиентские; uuidv5 инстансы (клиент+сервер) | UUIDv7 клиентские (нужны буферу); uuidv5 инстансы — порождает только сервер |
| Экспорт | Отсутствовал | JSON-дамп в слайсе 1 |
| Стек | Bun/TS/tRPC/Drizzle/Supabase/Vite React PWA/Tailwind4/Zustand/PowerSync/Vercel AI SDK | То же **минус PowerSync**, плюс TanStack Query (старый отказ от него был мотивирован Dexie и инвертируется) |

## 4. Роадмап: фаза 0 + три слайса

**Фаза 0 — спайки (блокирующие, не зависят от остального):**
- SPIKE-01: RLS через Drizzle/Bun — pooled reuse не течёт identity; service-role не fallback.
- SPIKE-05: деплой-связность — хостинг (регион, session-pool, персистентный контейнер, секреты, бэкапы, стоимость). Продакшен нужен рано: агентная петля живёт только на задеплоенном приложении.

**Слайс 1 — «Агентная петля + ввод» (продакшен):**
- Сущности/проекты/задачи/заметки: CRUD через executor, статусы, relations `parent`.
- Browser-lite: список, сайдбар pinned, три сидированных smart lists, минимальный detail-экран (title/tags/body/аспекты/подзадачи).
- Chat: глобальный тред + **треды сущностей**, fast-path (расходы копятся с первого дня без Budget-view — «данные существуют до views»), LLM-путь с тулами, карточки, журнал + Undo, политика подтверждений.
- MCP: PAT, тот же реестр тулов, «что нового»-паттерн.
- Retry-буфер, сидирование (категории/smart lists/настройки/глобальный тред), экспорт, деплой.
- **Приёмка слайса 1:** агент создал в Orbis проект «Orbis», перенёс документацию (спеки/планы как note-сущности), и разработка слайса 2 ведётся через задачи в Orbis: агент двигает статусы и пишет заметки, владелец наблюдает и отвечает в тредах с телефона.

**Слайс 2 — «Финансовый контур + время»:**
Budget Overview + конверты (показывают историю, накопленную со слайса 1), quick-add бар, recurring + Coming up + planned→fact, Agenda-lite, rollover (к границе первого месяца), CSV-импорт на `entity_origins`, экран памяти + эскалация исправлений в правила, полный detail-экран (блокировки, backlinks).

**Слайс 3 — «Полировка и глубина»:**
Suggestion chips, бейджи вкладок, deep links, query-builder-форма, перф-бюджеты как гейты, `orbis/goal` + планирование горизонтов (день/неделя/месяц/год/жизнь) как AI-сценарии поверх готового механизма `progress_source`.

Метрика «покрытие ≥95% транзакций» измеряется со слайса 2 (требует CSV-импорта). Новая приёмочная метрика слайса 1: работа над Orbis ведётся в Orbis.
